# AIレビューフロー

ユーザーに承認を求める前に、AIレビュー（Skills）を実行する。

## 設定確認

`.aidlc/config.toml` の `[rules.reviewing]` セクションを読み、`mode` の値を確認:

- `mode = "required"`: AIレビュー必須（外部AIレビューツールまたはセルフレビューによる実施を推奨。ユーザー判断により例外的にユーザーレビューへ移行可能。承認は履歴に記録する）
- `mode = "recommend"`: AIレビュー推奨（スキップ可能、デフォルト）
- `mode = "disabled"`: AIレビューを行わない（直接ユーザーレビューへ進む）

## tools設定と優先ツールヒント

`.aidlc/config.toml` の `[rules.reviewing]` セクションから `tools` を読み取る。

- `tools`: AIレビューに使用するツールの優先順位リスト（配列）。デフォルト: `["codex"]`
- `tools = []`（空配列）: `cli_available=false` として扱う（セルフレビューモードに直接遷移）
- `tools` リストの先頭を「優先ツール」としてスキル呼び出し時の引数に含める
- **ツール選択の最終決定はスキル内部の責務**。ここで提供するのは最善努力のヒントのみ

**有効なレビュー種別とスキル名**:

| レビュー種別 | Skills呼び出し |
|-------------|----------------|
| code | `skill="reviewing-code"` |
| architecture | `skill="reviewing-architecture"` |
| security | `skill="reviewing-security"` |
| inception | `skill="reviewing-inception"` |

**スキル呼び出し引数フォーマット**:

```text
skill="reviewing-[type]", args="[レビュー対象] 優先ツール: [codex|claude|gemini]"
```

**エラー処理**: 未設定→デフォルト使用、空配列→`cli_available=false`、未知ツール名→警告しスキップ、非配列型→警告しデフォルト使用

## レビュー種別の決定

呼び出し元のコンテキストに基づき、実行するレビュー種別を決定する。

**CallerContextマッピングテーブル**:

| 呼び出し元ステップ | デフォルトのレビュー種別 |
|---|---|
| 計画承認前 | architecture |
| Phase 1 ステップ3（設計レビュー） | architecture |
| Phase 2 ステップ4（コード生成後） | code |
| Phase 2 ステップ6（統合とレビュー） | code, security |
| Intent承認前 | inception |
| ユーザーストーリー承認前 | inception |
| Unit定義承認前 | inception |

**決定ルール**:

1. 呼び出し元ステップを確認し、上記テーブルからデフォルト種別を取得
2. 呼び出し元が不明な場合はユーザーに選択を求める
3. ユーザーが追加の種別を指定した場合はマージする

**複数種別実行時**: 直列実行、各種別ごとに反復レビュー（最大3回）を独立実施。全種別で指摘0件で「AIレビュー完了」。

## AIレビューツール利用可否の確認

**Skills確認**:

1. 全種別について `skill="reviewing-[type]"` の利用可否を個別確認
2. 全種別利用可能 → 外部CLI可用性チェックへ
3. 一部のみ利用可能: `required`→利用不可種別のスキップ承認要求、`recommend`→警告表示して利用可能種別のみで続行
4. すべて利用不可: AIレビュー不可として処理

**外部CLI可用性チェック**: `tools`リスト先頭から`which`で確認し、最初にインストールされているCLIを使用ツールとして決定。

> **テンポラリファイル規約**: 本ドキュメント内の一時ファイル操作は `common/rules.md` のテンポラリファイル規約に従う。パスは `mktemp` で事前に生成すること。コードブロック内の `/tmp/aidlc-*` で始まるパスはパターン例示である。

## 共通履歴記録パターン

本ドキュメント内の履歴記録は全て以下のパターンに従う:

1. `mktemp /tmp/aidlc-history-content.XXXXXX` でパス生成
2. Readツールで生成されたパスを読み取り
3. Writeツールで履歴コンテンツを書き込み
4. `scripts/write-history.sh` を実行:

```bash
scripts/write-history.sh \
    --cycle "{{CYCLE}}" \
    --phase "{{PHASE}}" \
    --unit "{N}" \
    --unit-name "[Unit名]" \
    --unit-slug "[unit-slug]" \
    --step "[ステップ名]" \
    --content-file /tmp/aidlc-history-content.txt
```

5. 一時ファイルを削除

- `{{PHASE}}`: 呼び出し元のフェーズ（`construction` または `inception`）
- `--unit`, `--unit-name`, `--unit-slug`: constructionフェーズの場合のみ指定。inceptionフェーズではこれらの引数を省略する

## 処理フロー

1. **mode確認**: `.aidlc/config.toml` を読んでmodeを確認
   - 空または取得失敗時は「recommend」として扱う
   - `disabled` → ステップ8へ、`required`/`recommend` → 次のステップへ

2. **レビュー種別決定**: 「レビュー種別の決定」セクションのルールに従う

3. **優先ツール決定・利用可否チェック**:
   - `tools`リスト先頭を優先ツールとする（デフォルト: `codex`）
   - **空配列チェック**: `tools = []` → `cli_available=false`としてwhichチェックスキップ
   - 全種別について`skill="reviewing-[type]"`の利用可否を個別確認
   - **外部CLI可用性チェック**（空配列でない場合のみ）: `tools`リスト先頭から`which`で確認。**ツール名バリデーション**: 固定ホワイトリスト（`codex`/`claude`/`gemini`）と照合、未知値は警告しスキップ（シェルインジェクション防止）

   - **遷移判定**（Skill可用性 × CLI可用性）:
     - 全種別利用可能 かつ `cli_available=true` → ステップ4へ
     - 全種別利用可能 かつ `cli_available=false` → ステップ7へ
     - 一部種別利用可能: `required`→利用不可種別のスキップ承認要求、`recommend`→警告表示。利用可能種別のみで`cli_available`に応じてステップ4または7へ
     - すべて利用不可 → ステップ7へ

4. **AIレビューツール利用可能時の選択**:
   - `required` → ステップ5へ
   - `recommend`: `semi_auto`→自動的にステップ5へ、`manual`→推奨メッセージ表示しユーザーに選択（はい→ステップ5、いいえ→ステップ8）

5. **AIレビューフロー**:
   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う

   - **機密情報除外スキャン**:

     レビュー対象ファイルを外部AIツールへ渡す前に、機密情報を含む可能性のあるファイルを除外する。セルフレビュー（ステップ6）ではスキップ。

     **デフォルト除外パターン**（常に適用）: `.env*`, `*.key`, `*.pem`, `credentials.*`, `*secret*`

     **カスタムパターン**: `.aidlc/config.toml` の `[rules.reviewing].exclude_patterns`（配列）。未設定/空→デフォルトのみ、設定あり→デフォルトに追加。非配列型→警告しデフォルトのみ。

     **パターンマッチルール**: `/`を含まないパターン→ファイルのベース名に適用、`/`を含むパターン→プロジェクトルート相対パスに適用。照合はケースインセンシティブ。

     **スキャン手順**:
     1. レビュー対象ファイル一覧取得
     2. パターン照合しマッチファイルを除外
     3. 全ファイル除外時は警告出力しステップ8へ遷移
     4. 除外ファイルがある場合はパスのみユーザーに通知（内容は含まない）
     5. 除外後のファイル一覧を以降のAIレビュー呼び出し引数に使用

   - **種別ごとの反復レビュー**（レビュー種別リストを先頭から順に直列実行）:

     各種別で以下を実行:

     1. AIレビューを実行: `skill="reviewing-[type]", args="[レビュー対象] 優先ツール: [tool]"`

     - **種別単位のフォールバック処理**:

       | エラー種類 | 判定条件 | フォールバック動作 |
       |:---|:---|:---|
       | CLI不在（恒久） | コマンド未検出（exit code 127等） | 該当種別のみセルフレビュー（ステップ6）へ遷移 |
       | CLI実行エラー（一時的可能性） | タイムアウト、認証失敗、ネットワークエラー等 | リトライ1回→再失敗時はユーザーに確認（セルフレビュー/スキップ/中断） |
       | CLI出力解析不能 | 応答が空、形式不一致 | 該当種別のみセルフレビューへ遷移 |

       CLI実行エラー時のスキップ選択: `required`→スキップ理由入力・履歴記録が必要、`recommend`→そのままスキップ

       **フォールバック発生時の履歴記録**: 「共通履歴記録パターン」に従い、ステップ名=「フォールバック」、コンテンツに種別・エラー種類・詳細（**機密情報マスキング済み**: `sk-****`, `Bearer ****`等）を記録。完了済み種別の結果は保持する。

     2. レビュー結果を確認
     3. 指摘があれば修正を反映
     4. 指摘がゼロになるまで1-3を繰り返す（最大3回）
     5. **全種別で指摘がゼロになった時点で**: 「【AIレビュー完了】指摘0件」を出力
     6. 「AIレビュー完了時の共通履歴記録手順」に従い履歴に記録→レビュー後コミット→ユーザーへの承認提示
     7. 3回後も指摘が残る場合は、「### 指摘対応判断フロー」を実行
     8. 判断フロー後、「修正する」選択あり→反復レビューの1に戻る

   ### Codex セッション管理（Codexを選択した場合）

   **反復レビュー時に必ずresume機能を使用する**:

   1. 初回レビュー後: Codex出力末尾の `session id: xxx` を記録
   2. 2回目以降: `codex exec resume <session-id> "<request>"` を使用
   3. resume失敗時（期限切れ等）: 新規セッションで再開し、session idを更新
   4. Unit切り替え時: セッションIDを破棄し新規セッションで開始

   ### 指摘対応判断フロー

   AIレビュー反復（最大3回）後に指摘が残っている場合に実行。ステップ5/6の反復レビューから呼び出される。

   #### 千日手検出（フロー実行前チェック）

   過去3回の反復レビュー結果で「同種の指摘」が3回とも出現しているか確認する。

   **「同種の指摘」の判定基準** — 以下の3キーがすべて実質的に一致:
   1. **レビュー種別**: 同一種別
   2. **対象ファイル**: 同一パス（パス未指定はワイルドカード扱い）
   3. **指摘内容の要約**: 本質が同じ（表現差異は無視）

   3ラウンドすべてに出現する同種グループが1つ以上あれば「千日手」と判定し、ユーザーに判断を求める（継続/ユーザー承認へ/中断）。

   千日手でユーザー承認へ進む場合: 「共通履歴記録パターン」に従い、ステップ名=「千日手判断」、コンテンツに成果物名・反復回数・繰り返し指摘の要約を記録。履歴記録後、「レビューサマリファイル更新手順」に従いサマリファイルを更新（計画承認前除く）。

   #### フロー

   1. **指摘一覧の表示**: 残りの指摘を番号付きで表示

   2. **各指摘への判断要求**（順番に繰り返す）:
      - 1: 指摘を修正する（推奨）
      - 2: 技術的理由で対応不可（TECHNICAL_BLOCKER、理由を記録）
      - 3: スコープ外として次サイクルで対応（OUT_OF_SCOPE、理由を記録）

   3. **理由入力とバリデーション**（選択肢2/3の場合）:
      - 空文字・空白のみは不可
      - **禁止パターン**のみの理由は拒否（「パッチだから」「小さい変更だから」「時間がないから」「急ぎだから」等）
      - 具体的な技術的理由や根拠を含める必要がある

   4. **先送り判断の履歴記録**: 「共通履歴記録パターン」に従い、ステップ名=「AIレビュー指摘対応判断」、コンテンツに指摘内容・判断種別（TECHNICAL_BLOCKER/OUT_OF_SCOPE）・先送り理由を記録

   5. **OUT_OF_SCOPEバックログ自動登録**（OUT_OF_SCOPE判断があった場合のみ）:

       **安全規則**: heredoc終端トークン含む入力は拒否、`{slug}`は`^[a-z0-9][a-z0-9-]{0,63}$`のみ許可、引数・パスは二重引用符で囲む

       **バックログ種別**: securityレビュー指摘→`type:security`、その他→`type:chore`

       **登録実行**: Writeツールで一時ファイルにIssue本文（スラッグ・概要・詳細・検出元）を書き出し、`gh issue create --title "[Backlog] {要約}" --label "backlog,type:{種別},priority:medium" --body-file <パス>` で作成。タイトルはシェル展開文字除去・80文字以内。失敗/gh CLI不可時は手動登録を依頼。

       **バックログ登録の履歴記録**: 「共通履歴記録パターン」に従い、ステップ名=「バックログ自動登録」、コンテンツに指摘内容・登録方法・登録先を記録

   6. **全指摘の判断完了後、サマリを履歴に記録**: 「共通履歴記録パターン」に従い、ステップ名=「AIレビュー指摘対応判断サマリ」、コンテンツに各指摘のRESOLVE/TECHNICAL_BLOCKER/OUT_OF_SCOPE判断と次のアクションを記録

   7. **次のアクション決定**:
      - RESOLVE選択あり → 修正実施後、呼び出し元の反復レビューへ戻る（ステップ5 項目8 / ステップ6 項目12）
      - 全て先送り → 「レビューステップ完了時の共通処理」に従う（シグナル `review_detected=true` が設定されるため、セミオートゲートで `fallback(review_issues)` に遷移しユーザー承認必須）

   - 「レビューステップ完了時の共通処理」に従う。

6. **セルフレビューフロー**（外部AIレビューツール不可時のフロー）:

   **注意**: セルフレビューは外部ツールに比べて品質が劣る可能性があります。

   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う

   - **レビュー実行方式の判定**（優先順位順）:

     - **A) スキル呼び出し方式（推奨）**: `skill="reviewing-{種別}"`, `args="self-review {target_files}"`。スキル側がサブエージェント起動・インラインフォールバックを管理
     - **B) インライン方式（フォールバック）**: スキル呼び出し失敗時、SKILL.mdの「レビュー観点」を読み取りメインエージェントがレビュー
     - **C) 簡易インライン方式（最終フォールバック）**: SKILL.md未発見時、以下の簡易観点でレビュー: code→コード品質・命名・エラーハンドリング、architecture→構造・責務分離・依存方向、security→入力検証・認証・機密情報、inception→要件明確性・整合性・網羅性

   - **A方式の引数パースとバリデーション**:
     - 先頭トークン`self-review`でモード判定、以降はファイルパス（スペース区切り）
     - パスバリデーション: `^[A-Za-z0-9._/-]+$`のみ許可、`..`・絶対パス拒否
     - 有効パス0件→失敗扱い（空リストでの「指摘0件」完了を禁止）
     - スキルからの結果: 指摘なし→0件処理、指摘あり→「重要度」「内容」「推奨修正」3項目解析（不揃い→無効除外）
     - フォールバック条件: スキル呼び出し失敗/結果解析不能→B方式へ

   - **反復・完了処理（方式共通）**:
     1. 指摘があれば修正反映（各指摘にid・judgement付与）
     2. 反復レビュー（最大3回）
     3. 全種別で指摘ゼロ→「【AIレビュー完了（セルフレビュー）】指摘0件」出力→履歴記録→レビュー後コミット→承認提示
     4. 3回後も指摘残→「指摘対応判断フロー」実行

   - **出力フォーマット**: `【セルフレビュー結果】`+種別+指摘一覧（重要度/内容/推奨修正/判定）+合計

   - **指摘対応判断フローとの接続**: 入力は`判定:未対応`のみ、反復時に`対応済`に更新、3回後`未対応`残→判断フロー起動

   - **外部入力検証の非適用**: セルフレビューは同一エージェントシステム内のため「AIレビュー応答の検証」は非適用。ただし構造チェック（3項目揃い）と妥当性チェック（根拠・関連性・禁止操作）は適用する。

   - 「レビューステップ完了時の共通処理」に従う。

7. **外部AIレビューで続行不能時**:
   - `required`: セルフレビュー実施（推奨）/ ユーザー承認へ（スキップ理由入力・バリデーション・履歴記録が必要）/ 中断の3択
     - スキップ理由バリデーション: 空文字不可、禁止パターン拒否（指摘対応判断フローに準ずる）、heredoc終端トークン拒否
     - スキップ時の履歴記録: 「共通履歴記録パターン」に従い、ステップ名=「AIレビュースキップ」、コンテンツにスキップ理由・対象成果物を記録
   - `recommend`: セルフレビュー推奨メッセージ（はい→ステップ6、いいえ→ステップ8）

8. **ユーザーレビューフロー**（mode=disabled または 外部AIレビューで続行不能時）:
   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う
   - 成果物をユーザーに提示→承認を求める→修正依頼があれば反映
   - **レビュー後コミット**: `common/commit-flow.md` の「レビュー反映コミット」手順に従う
   - 再度ユーザーに提示・承認を求める

## AIレビュー完了時の共通履歴記録手順

ステップ5/6で「全種別で指摘がゼロ」になった時点で、「共通履歴記録パターン」に従い、ステップ名=「AIレビュー完了」で記録する。

**コンテンツ**:

```text
【{AIレビュー完了 または AIレビュー完了（セルフレビュー）}】指摘0件
【対象タイミング】{呼び出し元のステップ名}
【対象成果物】{成果物名}
【レビュー種別】{実行した全種別}
【レビューツール】{使用したツール名 または self-review(skill) | self-review(inline)}
【注記】セルフレビューの場合のみ「セルフレビューによる実施。」を記載
```

**呼び出し元ステップ名マッピング**: 計画承認前→`計画承認前`、Phase 1 ステップ3→`設計レビュー`、Phase 2 ステップ4→`コード生成後`、Phase 2 ステップ6→`統合とレビュー`、Intent承認前→`Intent承認前`、ユーザーストーリー承認前→`ユーザーストーリー承認前`、Unit定義承認前→`Unit定義承認前`

## レビューステップ完了時の共通処理

ステップ5/6の完了時に実行:

0. **レビュー結果シグナル生成**: 当該承認ポイントの全レビュー種別の結果を集約し、以下のシグナルをコンテキスト変数として設定する（`common/rules.md`「レビュー結果シグナル」定義に準拠）
   - `review_detected`: 全種別を通じて1件以上の指摘が検出された場合 `true`、0件なら `false`
   - `deferred_count`: OUT_OF_SCOPE + TECHNICAL_BLOCKER の件数（全種別合算）
   - `resolved_count`: 修正済みの件数（全種別合算）
   - `unresolved_count`: 未対応の件数（全種別合算）
   ※ シグナルは同一承認ポイント内でのみ有効。ゲート判定後に破棄する
1. **レビュー後コミット**: `common/commit-flow.md` の「レビュー反映コミット」手順に従う
2. **レビューサマリ更新**【必須】: 「レビューサマリファイル更新手順」に従う（計画承認前除く）。千日手パスで更新済みならスキップ。
3. **セミオートゲート判定**（`common/rules.md` のセミオートゲート仕様参照）: レビュー結果シグナルに基づき判定。`review_detected=false` かつ他のフォールバック条件非該当の場合のみ `auto_approved`。`review_detected=true` の場合は `fallback(review_issues)` としてユーザーに提示・承認要求

## レビューサマリファイル更新手順

AIレビュー完了時にレビューサマリファイルを生成・追記する共通手順。

### 前提条件

- AIレビュー実施済みの場合のみ実行。**計画承認前のレビューでは実行しない**。レビュー未実施時はファイル非作成。

### ファイルパス決定ルール

| フェーズ | 呼び出し元 | ファイルパス |
|---------|----------|------------|
| Construction | 設計レビュー/コード生成後/統合とレビュー | `.aidlc/cycles/{{CYCLE}}/construction/units/{NNN}-review-summary.md` |
| Inception | Intent/ストーリー/Unit定義承認前 | `.aidlc/cycles/{{CYCLE}}/inception/{intent\|user-stories\|unit-definition}-review-summary.md` |

### 処理手順

1. ファイルパス決定（上記テーブル参照）
2. 存在しない→テンプレート（`templates/review_summary_template.md`）から新規作成、存在する→セパレータ（`---`）追記
3. セットを追記:

```markdown
## Set [N]: [YYYY-MM-DD HH:MM:SS]

- **レビュー種別**: [全種別]
- **使用ツール**: [codex / self-review(skill) / self-review(inline)]
- **反復回数**: [1〜3]
- **結論**: [指摘0件 / 指摘対応判断完了]

### 指摘一覧

| # | 重要度 | 内容 | 対応 | バックログ |
|---|--------|------|------|-----------|
| 1 | 高/中/低 | [対象箇所] - [問題事象] | 修正済み（[変更箇所]: [変更内容]） / TECHNICAL_BLOCKER（理由: [理由]） / OUT_OF_SCOPE（理由: [理由]） | - / #NNN |
```

**列の記述ガイダンス**:

- **「内容」列**: `[対象箇所] - [問題事象]` 形式。対象箇所（位置情報）と問題事象（具体的記述）は必須。
- **「対応」列**: 修正済み→変更箇所・内容必須、TECHNICAL_BLOCKER→技術的理由必須、OUT_OF_SCOPE→根拠必須。機密情報は記載禁止（マスク要）。
- **「バックログ」列**: OUT_OF_SCOPE時はバックログ参照を必須記載。有効値は `#NNN`（Issue作成済み）/ `PENDING_MANUAL`（gh CLI失敗等で手動登録待ち）。TECHNICAL_BLOCKER時は空可（`-`）。修正済み時は`-`。**security指摘の例外**: security種別の指摘は脆弱性詳細の漏洩を防ぐため、公開Issueへの詳細記載を禁止する。要約のみのマスク済みIssueを作成するか、`SECURITY_PRIVATE`（非公開対応）と記載すること。

## 外部入力検証ルール【重要】

外部からの入力（AIレビュー応答、ユーザー入力）を批判的に評価し、自己判断を明示する。

### AIレビュー応答の検証

- AIレビュー（Skills）からの応答をそのまま信頼せず、批判的に評価する
- 応答に誤りや不整合がないか確認し、自己判断を併記
- 相違がある場合はユーザーに確認を求める

```text
【AIレビュー応答の検証】
- AIレビュー応答: [応答内容の要約]
- AI判断: [自己判断]
- 相違点: [ある場合は記載、なければ「なし」]
- 結論: [採用する判断とその理由]
```

### ユーザー入力の検証

- 曖昧な入力は解釈を明示して確認。複数解釈可能な場合はすべて提示。
