# AIレビューフロー

ユーザーに承認を求める前に、AIレビュー（Skills）を実行する。

## 設定確認

`.aidlc/config.toml` の `[rules.reviewing]` セクションを読み、`mode` の値を確認:

- `mode = "required"`: AIレビュー必須（外部AIレビューツールまたはセルフレビューによる実施を推奨。ユーザー判断により例外的にユーザーレビューへ移行可能。承認は履歴に記録する）
- `mode = "recommend"`: AIレビュー推奨（スキップ可能、デフォルト）
- `mode = "disabled"`: AIレビューを行わない（AIレビューをスキップし、直接ユーザーレビューへ進む）

## tools設定と優先ツールヒント

`.aidlc/config.toml` の `[rules.reviewing]` セクションから `tools` を読み取る。

**設定形式**:

```toml
[rules.reviewing]
mode = "required"
tools = ["codex", "claude", "gemini"]  # 優先順位順
```

- `tools`: AIレビューに使用するツールの優先順位リスト（配列）
- デフォルト: `["codex"]`（未設定時）
- `tools = []`（空配列）: 外部CLIを使用しない明示的な意思表示。whichチェックをスキップし、`cli_available=false` として扱う（セルフレビューモードに直接遷移）
- `tools` リストの先頭を「優先ツール」としてスキル呼び出し時の引数に含める
- **ツール選択の最終決定はスキル内部の責務**。ここで提供するのは最善努力のヒントのみ

**有効なレビュー種別とスキル名**:

| レビュー種別 | Skills呼び出し |
|-------------|----------------|
| code | `skill="reviewing-code"` |
| architecture | `skill="reviewing-architecture"` |
| security | `skill="reviewing-security"` |
| inception | `skill="reviewing-inception"` |

**スキル呼び出し引数フォーマット**:

```text
skill="reviewing-[type]", args="[レビュー対象] 優先ツール: [codex|claude|gemini]"
```

**フォールバック**: 優先ツール指定はヒントであり強制ではない。スキルが解釈できない場合は `allowed-tools` の順序で自動選択。

**エラー処理**:

- `tools` 未設定: デフォルト `["codex"]` を使用
- 空配列: 外部CLIを使用しない意思表示として `cli_available=false` 扱い（セルフレビューモードへ遷移）
- 未知のツール名: 警告を出し、そのツールをスキップ
- 配列以外の型: 警告を出し、デフォルトを使用

## レビュー種別の決定

呼び出し元のコンテキストに基づき、実行するレビュー種別を決定する。

**CallerContextマッピングテーブル**:

| 呼び出し元ステップ | デフォルトのレビュー種別 |
|---|---|
| 計画承認前 | architecture |
| Phase 1 ステップ3（設計レビュー） | architecture |
| Phase 2 ステップ4（コード生成後） | code |
| Phase 2 ステップ6（統合とレビュー） | code, security |
| Intent承認前 | inception |
| ユーザーストーリー承認前 | inception |
| Unit定義承認前 | inception |

**決定ルール**:

1. 呼び出し元ステップを確認し、上記テーブルからデフォルト種別を取得
2. 呼び出し元が不明な場合はユーザーに選択を求める
3. ユーザーが追加の種別を指定した場合はマージする

**複数種別実行時のルール**:

- 複数種別は**直列実行**（1つずつ順番に実行）
- 各種別ごとに反復レビュー（最大3回）を独立して実施
- **全種別で指摘0件**になった時点で「AIレビュー完了」とする
- 履歴記録には実行した全種別を記載

## AIレビューツール利用可否の確認

**Skills確認**:

1. レビュー種別リストの**全種別**について、対応するスキルが利用可能か個別に確認
   - 各 `skill="reviewing-[type]"` で呼び出し可能かを確認
2. 全種別が利用可能 → 外部CLI可用性チェックへ
3. 一部のみ利用可能:
   - `mode = "required"`: 利用不可の種別ごとにユーザーにスキップ承認を求める（承認は履歴に記録）。全て承認後、利用可能な種別のみで外部CLI可用性チェックへ。承認拒否 → AIレビュー不可として処理
   - `mode = "recommend"`: 利用不可の種別について警告を表示し、利用可能な種別のみで外部CLI可用性チェックへ
4. すべて利用不可: AIレビュー不可として処理

**外部CLI可用性チェック**:

`tools` リストを先頭から順に `which` コマンドで確認し、最初にインストールされているCLIを「使用ツール」として決定する。`tools = []`（空配列）の場合はwhichチェックをスキップし、`cli_available=false` として扱う。CLI可用性に応じて外部CLIレビュー（ステップ5）またはセルフレビュー（ステップ6）に振り分ける。詳細は処理フローのステップ3を参照。

**環境別の確認方法**:

- Claude Code: Skillツールが存在し、`skill="reviewing-[type]"` で呼び出し可能かを確認
- KiroCLI等の他環境: Skill一覧取得APIがあればそれを利用

> **テンポラリファイル規約**: 本ドキュメント内の一時ファイル操作は `common/rules.md` のテンポラリファイル規約に従う。パスは `mktemp` で事前に生成すること。コードブロック内の `/tmp/aidlc-*` で始まるパスはパターン例示である。

## 処理フロー

1. **mode確認**: `.aidlc/config.toml` を読んでmodeを確認
   - 空または取得失敗時は「recommend」として扱う
   - `disabled` の場合: ステップ8（ユーザーレビューフロー）へ
   - `required` または `recommend` の場合: 次のステップへ

2. **レビュー種別決定**:
   - 「レビュー種別の決定」セクションのルールに従い、実行するレビュー種別を決定
   - 結果: レビュー種別リスト（例: `[code, security]`）

3. **優先ツール決定・利用可否チェック**:
   - `tools` リストの先頭を優先ツールとする（デフォルト: `codex`）
   - **空配列チェック**: `tools = []`（空配列）の場合、`cli_available=false` として外部CLIのwhichチェックをスキップし、遷移判定へ進む
   - レビュー種別リストの**全種別**について `skill="reviewing-[type]"` が利用可能か個別に確認
   - **外部CLI可用性チェック**（`tools` が空配列でない場合のみ実行）:
     `tools` リストを先頭から順に確認し、最初にインストールされているCLIを「使用ツール」として決定する

     **ツール名のバリデーション**: `tools` リストの各値は固定ホワイトリスト（`codex` / `claude` / `gemini`）と照合する。ホワイトリストに含まれない値は警告を出してスキップする（シェルインジェクション防止）

     ```text
     which codex >/dev/null 2>&1  # codexの場合
     which claude >/dev/null 2>&1 # claudeの場合
     which gemini >/dev/null 2>&1 # geminiの場合
     ```

     - いずれかのCLIが利用可能: `cli_available=true`、使用ツール名を記録
     - 全CLIが利用不可: `cli_available=false`

   - **遷移判定**（Skill可用性 × CLI可用性の組み合わせ）:
     - 全種別利用可能 かつ `cli_available=true` → ステップ4へ
     - 全種別利用可能 かつ `cli_available=false` → ステップ7（外部AIレビューで続行不能時）へ遷移し、mode に応じた既存フローで処理
     - 一部種別利用可能 かつ `cli_available=true`:
       - `mode = "required"`: 利用不可の種別ごとにユーザーにスキップ承認を求める（承認は履歴に記録）。全て承認後、利用可能な種別のみでステップ4へ。承認拒否 → ステップ7へ
       - `mode = "recommend"`: 利用不可の種別について警告を表示し、利用可能な種別のみでステップ4へ
     - 一部種別利用可能 かつ `cli_available=false`:
       - `mode = "required"`: 利用不可の種別ごとにユーザーにスキップ承認を求める（承認は履歴に記録）。全て承認後、利用可能な種別のみでステップ7へ遷移し、mode に応じた既存フローで処理。承認拒否 → ステップ7へ
       - `mode = "recommend"`: 利用不可の種別について警告を表示し、利用可能な種別のみでステップ7へ遷移し、mode に応じた既存フローで処理
     - すべて利用不可 → ステップ7（外部AIレビューで続行不能時）へ

4. **AIレビューツール利用可能時の選択**:
   - `mode = "required"` の場合: ステップ5（AIレビューフロー）へ
   - `mode = "recommend"` の場合:
     - **セミオートゲート判定**: `automation_mode=semi_auto`（`common/rules.md` のセミオートゲート仕様を参照）の場合、ユーザー確認をスキップし自動的にステップ5（AIレビューフロー）へ
     - `automation_mode=manual` の場合: 推奨メッセージを表示しユーザーに選択を求める

     ```text
     【レビュー推奨】AIレビューツール（Skills）が利用可能です。
     品質向上のため、この成果物のレビューを実施することを推奨します。
     レビューを実施しますか？
     ```

     - 「はい」の場合: ステップ5（AIレビューフロー）へ
     - 「いいえ」の場合: ステップ8（ユーザーレビューフロー）へ

5. **AIレビューフロー**:
   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う

   - **機密情報除外スキャン**:

     レビュー対象ファイルを外部AIツールへ渡す前に、機密情報を含む可能性のあるファイルを除外する。

     **スキップ条件**: セルフレビュー（ステップ6）は外部AIへの情報送信がないため、このスキャンをスキップする。

     **デフォルト除外パターン**（常に適用）:

     | パターン | 対象 |
     |---------|------|
     | `.env*` | 環境変数ファイル |
     | `*.key` | 秘密鍵ファイル |
     | `*.pem` | 証明書・鍵ファイル |
     | `credentials.*` | 認証情報ファイル |
     | `*secret*` | シークレット関連ファイル |

     **カスタムパターン**: `.aidlc/config.toml` の `[rules.reviewing].exclude_patterns` を読み取る。

     ```toml
     [rules.reviewing]
     exclude_patterns = ["*.p12", "config/secrets/**"]
     ```

     - 未設定・空配列: デフォルトパターンのみ適用
     - 非配列型（文字列・数値等）: 警告を出力しデフォルトパターンのみ適用
     - 設定あり: デフォルトパターンに**追加**して適用（デフォルトは常に適用）

     **パターンマッチルール**:
     - デフォルトパターンおよび `/` を含まないカスタムパターン: ファイルの**ベース名**に適用（例: `credentials.*` は `config/credentials.json` のベース名 `credentials.json` にマッチ）
     - `/` を含むカスタムパターン: **フルパス**（プロジェクトルート相対）に適用（例: `config/secrets/**`）
     - 照合は**ケースインセンシティブ**で行う（例: `Credentials.JSON` も `credentials.*` にマッチ）

     **スキャン手順**:
     1. レビュー対象ファイル一覧を取得（git diff や変更ファイル一覧から）
     2. 各ファイルのベース名（またはフルパス）をデフォルト+カスタムパターンと照合
        - `/` を含まないパターン: ファイルのベース名と照合（例: `*.key` → `server.key` にマッチ）
        - `/` を含むパターン: プロジェクトルート相対パスと照合（例: `config/secrets/**` → `config/secrets/db.yml` にマッチ）
     3. マッチしたファイルをレビュー対象から除外
     4. **全ファイル除外チェック**: 除外後のファイル一覧が空（全ファイルが除外された）場合、以下の警告を出力しレビューを続行しない:

        ```text
        【警告】除外パターンにより全てのレビュー対象ファイルが除外されました。
        exclude_patterns の設定を確認してください。
        適用パターン: [適用されたパターン一覧]
        ```

        この場合、AIレビューはスキップされ、ステップ8（ユーザーレビューフロー）へ遷移する。
     5. 除外後のファイル一覧を以降のAIレビュー呼び出し引数に使用する

     **除外通知フォーマット**:

     除外ファイルがある場合、ファイルパスのみをユーザーに通知（ファイル内容は含まない）:

     ```text
     【機密情報除外】以下のファイルをレビュー対象から除外しました:
     - .env.local（パターン: .env*）
     - config/credentials.json（パターン: credentials.*）

     除外パターンは .aidlc/config.toml の [rules.reviewing].exclude_patterns でカスタマイズできます。
     ```

     除外ファイルがない場合: 通知なし（無言で続行）。

   - **種別ごとの反復レビュー**（レビュー種別リストを先頭から順に直列実行）:

     各種別で以下を実行:

     1. AIレビューを実行: `skill="reviewing-[type]", args="[レビュー対象] 優先ツール: [tool]"`

     - **種別単位のフォールバック処理**:

       スキル呼び出し時に外部CLIがエラーを返した場合、エラー種類に応じて以下の処理を行う:

       **エラー分類**:

       | エラー種類 | 判定条件 | フォールバック動作 |
       |:---|:---|:---|
       | CLI不在（恒久） | コマンド未検出（exit code 127等） | 該当種別のみセルフレビュー（ステップ6のA/B/C方式）へ遷移 |
       | CLI実行エラー（一時的可能性） | タイムアウト、認証失敗、ネットワークエラー等 | リトライ1回 → 再失敗時はユーザーに確認 |
       | CLI出力解析不能 | 応答が空、形式不一致 | 該当種別のみセルフレビューへ遷移 |

       **CLI実行エラー時のユーザー確認**:

       ```text
       【外部CLIエラー】{種別}レビューで外部CLIがエラーを返しました（リトライ後も失敗）。
       エラー内容: {エラー詳細（機密情報マスキング済み）}

       どのように対応しますか？
       1. セルフレビューへ遷移する（推奨）
       2. スキップする
       3. 処理を中断する
       ```

       - 選択肢1: 該当種別のみセルフレビューへ遷移
       - 選択肢2: 該当種別のみスキップし、次種別の処理を続行する。`mode = "required"` の場合はスキップ理由の入力と履歴記録が必要（ステップ7のスキップ手続きに準ずる）。`mode = "recommend"` の場合はそのままスキップ
       - 選択肢3: 処理を中断

       **フォールバック発生時の履歴記録**:

       各種別のフォールバック発生時に以下を履歴に記録する。**エラー詳細にはCLI stderrに含まれるAPIキー・トークン・クレデンシャル等の機密情報をマスキング**（例: `sk-****`、`Bearer ****`）してから記録すること:

       ```text
       【フォールバック】{種別}の外部CLIレビューからセルフレビューへ遷移
       【エラー種類】{CLI不在 / CLI実行エラー / 出力解析不能}
       【詳細】{エラー詳細（機密情報マスキング済み）}
       ```

       **完了済み種別の保護**:

       - 種別Aが外部CLIで成功した結果は保持する
       - 種別Bがフォールバックしても、種別Aの結果には影響しない
       - 履歴記録には各種別の実行方法（external_cli / self_review）を個別に記載する

     2. レビュー結果を確認
     3. 指摘があれば修正を反映
     4. 指摘がゼロになるまで1-3を繰り返す（最大3回）
     5. **全種別で指摘がゼロになった時点で**: 以下のメッセージを出力する

        ```text
        【AIレビュー完了】指摘0件
        ```

     6. 「AIレビュー完了時の共通履歴記録手順」に従い履歴に記録してから、レビュー後コミット→ユーザーへの承認提示ステップへ進む
     7. 3回後も指摘が残る場合は、「### 指摘対応判断フロー」を実行
     8. 判断フロー後、「修正する」を選択した指摘がある場合は以下を通知し、反復レビューの1に戻る:

        ```text
        指摘の修正を行いました。さらに最大3回のAIレビューを実施します。
        ```

   ### Codex セッション管理（Codexを選択した場合）

   **Codex Skillを使用したAIレビューでは、反復レビュー時に必ずresume機能を使用する**

   #### セッションID保持

   1. **初回レビュー実行後**: Codex出力末尾の `session id: xxx` を記録
   2. **2回目以降のレビュー**: `codex exec resume <session-id> "<request>"` を使用
      - resumeにより前回の指摘内容・コンテキストが継続される
      - `-C` や `-s` オプションは通常継承されるため省略可能

   #### 使用例

   ```text
   [1回目] codex exec -s read-only -C . "コードをレビューしてください"
           → 指摘あり、session id: abc123 を記録

   [修正を実施]

   [2回目] codex exec resume abc123 "指摘を修正しました。再度レビューしてください"
           → 前回の文脈を踏まえて継続レビュー
   ```

   #### フォールバック処理

   resume失敗時（セッション期限切れ、エラー等）は新規セッションで再開:

   ```text
   【フォールバック】セッション期限切れのため、新規セッションで再開します。
   ```

   新規セッション開始後、session idを更新して記録する。

   #### Unit切り替え時のリセット

   - 別Unitの作業を開始する際は、セッションIDを破棄
   - 新しいUnitでは新規セッションで開始する
   - 理由: 異なるUnitのコンテキストが混在することを防ぐ

   ### 指摘対応判断フロー

   AIレビューで指摘が残っている場合に、各指摘への対応を判断するフロー。

   #### 起動条件

   - AIレビュー反復（最大3回）後に指摘が残っている場合
   - ステップ5（AIレビューフロー）またはステップ6（セルフレビューフロー）の反復レビューから呼び出される

   #### 千日手検出（フロー実行前チェック）

   判断フロー実行前に、過去3回の反復レビュー結果を確認する。

   ##### 「同種の指摘」の判定基準

   以下の3つのキーがすべて実質的に一致する指摘を「同種」とみなす:

   1. **レビュー種別**: 同一のレビュー種別（code / architecture / security / inception）
   2. **対象ファイル**: 同一のファイルパス（パスが指定されていない指摘はワイルドカードとして扱い、常に一致とみなす）
   3. **指摘内容の要約**: 同一の問題を指摘している（表現の差異は無視し、指摘の本質が同じかで判定）

   **判定手順**:
   1. 各ラウンドの指摘リストを取得する
   2. ラウンド間で上記3キーを比較し、同種の指摘をグルーピングする
   3. 3ラウンドすべてに出現する同種グループが1つ以上あれば「千日手」と判定する

   **サンプルケース**:

   | ケース | ラウンド1 | ラウンド2 | ラウンド3 | 判定 |
   |--------|----------|----------|----------|------|
   | 同種 | code, src/a.ts, 「未使用変数」 | code, src/a.ts, 「未使用の変数がある」 | code, src/a.ts, 「不要な変数」 | 千日手（本質が同じ） |
   | 非同種 | code, src/a.ts, 「未使用変数」 | code, src/b.ts, 「未使用変数」 | code, src/a.ts, 「未使用変数」 | 非千日手（ラウンド2のファイルが異なる） |

6. **セルフレビューフロー**（外部AIレビューツール不可時のフロー）:

   **注意**: セルフレビューは外部ツールに比べて品質が劣る可能性があります。重要な変更では、可能な限り外部AIレビューツールの利用を推奨します。

   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う

   - **レビュー実行方式の判定**:

     セルフレビューは以下の優先順位で実行方式を決定する:

     - **A) スキル呼び出し方式（推奨）**: 対応するレビュースキルを `self-review` モードで呼び出す。スキル側がサブエージェント起動・インラインフォールバックを管理する
     - **B) インライン方式（スキル呼び出し失敗時のフォールバック）**: スキル呼び出しが失敗した場合、SKILL.mdの「レビュー観点」を読み取り、メインエージェント自身がレビューを実施する
     - **C) 簡易インライン方式（SKILL.mdも取得不可時の最終フォールバック）**: SKILL.mdが見つからない場合、review-flow.md内のレビュー種別説明を観点として使用しレビューを実施する

     **注意**: A)方式では、サブエージェント管理（起動・結果受信・インラインフォールバック）はスキル側の責務。メインエージェントはスキルからの結果を受け取り、反復制御・修正判断・履歴記録を行う。

   - **セルフレビュー実行手順**:

     **種別ごとの直列実行**: レビュー種別リストの全種別について、種別ごとに以下のA/B/C方式で実行する（ステップ5の「種別ごとの反復レビュー」と同様の直列実行）。

     #### A) スキル呼び出し方式（推奨）

     1. 対象の成果物ファイルパス一覧を `target_files` として整理する
     2. レビュー種別に対応するレビュースキルを self-review モードで呼び出す:

        ```text
        Skill ツール呼び出し:
          skill: "reviewing-{種別}"
          args: "self-review {target_files を半角スペース区切り}"
        ```

        **引数パース規則**:
        - 先頭トークン（最初の半角スペースまで）が `self-review` であればセルフレビューモード
        - `self-review` 以降の文字列をレビュー対象ファイルパスとして扱う（半角スペース区切り）
        - 空白を含むファイルパスは非対応

        **ファイルパスのバリデーション**（各パスに適用）:
        - 許可文字: `^[A-Za-z0-9._/-]+$` のみ（制御文字・改行・タブ・`:` は拒否）
        - `..` を含むパスは拒否（パストラバーサル防止）
        - 絶対パス（`/` で始まる）は拒否（リポジトリ外参照防止）
        - バリデーション失敗時はそのパスをスキップし、警告を出力する
        - 有効パスが0件の場合はセルフレビュー失敗として扱い、ユーザーに対象ファイルの再指定を求める（空リストでの「指摘0件」完了を禁止）

     3. スキルからの結果を受信し判定する:
        - **指摘なし**: 指摘0件として処理（手順7へ）
        - **指摘あり**: 各指摘から「重要度」「内容」「推奨修正」の3項目を解析。3項目が揃わない指摘は無効として除外する
        - **フォールバック条件**（以下のいずれかに該当した場合、B)方式にフォールバック。理由を履歴に記録する）:
          - スキル呼び出し失敗（スキルが見つからない、Skillツール自体の実行エラー）
          - 結果解析不能（全指摘が無効、応答が空、形式不一致）

     #### B) インライン方式（スキル呼び出し失敗時のフォールバック）

     4. レビュー種別に応じた観点を確認:
        - 対応するレビュースキルのSKILL.md（正本: `prompts/package/skills/reviewing-{種別}/SKILL.md`）を読み取り、「レビュー観点」セクションの内容を `review_criteria` として保持する（注: Skillの実行は不可でもSKILL.mdファイルはディスク上に存在するため参照可能）
        - SKILL.mdが見つからない場合: C)方式にフォールバック
     5. `review_criteria` の観点に基づいて対象ファイルをレビューし、出力フォーマットに従って記録する

     #### C) 簡易インライン方式（最終フォールバック）

     6. review-flow.md内の「レビュー種別の決定」セクション（CallerContextマッピングテーブル等）を観点として使用する
     7. 上記観点に基づいてレビューを実施し、出力フォーマットに従って記録する

     #### 反復・完了処理（方式共通）

     8. 指摘がある場合は修正を反映
        - メインエージェントが各指摘に `id`（指摘番号）と `judgement`（未対応/対応済）を付与・管理する
     9. 反復レビュー（最大3回）: 修正後に再度レビューを実施（A)方式の場合はスキルを再呼び出し。スキル呼び出しはステートレス）
     10. **全種別で指摘がゼロになった時点で**: 以下のメッセージを出力し、「AIレビュー完了時の共通履歴記録手順」に従い履歴に記録してからレビュー後コミット→ユーザーへの承認提示ステップへ進む

        ```text
        【AIレビュー完了（セルフレビュー）】指摘0件
        ```

     11. 3回後も指摘が残る場合は、「### 指摘対応判断フロー」（ステップ5と同じフロー）を実行
     12. 判断フロー後、「修正する」を選択した指摘がある場合は以下を通知し、セルフレビュー実行手順の8に戻る:

        ```text
        指摘の修正を行いました。さらに最大3回のセルフレビューを実施します。
        ```

   - **出力フォーマット**:

     指摘ありの場合:

     ```text
     【セルフレビュー結果】
     レビュー種別: {code | architecture | security | inception}

     指摘 #1
     - 重要度: {高 | 中 | 低}
     - 内容: {指摘内容の要約}
     - 推奨修正: {修正方法の提案}
     - 判定: {未対応 | 対応済}

     指摘 #2
     ...

     合計: {N}件（高: {n}件 / 中: {n}件 / 低: {n}件）
     ```

     指摘なしの場合:

     ```text
     【セルフレビュー結果】指摘0件
     レビュー種別: {種別}
     ```

     **指摘対応判断フローとの接続ルール**:
     - 判断フローへの入力は `判定: 未対応` の指摘のみ（対応済は除外）
     - 反復レビュー時、同一指摘IDの判定を `未対応 → 対応済` に更新する
     - 3回後の収束判定: `判定: 未対応` の指摘が残っている場合に指摘対応判断フローを起動

   - **外部入力検証の非適用**: セルフレビューは同一エージェントシステム内（サブエージェント含む）が実施するため、「外部入力検証ルール」の「AIレビュー応答の検証」は適用しない（自己の判断を自ら検証することは無意味であるため）。ただし以下のチェックは適用する:
     - **構造チェック**: A)方式手順3 / B)方式手順5 / C)方式手順7の「重要度」「内容」「推奨修正」3項目が揃っていること
     - **妥当性チェック**（全方式 A/B/C 共通）: メインエージェントが各指摘の修正判断時に、指摘内容の妥当性（根拠の有無、対象ファイルとの関連性、禁止操作の提案がないか等）を確認する。A)方式の場合はスキル返却結果にも同基準を適用する

   - 「レビューステップ完了時の共通処理」に従う。

7. **外部AIレビューで続行不能時**（全種別利用不可、または一部利用不可で承認拒否の場合）:
   - `mode = "required"` の場合:

     ```text
     【警告】AIレビューが必須設定ですが、外部AIレビューを継続できません。

     どのように対応しますか？
     1. セルフレビューを実施する（推奨） - エージェント自身がレビューを実施
     2. ユーザー承認へ進む - レビュースキップを履歴に記録
     3. 処理を中断
     ```

     - 選択肢1の場合: ステップ6（セルフレビューフロー）へ。セルフレビューは「必須AIレビュー充足」として扱う
     - 選択肢2の場合:

       1. スキップ理由の入力を要求:

          ```text
          AIレビュースキップの理由を入力してください。

          【注意】「指摘対応判断フロー」セクションの禁止パターンに準ずるバリデーションが適用されます。
          具体的な技術的理由や根拠を含めて記載してください。
          ```

       2. バリデーション（「指摘対応判断フロー」セクションの禁止パターンに準ずる）:
          - 空文字・空白のみは不可（再入力を要求）
          - 禁止パターンのみで構成される理由は拒否し、再入力を要求
          - heredoc終端トークン（`CONTENT_EOF`）を含む入力は拒否し、再入力を要求（heredocインジェクション防止）
          - 上記を通過した場合、受理

       3. 以下を履歴に記録してステップ8へ:

          1. Writeツールで一時ファイルを作成（内容: 履歴コンテンツ）:

          ```text
          【AIレビュースキップ】外部AIレビュー継続不能
          【スキップ理由】{ユーザーが入力した理由}
          【対象成果物】{成果物名}
          ```

          2. 以下を実行:

          ```bash
          skills/aidlc/scripts/write-history.sh \
              --cycle "{{CYCLE}}" \
              --phase "{{PHASE}}" \
              --unit "{N}" \
              --unit-name "[Unit名]" \
              --unit-slug "[unit-slug]" \
              --step "AIレビュースキップ" \
              --content-file /tmp/aidlc-history-content.txt
          ```

          3. 一時ファイルを削除

          - `{{PHASE}}`: 呼び出し元のフェーズ（`construction` または `inception`）
          - `--unit`, `--unit-name`, `--unit-slug`: constructionフェーズの場合のみ指定。inceptionフェーズではこれらの引数を省略する

     - 選択肢3の場合: 処理を中断し、ユーザー再指示待ち状態へ

   - `mode = "recommend"` の場合:

     ```text
     【レビュー推奨】AIレビューツール（Skills）が利用できませんが、セルフレビューを実施できます。
     品質向上のため、セルフレビューの実施を推奨します。
     セルフレビューを実施しますか？
     1. はい - セルフレビューを実施する（推奨）
     2. いいえ - ユーザーレビューへ進む
     ```

     - 「はい」の場合: ステップ6（セルフレビューフロー）へ
     - 「いいえ」の場合: ステップ8（ユーザーレビューフロー）へ

8. **ユーザーレビューフロー**（mode=disabled または 外部AIレビューで続行不能時）:
   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う

   - 成果物をユーザーに提示
   - ユーザーの承認を求める
   - 修正依頼があれば修正を反映
   - **レビュー後コミット**: `common/commit-flow.md` の「レビュー反映コミット」手順に従う

   - 再度ユーザーに提示・承認を求める

## レビューステップ完了時の共通処理

ステップ5（AIレビューフロー）およびステップ6（セルフレビューフロー）の完了時に実行する共通処理。

1. **レビュー後コミット**: `common/commit-flow.md` の「レビュー反映コミット」手順に従う
2. **レビューサマリ更新**【必須】: 「## レビューサマリファイル更新手順」に従いサマリファイルを更新する（計画承認前を除く）。指摘対応判断フローの「全て先送り」パスまたは千日手パスで既にレビューサマリを更新済みの場合はスキップする。

## レビューサマリファイル更新手順

AIレビュー完了時にレビューサマリファイルを生成・追記する共通手順。ステップ5（AIレビューフロー）、ステップ6（セルフレビューフロー）、および指摘対応判断フローから参照される。

### 前提条件

- AIレビュー（外部ツールまたはセルフレビュー）が実施された場合のみ実行する
- **計画承認前のレビューでは実行しない**（計画レビューは設計前段階のため対象外）
- レビュー未実施（mode=disabled、スキップ等）の場合はファイルを作成しない

### ファイルパス決定ルール

| フェーズ | 呼び出し元ステップ | ファイルパス |
|---------|------------------|------------|
| Construction | Phase 1 ステップ3（設計レビュー） | `.aidlc/cycles/{{CYCLE}}/construction/units/{NNN}-review-summary.md` |
| Construction | Phase 2 ステップ4（コード生成後） | 同上（追記） |
| Construction | Phase 2 ステップ6（統合とレビュー） | 同上（追記） |
| Inception | Intent承認前 | `.aidlc/cycles/{{CYCLE}}/inception/intent-review-summary.md` |
| Inception | ユーザーストーリー承認前 | `.aidlc/cycles/{{CYCLE}}/inception/user-stories-review-summary.md` |
| Inception | Unit定義承認前 | `.aidlc/cycles/{{CYCLE}}/inception/unit-definition-review-summary.md` |

- `{NNN}`: Unit番号（3桁ゼロパディング、例: `004`）

### 処理手順

1. フェーズと呼び出し元ステップからファイルパスを決定（上記テーブル参照）
2. ファイルの存在を確認:
   - **存在しない場合**: テンプレート（`skills/aidlc/templates/review_summary_template.md`）から新規作成し、基本情報セクションを埋める
   - **存在する場合**: ファイル末尾にセパレータ（`---`）を追記
3. 新しいセットを追記（以下のフォーマット）:

```markdown
## Set [N]: [YYYY-MM-DD HH:MM:SS]

- **レビュー種別**: [実行した全種別（例: code, security）]
- **使用ツール**: [codex / self-review(skill) / self-review(inline)]
- **反復回数**: [1〜3]
- **結論**: [指摘0件 / 指摘対応判断完了]

### 指摘一覧

| # | 重要度 | 内容 | 対応 |
|---|--------|------|------|
| 1 | 高/中/低 | [対象箇所] - [問題事象] | 修正済み（[変更箇所]: [変更内容]） / TECHNICAL_BLOCKER（理由: [理由]） / OUT_OF_SCOPE（理由: [理由]） |
```

- セット番号 `[N]` はファイル内で連番（Set 1, Set 2, ...）
- 指摘0件の場合は指摘一覧テーブルの代わりに「指摘なし」と記載
- 日時は `write-history.sh` と同様に取得

**列の記述ガイダンス**:

- **「内容」列**: `[対象箇所] - [問題事象]` の形式で記述する（ハイフン区切り）
  - 対象箇所: ファイル名・セクション名・行番号等の位置情報（必須）
  - 問題事象: 何がどう問題なのかの具体的記述（必須）
  - 例: `review-flow.md のセルフレビュー手順 - 反復回数上限が未定義で無限ループリスク`
  - NG: `命名が不適切`（対象箇所・問題事象が不明）

- **「対応」列**: 対応種別に応じた形式で記述する
  - 修正済み: `修正済み（[変更箇所]: [変更内容]）` - 変更箇所と変更内容は必須
  - TECHNICAL_BLOCKER: `TECHNICAL_BLOCKER（理由: [技術的理由の詳細]）` - 技術的理由と判断根拠は必須
  - OUT_OF_SCOPE: `OUT_OF_SCOPE（理由: [スコープ外である根拠]）` - 根拠は必須、対応予定は任意
  - 千日手によるユーザー承認: `千日手によりユーザー承認へ移行`（千日手検出時の特殊ケース）
  - 例: `修正済み（review-flow.md L640: 反復回数上限3回を明記）`
  - NG: `修正済み`（変更内容が不明）
  - **注意**: 機密情報（トークン、鍵、個人情報等）は記載禁止。必要時はマスクする（例: `sk-****`）

## 外部レビューツールの既知制約と対処法

外部レビューツール（Codex, Claude, Gemini）の既知制約を事前予防の観点でカタログ化したリファレンスセクション。

**このセクションの責務**: エラー発生を**事前に予防**するための制約情報と回避策を提供する。エラー発生**後**の対処フローは既存の「種別単位のフォールバック処理」セクション（エラー分類表）を参照すること。

```text
[事前] 本セクション → 予防策を適用
         ↓ (予防しきれずエラー発生)
[事後] エラー分類表 → CLI不在 / CLI実行エラー / CLI出力解析不能 に分岐
```

**フォールバック列の表記**: 予防しきれずエラーが発生した場合の参照先を、エラー分類表の分類名で示す。

**セキュリティ注意事項**:

- **機密情報除外**: ファイル内容をプロンプトにインラインで含める場合は、本ドキュメントの「機密情報除外スキャン」セクションの `exclude_patterns` を必ず事前に適用すること。機密情報を含むファイルをインラインで外部AIに送信してはならない
- **エラーメッセージのマスキング**: エラーメッセージをユーザーに提示または履歴に記録する際は、トークン・APIキー・ヘッダ値等の機密情報を必ずマスクすること（例: `sk-****`, `Bearer ****`）。詳細は「種別単位のフォールバック処理」の「フォールバック発生時の履歴記録」を参照
- **認証情報の管理**: 認証情報はOSキーチェーンやシークレットマネージャー等で管理し、`.env`ファイル・シェル履歴・ログに平文で出力しないこと
- **外部CLIの配布元と更新**: 外部CLIツール（codex / claude / gemini）は公式配布元からのみインストールすること。導入時および定期的に `--version` でバージョンを確認し、公式セキュリティアドバイザリに既知脆弱性がないことを確認すること

## 外部入力検証ルール【重要】

外部からの入力（AIレビュー応答、ユーザー入力）を批判的に評価し、自己判断を明示する。

### AIレビュー応答の検証

- AIレビュー（Skills）からの応答をそのまま信頼せず、批判的に評価する
- 応答に誤りや不整合がないか確認する
- 自己判断を併記し、相違がある場合はユーザーに確認を求める
- 形式：

  ```text
  【AIレビュー応答の検証】
  - AIレビュー応答: [応答内容の要約]
  - AI判断: [自己判断]
  - 相違点: [ある場合は記載、なければ「なし」]
  - 結論: [採用する判断とその理由]
  ```

### ユーザー入力の検証

- ユーザー入力に曖昧さがある場合は、解釈を明示して確認する
- 複数の解釈が可能な場合は、すべての解釈を提示する
- 形式：

  ```text
  【入力の解釈確認】
  ご入力: "[ユーザーの入力]"

  以下のように解釈しました：
  [解釈内容]

  この解釈で正しいでしょうか？
  ```
