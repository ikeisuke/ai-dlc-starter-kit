# エラーハンドリングガイド

AI-DLCにおけるエラーハンドリングの基本方針を定義する。エラーの重大度分類、共通原則、および各フェーズの代表的なエラーと復旧手順を記載する。

## Part 1: エラーポリシー

### エラー重大度レベル

| レベル | 説明 | 判断基準 | 対応方針 |
|-------|------|---------|---------|
| Critical | フロー続行不能 | データ損失リスク、成果物破損、リポジトリ状態の不整合 | 即時停止、ユーザー通知、手動復旧 |
| High | フロー品質に重大な影響 | レビュー未実施での完了、設計と実装の重大な乖離、必須設定の欠落 | 作業中断、原因特定後に再開 |
| Medium | フロー品質に軽微な影響 | 外部ツール一時的エラー、非クリティカルな検証失敗、任意設定の欠落 | 警告表示、フォールバック実行 |
| Low | 情報提供レベル | スタイル違反、推奨設定の未適用 | 記録のみ、続行 |

### 共通原則

1. **状態保全**: エラー発生時は現在の状態を保全する。破壊的操作（`git reset --hard`、ファイル削除等）をエラー復旧の手段として使用しない
2. **具体的な通知**: ユーザーへの通知は具体的なエラー内容と推奨アクションを含める
3. **フォールバック条件**: Medium以下のエラーのみフォールバック（代替手段での自動続行）を許可する。High以上のエラーは自動続行せず停止する
4. **フォールバック時の記録**: フォールバック実行時は必ず警告メッセージを出力し、履歴に記録する

### エラー通知フォーマット

エラー発生時は以下のフォーマットでユーザーに通知する。**機微情報（トークン、認証情報、秘密鍵、個人情報等）は通知・履歴記録に含めず、必ずマスクまたは除去する。**

```text
【{レベル}】{エラー概要}
- フェーズ: {Inception / Construction / Operations}
- 原因: {具体的な原因（機微情報除去済み）}
- 推奨アクション: {次に取るべき手順}
```

## Part 2: フェーズ別プレイブック

### Inception Phase

| エラー | 重大度 | 復旧手順 |
|-------|--------|---------|
| サイクルディレクトリ作成失敗 | High | **前提**: `docs/cycles/` ディレクトリが存在すること。**手順**: ディレクトリ権限を確認し、手動で `mkdir -p docs/cycles/vX.X.X` を実行する。**失敗時**: ファイルシステムの権限設定を管理者に確認する |
| GitHub Issue作成失敗 | Medium | `gh auth status` で認証状態とスコープを確認。認証失敗時は `gh auth login` を実行（必要最小限のスコープのみ付与すること）。復旧不可の場合は手動でIssueを作成する |
| 設定ファイル（`docs/aidlc.toml`）読み取り失敗 | Medium（任意設定） / High（必須設定） | 任意設定: デフォルト値で続行し警告を出力する。必須設定（`project.name` 等）: 停止しユーザーに修正を依頼する |

### Construction Phase

| エラー | 重大度 | 復旧手順 |
|-------|--------|---------|
| AIレビューツール利用不可 | Medium | **前提**: `rules.reviewing.mode` が `required` または `recommend` であること。**手順**: セルフレビューにフォールバックする。フォールバック実行を警告出力し履歴に記録する。**失敗時**: セルフレビューも不可の場合、ユーザーに手動レビューを依頼する |
| コミット失敗 | High | **前提**: 変更がステージングされていること。**手順**: `git status` で状態を確認し、原因（未ステージ、コンフリクト等）を特定する。手動コミット手順を案内する。**失敗時**: pre-commitフック失敗の場合はフック出力を確認し、修正後に再コミットする |
| squash失敗 | High | `docs/aidlc/bin/squash-unit.sh` のエラー出力とrecoveryコマンドを確認する。recoveryコマンドはユーザーが内容を確認してから実行すること。自動abortが実行された場合は元の状態に復帰済み |
| テスト失敗 | High | **前提**: テストランナーが正常にインストールされていること。**手順**: 失敗したテストの原因を特定し、修正後に再実行する。**失敗時**: テスト環境の問題の場合は依存関係の再インストールを試行する |

### Operations Phase

| エラー | 重大度 | 復旧手順 |
|-------|--------|---------|
| PR作成/マージ失敗 | High | `gh auth status` で認証状態とスコープを確認する。CI失敗の場合はログを確認し原因を特定する。手動操作手順を案内する |
| `post-merge-cleanup.sh` 失敗 | High | dry-runのエラー出力を確認する。`main_repo_path` を参照し、メインリポジトリ側で手動クリーンアップを実行する |
| バージョンタグ付け失敗 | Medium | **前提**: マージ済みコミットが存在すること。**手順**: `git tag -l` で既存タグを確認する。重複タグの場合は削除後に再作成する。**失敗時**: リモートタグが既に存在する場合はユーザーに削除確認を取る |
