# 論理設計: プレリリースバージョン対応

## 概要

`init-cycle-dir.sh` のバージョンバリデーションを緩和し、任意の文字列を許容する。

## 現状分析

### 現在のバリデーション仕様

```
入力: バージョン文字列
検証: ^v[0-9]+\.[0-9]+\.[0-9]+$ にマッチするか
出力: マッチすれば成功、しなければエラー
```

### 問題点

- プレリリースバージョン（`v2.0.0-alpha.4`）が拒否される
- 実験的なサイクル名（`experiment-1`）が使えない
- バージョン形式の制約はディレクトリ作成に本質的に不要

## 新バリデーション仕様

### 検証ルール

| チェック項目 | 条件 | エラーメッセージ |
|-------------|------|------------------|
| 空文字 | 入力が空 | `VERSION argument is required` |
| スラッシュ含有 | `/` を含む | `Version cannot contain slashes` |

### 許容される入力例

- `v1.8.2` - 従来のセマンティックバージョン
- `v2.0.0-alpha.4` - プレリリースバージョン
- `2.0.0` - v プレフィックスなし
- `feature-branch` - 任意の識別子
- `my-experiment` - 自由形式

### 拒否される入力例

- `` (空文字)
- `v1.8/test` - スラッシュ含有（パス生成で問題）

## 変更対象

### ファイル

- `prompts/package/bin/init-cycle-dir.sh`

### 関数

- `validate_version()` - バリデーションロジック変更
- `show_help()` - ヘルプメッセージ更新（任意）

## 後方互換性

- 既存の `vX.X.X` 形式は引き続き有効
- 新しい形式も追加で許容
- 破壊的変更なし

## テスト観点

1. 従来形式 `v1.8.0` が成功すること
2. プレリリース形式 `v2.0.0-alpha.4` が成功すること
3. 任意形式 `feature-test` が成功すること
4. 空文字でエラーになること
5. スラッシュ含有でエラーになること
