# 論理設計: 名前付きサイクル設定

## 概要

`rules.cycle.mode` 設定の読み取り・バリデーション・フォールバック指示をInception Phaseプロンプトに追加する論理設計。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。

## アーキテクチャパターン

プロンプト内指示パターン: AIが実行時に `read-config.sh` を呼び出し、結果を検証する。既存の `rules.branch.mode` バリデーション（inception.md ステップ7-1〜7-2）と同じパターンを踏襲。

### コンポーネント詳細

#### Step 5.5 サイクルモード確認（新規）

- **責務**: `rules.cycle.mode` の読み取り・バリデーション・フォールバック
- **依存**: `docs/aidlc/bin/read-config.sh`（既存）
- **出力**: コンテキスト変数 `cycle_mode`（後続ステップおよびUnit 003で使用）

## 処理フロー概要

### サイクルモード確認の処理フロー

**ステップ**:

1. `read-config.sh rules.cycle.mode --default "default"` を実行
2. 終了コード確認:
   - exit code 2 → 警告を出力し `"default"` にフォールバック。フロー終了
   - exit code 0 → 次へ
3. 取得した値を有効値リスト（`default` / `named` / `ask`）と照合
4. 一致 → `cycle_mode` に設定。フロー終了
5. 不一致 → 警告（値の表示 + 有効値一覧 + フォールバック告知）を出力し `"default"` にフォールバック

**関与するコンポーネント**: read-config.sh, inception.md

## 非機能要件（NFR）への対応

### パフォーマンス

- **要件**: read-config.sh呼び出しの追加遅延は無視可能レベル
- **対応策**: 1回の呼び出しのみ。結果をコンテキスト変数に保持し再呼び出しを回避

### セキュリティ

- **要件**: 設定値のバリデーション
- **対応策**: 許可値のホワイトリスト判定（有効値以外はすべて拒否 + フォールバック）

### スケーラビリティ

- **要件**: 将来的なモード値の追加が容易
- **対応策**: 有効値リストへの追加のみで対応可能な設計

### 可用性

- **要件**: 設定読み取り失敗時の動作停止防止
- **対応策**: 読み取り失敗・無効値いずれの場合も `"default"` にフォールバックし処理を継続

## 実装上の注意事項

- 警告メッセージフォーマットは `rules.branch.mode`（ステップ7-1〜7-2）の2段構成を踏襲する
- バリデーション後の `cycle_mode` の使用（モード別分岐）はUnit 003の責務であり、本Unitでは実装しない
- **STARTER_KIT_DEV分岐への対応**: 既存のStep 3で `STARTER_KIT_DEV` 判定時に「ステップ6へ進む」となっている遷移先を「ステップ5.5へ進む」に変更する。これによりスターターキット開発リポジトリでもモード読み取りが確実に実行される
