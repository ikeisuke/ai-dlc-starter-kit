# ユーザーストーリー - v1.15.0

## Epic 1: コミット履歴の整理（#187）

### ストーリー 1: Unit完了時のコミットまとめ（git環境）

**優先順位**: Must-have

As a AI-DLCを使用する開発者
I want to Unit完了時に中間コミット（レビュー前/反映コミット等）を自動的に1つのコミットにまとめたい
So that PRのコミット履歴が簡潔になり、レビュー時の可読性が向上する

**受け入れ基準**:

- [ ] Unit完了コミット直後にsquashスクリプトを実行すると、Unit作業開始以降の中間コミットが1つのUnit完了コミットにまとめられる
- [ ] squash後のコミットメッセージは `feat: [vX.X.X] Unit NNN完了 - {内容}` 形式を維持する
- [ ] `git rebase -i` などインタラクティブな操作を使用せず、`git reset --soft` + `git commit` 方式で実装する
- [ ] working treeがcleanでない場合はエラーメッセージを表示して中断する
- [ ] squash前の状態に `git reflog` で復旧可能である
- [ ] Co-Authored-Byがコミットメッセージに含まれる

**技術的考慮事項**:

- `git reset --soft {起点コミット}` で中間コミットの変更を保持したままHEADを戻し、`git commit` で再コミットする方式
- 起点コミットの特定方法（直前のUnit完了コミットまたはPhase完了コミットを検索）

### ストーリー 2: Unit完了時のコミットまとめ（jj環境）

**優先順位**: Must-have

As a jj（Jujutsu）を使用してAI-DLCで開発する開発者
I want to Unit完了時に中間コミットを自動的にまとめたい
So that git環境と同等のコミット履歴整理が実現できる

**受け入れ基準**:

- [ ] jj環境でUnit完了後にsquashスクリプトを実行すると、中間リビジョンが1つにまとめられる
- [ ] `jj squash` コマンドを使用して非インタラクティブに実行する
- [ ] squash後のリビジョンメッセージは `feat: [vX.X.X] Unit NNN完了 - {内容}` 形式を維持する
- [ ] `jj undo` で復旧可能である
- [ ] bookmarkの位置が正しく維持される

**技術的考慮事項**:

- jjのcolocateモードでの動作を前提とする
- `jj squash` は非インタラクティブで動作するため、git環境より実装が容易
- bookmark操作（`jj bookmark set`）のタイミングに注意

### ストーリー 3: プロンプトへのsquash手順組み込み

**優先順位**: Must-have

As a AI-DLCプロンプトの利用者
I want to Unit完了時の手順にsquash処理が自動的に含まれるようにしたい
So that 手動でのsquash操作を忘れずに実行できる

**受け入れ基準**:

- [ ] `construction.md` のUnit完了時の必須作業にsquashスクリプト呼び出しが含まれる
- [ ] jj環境の場合はjj用のsquash手順が自動的に選択される
- [ ] squash実行前にユーザーに確認を求める（任意でスキップ可能）
- [ ] squashスキップ時は従来通り複数コミットが維持される

**技術的考慮事項**:

- `docs/aidlc.toml` にsquash設定を追加する可能性（enabled/disabled）
- jj判定は既存の `[rules.jj].enabled` を参照

## Epic 2: Skills化に向けた全体整理（#116）

### ストーリー 4: プロンプト構造の分析と方針策定

**優先順位**: Must-have

As a AI-DLCの開発者
I want to 現在のプロンプト構造を分析し、Skills化に向けた整理方針を策定したい
So that 次サイクル以降のSkills化がスムーズに進められる

**受け入れ基準**:

- [ ] 現在のプロンプト構造（ファイル一覧、依存関係、行数）の分析ドキュメントが作成されている
- [ ] Skills化する際の分離ポイント（各フェーズの独立化に必要な変更）が特定されている
- [ ] 整理方針ドキュメントが作成され、具体的な変更内容が記載されている
- [ ] 方針がユーザーに承認されている

**技術的考慮事項**:

- 各フェーズプロンプトが `common/` を読み込む構造を維持しつつ、Skills化時に独立動作できるようにする
- Lite版との整合性も考慮する

### ストーリー 5: コミット処理の統合

**優先順位**: Should-have

I want to 分散しているコミット関連のロジックを統合したい
So that コミット処理の一貫性が保たれ、Skills化時の分離が容易になる

**受け入れ基準**:

- [ ] コミット関連の処理が `common/` 配下に集約されている
- [ ] 各フェーズプロンプトからコミット処理への参照が統一されている
- [ ] squash機能もこの統合された場所から呼び出される

**技術的考慮事項**:

- `common/review-flow.md` 内のコミット部分を分離する可能性
- 各フェーズプロンプトのコミット指示を統一フォーマットに変更

**優先順位**: Should-have
