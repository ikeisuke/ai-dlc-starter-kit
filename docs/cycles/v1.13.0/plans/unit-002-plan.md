# Unit 002 計画: Inception Phase AIレビュー導入

## 概要

Inception PhaseにAIレビューを導入し、Intent明確化の深掘りと受け入れ条件の厳格化を行い、上流工程での品質を向上させる。

## 関連Issue

- Closes #154

## 変更対象ファイル

| ファイル | 変更内容 |
|----------|----------|
| `prompts/package/prompts/inception.md` | AIレビュー呼び出しの追加、Intent質問項目強化、受け入れ条件チェック観点追加 |

**注意**: `docs/aidlc/`は直接編集禁止。`prompts/package/`を編集し、Operations PhaseでAI-DLC環境をアップグレードすることで反映される。

**運用ドキュメントについて**: `review-flow.md`は既存のAIレビューフローを定義しており、Inception Phase固有の変更は不要。`inception.md`内でAIレビューを呼び出す記述を追加することで、既存フローを再利用する。

## 実装計画

### Phase 1: 設計

このUnitはプロンプト修正のみのため、コードレベルの設計（ドメインモデル・論理設計）は省略し、変更内容の設計を行う。

#### 1.1 Intent明確化（ステップ1）の質問項目強化

**追加場所**: `### ステップ1: Intent明確化【重要】` セクション内

**追加する質問項目**:

```markdown
**Intent明確化の質問観点【推奨】**:

以下の観点で質問を行い、Intentを明確化する：

1. **目的の妥当性**
   - なぜこの機能/改善が必要か？
   - 期待する成果は何か？
   - この目的を達成しないとどうなるか？

2. **スコープの明確さ**
   - 含まれる機能は何か？
   - 明示的に除外するものは何か？
   - 境界が曖昧な部分はないか？

3. **既存機能との関連**
   - 既存の類似機能はあるか？
   - 既存機能への影響はあるか？
   - 依存関係や前提条件は何か？
```

#### 1.2 AIレビュー呼び出しの追加

**実装方法**: 既存の`review-flow.md`を参照する記述を追加。Construction Phaseと同様のパターンを使用。

**追加する記述（各ステップ完了後）**:

```markdown
**AIレビュー**: 承認前に `docs/aidlc/prompts/common/review-flow.md` に従ってAIレビューを実施すること。

**Inception固有のレビュー観点**:
- [ステップごとの観点をここに記載]
```

**追加箇所と観点**:

1. **ステップ1完了後（Intent承認前）** - `### ステップ1: Intent明確化【重要】` の末尾
   ```markdown
   **AIレビュー**: Intent承認前に `docs/aidlc/prompts/common/review-flow.md` に従ってAIレビューを実施すること。

   **Inception固有のレビュー観点**:
   - 目的・狙いが明確で妥当か
   - スコープが明確に定義されているか
   - 曖昧な表現や解釈の余地がないか
   ```

2. **ステップ3完了後（ユーザーストーリー承認前）** - `### ステップ3: ユーザーストーリー作成` の末尾
   ```markdown
   **AIレビュー**: ユーザーストーリー承認前に `docs/aidlc/prompts/common/review-flow.md` に従ってAIレビューを実施すること。

   **Inception固有のレビュー観点**:
   - INVEST原則（Independent, Negotiable, Valuable, Estimable, Small, Testable）への準拠
   - 受け入れ条件が具体的で検証可能か
   - ユーザー視点で価値が明確か
   ```

3. **ステップ4完了後（Unit定義承認前）** - `### ステップ4: Unit定義【重要】` の末尾
   ```markdown
   **AIレビュー**: Unit定義承認前に `docs/aidlc/prompts/common/review-flow.md` に従ってAIレビューを実施すること。

   **Inception固有のレビュー観点**:
   - Unit分割が適切か（独立性、凝集性）
   - 依存関係が正しく定義されているか
   - 見積もりが妥当か
   - 実装順序に矛盾がないか
   ```

#### 1.3 受け入れ条件のチェック観点追加

**追加場所**: `### ステップ3: ユーザーストーリー作成` セクション内の「受け入れ基準の書き方【重要】」の後

**追加する記述**:

```markdown
**受け入れ条件のチェック観点【必須】**:

ユーザーストーリー作成時に、以下の観点で受け入れ条件をチェックする：

| チェック項目 | 確認内容 |
|-------------|---------|
| 具体性 | 数値、状態、動作が具体的に記述されているか |
| 検証可能性 | テストで確認できる形式になっているか |
| 完全性 | 正常系・異常系の両方が網羅されているか |
| 独立性 | 他の条件と重複や矛盾がないか |
```

2. **Intent質問項目の確認**

   - 具体性、検証可能性、完全性、独立性の4項目が含まれていること
