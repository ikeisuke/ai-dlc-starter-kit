## 評価スキーマ

### 判定基準

| 判定 | 定義 | 証跡要件 |
|------|------|---------|
| 対応 | AI-DLCの当該機能が動作する仕組みがエージェント側に存在する | プロジェクト内の設定ファイル・スクリプトで確認可能、または公式ドキュメントに記載 |
| 部分対応 | 機能の一部が動作する、または代替手段で実現可能 | 制約内容と代替手段を具体的に記述 |
| 未対応 | 該当機能を実現する仕組みがエージェント側に存在しない | 公式ドキュメントまたはプロジェクト内コードで未確認であることを記述 |

### 判定単位

各機能を以下の単位で評価:

| 機能 | 判定対象 |
|------|---------|
| 設定ファイル | プロジェクトレベルの指示注入メカニズム（CLAUDE.md, .cursorrules等） |
| スキル連携 | スキルファイル（SKILL.md）の読み込み・呼び出し |
| コミット属性 | AI著者情報（Co-Authored-By）の自動付与 |
| レビュー | 外部CLIとしてのAIレビュー実行 |
| サブエージェント | 並行タスク実行・委任 |
| Plan Mode | 計画モード（読み取り専用モード） |

## 実装計画

### Phase A: 収集（事実）

**入力**: 既存資料、プロジェクト内コード

**作業**:

1. **対応状況マトリクスの作成**: 対象エージェント × 機能の対応状況を評価スキーマに基づき整理

   **対象エージェント**: Claude Code, KiroCLI, Codex CLI, Gemini CLI, Cursor, Cline, Windsurf

2. **Claude Code固有表現の抽出**: `prompts/package/` 配下のClaude Code固有表現を体系的に収集
   - ツール名参照（Writeツール、Readツール等）
   - AskUserQuestion
   - タスク管理機能（TaskCreate/TaskUpdate）
   - Plan Mode
   - `.claude/` ディレクトリ参照
   - settings.json参照

**出力**: 事実ベースのマトリクスと使用箇所一覧

### Phase B: 分析（差分）

**入力**: Phase Aの出力

**作業**:

3. **ギャップ分析**: 未対応・部分対応項目について不足内容を分析
   - 各エージェントの制約と代替手段
   - プロンプト互換性の課題
   - 影響範囲の評価

**出力**: エージェント別のギャップ一覧と影響度

### Phase C: 提案（方針）

**入力**: Phase A・Bの出力

**作業**:

4. **次期サイクルへの優先対応提案**: 調査結果に基づき、優先度（高/中/低）と理由を付けて提案を作成

**出力**: 優先度付きの対応提案リスト
