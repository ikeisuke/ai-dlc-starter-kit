# 論理設計: マルチプラットフォーム対応調査

## 概要

マルチプラットフォーム対応状況の調査結果を `multi-platform-analysis.md` として出力するためのドキュメント構成とデータ収集方法を定義する。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。

## アーキテクチャパターン

3層パイプライン構成: 収集（Phase A）→ 分析（Phase B）→ 提案（Phase C）。各層は前層の出力を入力として受け取る。

## データ収集インターフェース

### Phase A: 収集

#### 対応状況マトリクスの入力ソース

| ソース | 取得方法 | 取得対象 |
|--------|---------|---------|
| 既存分析 (`existing_analysis.md`) | ファイル読み取り | マルチプラットフォーム対応の現状テーブル |
| AIエージェント許可リスト (`ai-agent-allowlist.md`) | ファイル読み取り | セクション4: AIエージェント別設定方法 |
| Amazon AI-DLCレポート (`amazon-aidlc-report.md`) | ファイル読み取り | 対応プラットフォームテーブル |
| プロジェクト内設定ファイル | ファイル存在確認 | `.claude/`, `.kiro/`, `.cursorrules` 等 |

#### Claude Code固有表現の収集方法

`prompts/package/` 配下のファイルを走査し、以下のパターンを検出:

| カテゴリ | 検出パターン | 例 |
|---------|------------|-----|
| ツール名 | `Writeツール`, `Readツール`, `Grepツール`, `Globツール`, `Editツール` | ファイル操作指示 |
| 質問機能 | `AskUserQuestion` | ユーザーへの質問 |
| タスク管理 | `TaskCreate`, `TaskUpdate`, `TaskList`, `TaskGet`, `TodoWrite` | 進捗管理 |
| 計画モード | `Plan Mode`, `EnterPlanMode`, `ExitPlanMode` | 設計フェーズ |
| ディレクトリ | `.claude/`, `CLAUDE.md`, `settings.json`, `settings.local.json` | 設定参照 |
| スキル実行 | `Skill(`, `/skill-name` | スキル呼び出し |

### Phase B: 分析

**入力**: Phase Aで収集したマトリクスデータと固有表現一覧

**分析観点**:

1. **機能カバレッジ**: 各エージェントが何割の機能をカバーしているか
2. **代替手段の有無**: 未対応機能に対する代替手段の実現可能性
3. **プロンプト互換性**: 共有プロンプト内の固有表現がどの程度他エージェントに影響するか

**出力**: エージェント別ギャップ一覧（未対応/部分対応の項目、影響度、代替手段）

### Phase C: 提案

**入力**: Phase Bのギャップ分析結果

**提案の優先度判定基準**:

| 優先度 | 基準 |
|--------|------|
| 高 | 3つ以上のエージェントに影響し、代替手段なし |
| 中 | 2つ以上のエージェントに影響、または代替手段が複雑 |
| 低 | 1つのエージェントのみ影響、または簡易な代替手段あり |

**出力**: 優先度付き提案リスト（タイトル、優先度、理由、影響エージェント、概算規模）

## 実装上の注意事項

- 調査結果はv1.21.0時点のスナップショットであり、エージェントのバージョンアップにより変化しうる
- `prompts/package/` のファイルを調査対象とし、`docs/aidlc/`（ミラー）は参照しない
- 対応状況の判定は計画ファイルの評価スキーマに厳密に従う
- 提案は具体的な実装方法ではなく、方向性と優先度の提示に留める（実装は別サイクル）
