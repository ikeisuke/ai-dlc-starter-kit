# Unit 002 計画: PRマージ前のリモート同期確認ステップ追加

## 概要

Operations Phaseプロンプトに、PRマージ前のローカル/リモート同期確認ステップ `6.6.6 リモート同期確認` を追加する。Issue #196 で報告された、push漏れによるPRマージ時のデータ損失リスクを防止する。

## 変更対象ファイル

| ファイル | 変更内容 |
|----------|----------|
| `prompts/package/prompts/operations.md` | ステップ 6.6.6 を追加（6.6.5と6.7の間）+ 6.6.5の次ステップ参照を更新 |

## 実装計画

### Phase 1: 設計

1. **ドメインモデル設計**: リモート同期確認のロジックフロー定義
   - upstream tracking branch (`@{u}`) を第一優先で使用し、未設定時は `origin/{current_branch}` にフォールバック
   - `git fetch` でリモート最新化
   - `git log @{u}..HEAD`（または `git log origin/{branch}..HEAD`）で未pushコミット検出
   - `git fetch` 失敗時はマージ停止 + ユーザーに対処を提示
   - upstream未設定かつ `origin/{branch}` も存在しない場合は警告を出して手動確認を促す
   - `gh:available` 依存なし: gitコマンドのみ使用するため常に実行（6.6.5と同じパターン）

2. **論理設計**: ステップの挿入位置とプロンプト構造定義
   - 挿入位置: ステップ 6.6.5（コミット漏れ確認）の直後、6.7（PRマージ）の直前
   - サブステップ一覧への追記（`10. 6.6.6 リモート同期確認` を挿入、6.7は11番に繰り下げ）
   - ステップ本文の追加
   - 6.6.5本文中の「次のステップ（6.7 PRマージ）」参照を「次のステップ（6.6.6 リモート同期確認）」に更新

### Phase 2: 実装

3. **operations.md 編集**:
   - サブステップ一覧に `6.6.6 リモート同期確認` を追加
   - ステップ本文セクションを追加（6.6.5の後、6.7の前）
   - 6.6.5の次ステップ参照を更新

### スコープ外の判断

Issue #196 では「破壊的操作の防止ルール」の `common/rules.md` への追加も提案されているが、Unit 002の境界定義（「他フェーズのプロンプト変更は含まない」）に基づき、本Unitのスコープ外とする。必要に応じてバックログに記録する。
