# 論理設計: CHANGELOG作成

## 概要
CHANGELOG.md に v1.1.0〜v1.6.0 の変更履歴を追加する際の方針と構造を定義する。

## CHANGELOG.md の構造

### ファイル構成
```markdown
# Changelog

[ヘッダー説明文]

---

## [Unreleased]
[今後リリース予定の変更]

## [1.6.0] - YYYY-MM-DD
### Added
### Changed
### Fixed

## [1.5.4] - YYYY-MM-DD
...

## [1.0.1] - 2025-11-28
[既存エントリ]

## [1.0.0] - 2025-11-27
[既存エントリ]
```

### エントリ順序
- 新しいバージョンが上、古いバージョンが下
- v1.6.0 → v1.5.4 → ... → v1.1.0 → v1.0.1 → v1.0.0

### カテゴリ順序
各バージョン内のカテゴリ順序:
1. Added
2. Changed
3. Deprecated
4. Removed
5. Fixed
6. Security

### 記載粒度
- **粗粒度**: 各変更項目を1行で簡潔に記載
- **ユーザー視点**: 技術的詳細よりもユーザーへの影響を重視
- **重複排除**: 複数バージョンで同じ機能が拡張された場合、各バージョンでの変更点のみ記載

## リリース日の決定方針

### 情報ソース優先順位
1. Operations Phase 完了時の履歴日時（`history/operations.md`）
2. 該当サイクルの最終コミット日時
3. 次バージョンサイクルの開始日の前日

### v1.6.0 の日付
- まだリリースされていないため、リリース時に確定
- 一時的に `YYYY-MM-DD` とするか、現在日付を使用

## 既存 CHANGELOG.md との統合

### 既存内容
- v1.0.0: 2025-11-27
- v1.0.1: 2025-11-28

### 追加方法
1. 既存の v1.0.1 エントリの前に新バージョンを挿入
2. ヘッダー部分は維持
3. `[Unreleased]` セクションは追加しない（必要に応じて後で追加）

## 実装手順

1. CHANGELOG.md を読み込む
2. v1.0.1 の直前に v1.1.0〜v1.6.0 のエントリを挿入
3. 各バージョンのリリース日を決定
4. フォーマットの整合性を確認

- [ ] 各カテゴリの順序が統一
- [ ] Markdown 構文が正しい
