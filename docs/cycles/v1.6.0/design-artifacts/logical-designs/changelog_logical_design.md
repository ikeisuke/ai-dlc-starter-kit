# 論理設計: CHANGELOG作成

## 概要
CHANGELOG.md に v1.1.0〜v1.6.0 の変更履歴を追加する際の方針と構造を定義する。

## CHANGELOG.md の構造

### ファイル構成
```markdown
# Changelog

[ヘッダー説明文]

---

## [Unreleased]
### Added
### Changed
### Fixed
（v1.6.0 の変更内容。リリース時に日付を付与して正式バージョンに変更）

## [1.5.4] - YYYY-MM-DD
...

## [1.0.1] - 2025-11-28
[既存エントリ]

## [1.0.0] - 2025-11-27
[既存エントリ]
```

### エントリ順序
- 新しいバージョンが上、古いバージョンが下
- [Unreleased] → v1.5.4 → v1.5.3 → v1.5.2 → v1.5.1 → v1.5.0 → v1.4.1 → v1.4.0 → v1.3.2 → v1.3.1 → v1.3.0 → v1.2.3 → v1.2.2 → v1.2.1 → v1.2.0 → v1.1.0 → v1.0.1 → v1.0.0
- **全バージョンを降順で網羅すること**

### カテゴリ順序
各バージョン内のカテゴリ順序:
1. Added
2. Changed
3. Deprecated
4. Removed
5. Fixed
6. Security

### 記載粒度
- **粗粒度**: 各変更項目を1行で簡潔に記載
- **ユーザー視点**: 技術的詳細よりもユーザーへの影響を重視
- **重複排除**: 複数バージョンで同じ機能が拡張された場合、各バージョンでの変更点のみ記載

## リリース日の決定方針

### 情報ソース優先順位
1. Operations Phase 完了時の履歴日時（`history/operations.md`）
2. 該当サイクルの最終コミット日時
3. 次バージョンサイクルの開始日の前日

### v1.6.0 の扱い
- まだリリースされていないため、`[Unreleased]` セクションに記載
- Operations Phase 完了時（リリース時）に `[1.6.0] - YYYY-MM-DD` 形式に変更

## 既存 CHANGELOG.md との統合

### 既存内容
- v1.0.0: 2025-11-27
- v1.0.1: 2025-11-28

### 追加方法
1. 既存の v1.0.1 エントリの前に新バージョンを挿入
2. ヘッダー部分は維持
3. v1.6.0 は `[Unreleased]` セクションとして追加（リリース時に正式バージョンに変更）

## 実装手順

1. CHANGELOG.md を読み込む
2. v1.0.1 の直前に v1.1.0〜v1.6.0 のエントリを挿入
3. 各バージョンのリリース日を決定
4. フォーマットの整合性を確認
