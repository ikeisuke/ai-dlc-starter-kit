# 論理設計: コンテキストリセット改善

## 概要

3つのフェーズプロンプトファイル（construction.md, inception.md, operations.md）の完了フロー構造を変更し、コンテキストリセット指示の省略防止とセッションサマリの追加を実現する。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。具体的なプロンプト文言はImplementation Phase（コード生成ステップ）で作成します。

## アーキテクチャパターン

**プロンプトテンプレートパターン**: 各フェーズプロンプトは独立したファイルとして管理され、共通構造（完了フロー → リセット提示）を持つ。変更は「共通パターンの適用 + フェーズ固有カスタマイズ」の2層で行う。

## コンポーネント構成

### ファイル構成

```text
prompts/package/prompts/
├── construction.md     # Unit完了フローの変更
├── inception.md        # Phase完了フローの変更
├── operations.md       # サイクル完了フローの変更
└── common/
    └── context-reset.md  # 変更しない（ユーザー主導リセット）
```

### リセット種別の責務境界【重要】

本設計で扱うリセットと、`common/context-reset.md` が扱うリセットは異なる責務を持つ:

| 種別 | 責務 | トリガー | 担当ファイル |
|------|------|---------|-------------|
| **完了時リセット** | Unit/Phase/サイクル完了後の次フェーズ遷移 | AIが完了フローの最終ステップで自動提示 | 各フェーズプロンプト（本設計の変更対象） |
| **中断時リセット** | 作業途中でのコンテキスト溢れ対策 | ユーザーが「リセットしたい」等の発言 | `common/context-reset.md`（変更なし） |

**注意**: `construction.md` と `operations.md` は `【次のアクション】` ディレクティブで `common/context-reset.md` を参照読み込みしている。これはAIが中断時リセット対応を認識するためであり、完了時リセットとは独立した仕組み。`inception.md` には同様の参照は含まれていない（中断時リセットは `common/context-reset.md` を直接参照する他の仕組みで対応）。本設計では完了時リセットのみを変更し、中断時リセットの参照構造には一切触れない。

### コンポーネント詳細

#### construction.md の変更箇所

- **責務**: Unit完了時のリセット提示を必須化し、セッションサマリ生成指示を追加
- **変更セクション**:
  1. `## 完了基準` - リセット提示を必須項目として追加
  2. `### 6. コンテキストリセット【必須】` - 「次のステップ」セクションを統合
  3. `## 次のステップ【コンテキストリセット必須】` - 削除（Step 6に統合）

#### inception.md の変更箇所

- **責務**: Phase完了時のリセット提示を番号付き必須ステップ化し、セッションサマリ生成指示を追加
- **変更セクション**:
  1. `## 完了基準` - リセット提示を必須項目として追加
  2. `## 完了時の必須作業【重要】` - Step 6 を追加
  3. `## 次のステップ【コンテキストリセット必須】` - 削除（Step 6に統合）

#### operations.md の変更箇所

- **責務**: サイクル完了時のリセット提示の必須性を強調し、セッションサマリ生成指示を追加
- **変更セクション**:
  1. `## 完了基準` - リセット提示を必須項目として追加（construction/inceptionと同一パターン）
  2. `## 完了時の確認【重要】` - リセット提示を確認項目として追加
  3. `### 6. 次のサイクル開始` - 【必須】タグ追加、省略防止注記追加

## セッションサマリの設計

### 共通最小テンプレート（3ファイル共通）

リセットメッセージ内にAIが動的に生成する3行サマリ:

```text
**セッションサマリ**:
- **完了**: {{CYCLE}} / [完了作業名]
- **リポジトリ**: [ブランチ名]、[PR状態/コミット状態]
- **次のアクション**: [次のアクション指示]
```

### フェーズ固有の値

| フェーズ | 完了作業名 | 次のアクション |
|---------|-----------|--------------|
| construction（次Unit残） | Unit NNN [Unit名] | 「コンストラクション進めて」で次のUnitを開始 |
| construction（全Unit完了） | Construction Phase（全Unit完了） | 「オペレーション進めて」でOperations Phaseを開始 |
| inception | Inception Phase | 「コンストラクション進めて」でConstruction Phaseを開始 |
| operations | サイクル {{CYCLE}} | 「start setup」で次のサイクルを開始 |

### サマリ生成指示

プロンプト内にAIへの生成指示として以下を記載:

```text
AIが以下の情報を収集してセッションサマリを生成してください:
1. サイクル番号（{{CYCLE}}）と完了した作業名（Unit名またはPhase名）
2. 現在のブランチ名（git branch --show-current）とPR/コミット状態
3. 次に実行すべきアクション
```

## 非機能要件（NFR）への対応

Unit定義の通り、パフォーマンス/セキュリティ/スケーラビリティ/可用性はすべて該当なし。

## 実装上の注意事項

- `prompts/package/prompts/` 配下を編集し、`docs/aidlc/` は直接編集しない
- `common/context-reset.md`（ユーザー主導リセット）は変更対象外
- ストーリー3（省略防止）を先に完了し、その後ストーリー4（サマリ追加）を実装
- 既存のリセットメッセージの構造（markdownコードブロック内）を維持
- operations.mdの `${SETUP_PROMPT}` 置換ロジックはそのまま維持
