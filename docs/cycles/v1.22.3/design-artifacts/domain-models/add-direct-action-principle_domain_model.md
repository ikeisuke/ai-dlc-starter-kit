# ドメインモデル: 直接実行優先原則の追加

## 概要

`rules.md` に新規セクション「直接実行優先原則」を追加し、AIエージェントの行動指針として3つの原則を明文化する。

**重要**: このドメインモデル設計では**コードは書かず**、構造と責務の定義のみを行います。

## 原則定義

### 原則1: 直接実行優先（Prefer Direct Action）

- **定義**: タスクを依頼されたら直接実行する。スキルlookupや間接的なアクションを挟まない
- **適用場面**: ユーザーからの明示的なタスク依頼時

### 原則2: バックログ後回し禁止（No Deferring to Backlog）

- **定義**: ユーザーの明示的承認なしにタスクをバックログに登録して先に進まない
- **適用場面**: ユーザーが即時実行を期待しているタスク

### 原則3: 最小複雑性の原則（Minimum Complexity）

- **定義**: 複数のアプローチがある場合、最もシンプルなものを選ぶ
- **適用場面**: 技術的な選択肢がある場合

## 既存ルールとの関係整理

### Overconfidence Prevention原則との関係

- **競合条件**: 要件が曖昧な場合
- **優先順位**: Overconfidence Prevention原則が優先
- **適用順**: まず要件の明確さを判定 → 明確なら直接実行優先 → 曖昧ならOverconfidence Prevention原則に従い質問
- **理由**: 「誤った直接実行」は「質問して確認」よりリスクが高い

### 改善提案のバックログ登録ルールとの関係

- **区別**: 直接実行優先原則は「ユーザーから依頼されたタスク」の実行方法に関する規則。改善提案のバックログ登録ルールは「ユーザーから依頼されていない改善提案」を行う際の規則
- **適用判定**: ユーザーが依頼したタスクか、AIが自発的に提案した改善か
- **共存**: 両ルールは適用場面が異なるため競合しない

### フェーズ固有ゲートとの関係（スコープ外）

- **ユーザーの承認プロセス**: フェーズワークフローで定義された承認ゲート。直接実行優先原則はこれらのゲートを上書きしない
- **セミオートゲート仕様**: 自動承認のフロー制御。直接実行優先原則のスコープ外
- **Depth Level仕様**: 成果物詳細度の制御。直接実行優先原則のスコープ外
- **原則**: 直接実行優先原則は「タスクへのアプローチ選択」に関する原則であり、フェーズ固有のワークフロー制御・承認ゲートには影響しない

## ユビキタス言語

- **直接実行優先**: タスクを依頼されたら、中間ステップを最小化してすぐに実行する原則
- **バックログ後回し禁止**: ユーザーの明示的承認なしにタスクをバックログに登録して先に進まない原則
- **最小複雑性**: 複数のアプローチがある場合、最もシンプルなものを選ぶ原則
- **間接的アクション**: スキルlookup、バックログ登録、過度な計画策定等、実行前に不要なステップを挟むこと
