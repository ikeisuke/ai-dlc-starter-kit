# 論理設計: 直接実行優先原則の追加

## 概要

`prompts/package/prompts/common/rules.md` に新規セクション「直接実行優先原則」を追加する。配置位置、構成、既存ルールとの関係を定義する。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。

## セクション配置位置

`rules.md` の既存構造:

```text
rules.md
├── 設定読み込み【重要】
├── ユーザーの承認プロセス【重要】
├── 質問と回答の記録【重要】
├── Overconfidence Prevention原則【重要】
├── Gitコミットのルール
├── アップグレードチェック設定
├── Depth Level仕様【重要】
├── セミオートゲート仕様【重要】
├── 改善提案のバックログ登録ルール【重要】
└── コード品質基準
```

**配置位置**: 「Overconfidence Prevention原則」セクションの直後、「Gitコミットのルール」セクションの直前

**理由**: Overconfidence Prevention原則は「質問すべき場面」を定義し、直接実行優先原則は「質問不要な場面では直接実行」を定義する。両者は相補的な関係にあり、連続して配置することで適用判定のフローが明確になる

## セクション構成

```text
## 直接実行優先原則【重要】
├── 原則の概要（1-2文）
├── 3つの原則
│   ├── 1. 直接実行優先
│   ├── 2. バックログ後回し禁止
│   └── 3. 最小複雑性
├── 既存ルールとの関係
│   ├── Overconfidence Prevention原則との適用順
│   └── 改善提案のバックログ登録ルールとの区別
└── 適用判定フロー（テキストまたはテーブル）
```

## 適用判定フロー

```text
タスク依頼を受けた場合:
1. 要件は明確か？
   → 不明確 → Overconfidence Prevention原則に従い質問する
   → 明確 → 次へ
2. ユーザーが依頼したタスクか、AIが自発的に提案した改善か？
   → AI自発的提案 → 改善提案のバックログ登録ルールに従う
   → ユーザー依頼 → 直接実行優先原則を適用
3. 複数のアプローチがあるか？
   → ある → 最もシンプルなものを選択
   → ない → そのまま実行
```

## 背景

Claude Code Insightsレポートで最大のフリクション源が「Wrong Approach」（14件/40%）。主なパターン: 間接的なスキルlookupの経由、バックログへの後回し、複雑すぎるアプローチの選択。

## スコープ制約

直接実行優先原則は「タスクへのアプローチ選択」に関する原則であり、以下のフェーズ固有ゲートには影響しない:
- ユーザーの承認プロセス（計画承認、設計承認等）
- セミオートゲート仕様（自動承認フロー制御）
- Depth Level仕様（成果物詳細度制御）

## 実装上の注意事項

- 既存ルールの文言は一切変更しない
- 新規セクションの追加のみ
- 【重要】タグを付与して他の重要ルールと同等の位置づけとする
