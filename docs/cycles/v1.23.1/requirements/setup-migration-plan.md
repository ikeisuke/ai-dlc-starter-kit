## 技術的課題

### 1. 対話フローの移行困難性

**課題**: setup-prompt.mdは対話的なプロンプトとして設計されており、ユーザーとの対話（質問→回答→次ステップ）が多い。スキルは比較的定型的な処理向き。

**影響**: S07（プロジェクト情報収集）、S08（プロジェクトタイプ選択）は対話的なフローが必須であり、スキルへの移管が困難。

**対策**: 初回セットアップの対話フローはsetup-prompt.mdに残存させ、スキルにはアップグレード（定型処理）のみを移管する。

### 2. スキルの起動コンテキスト

**課題**: aidlc-setupスキルはプロジェクト内から起動される前提だが、setup-prompt.mdは外部（スターターキット）から読み込まれることがある。

**影響**: S03（実行環境確認）のメタ開発モード判定や、S10（8.1.1）のスターターキットパス判定がスキルのコンテキストと合わない可能性がある。

**対策**: スキルは「プロジェクト内にdocs/aidlc/が存在する」ことを前提とし、パス解決はaidlc-setup.sh内で処理する（現状通り）。

### 3. aidlc-setup.shとの責務重複

**課題**: 現在のaidlc-setup.shは既にアップグレードの主要処理（rsync同期、バージョン更新）を担当しており、スキルのSKILL.mdはそのラッパーとして動作している。

**影響**: S10の処理は既にスクリプトに移管済みであり、SKILL.mdの変更は主にフロー記述の拡充になる。

**対策**: SKILL.mdは薄い入口（起動条件判定と委譲のみ）に留め、フロー制御はaidlc-setup.sh側に一本化する。設定マイグレーションやIssueテンプレート同期等の追加処理もスクリプトに統合し、SKILL.mdからはスクリプトを呼び出すだけの構造にする。

### 4. 後方互換性

**課題**: setup-prompt.mdからアップグレードフローを削除すると、旧バージョンのAI-DLCユーザーがsetup-prompt.mdを直接読み込んだ際にアップグレードできなくなる。

**影響**: ケースC（アップグレード）でsetup-prompt.mdを読み込んだユーザーが、スキル未インストール環境で行き詰まる可能性。

**対策**: setup-prompt.mdにはケースCのリダイレクト処理（「aidlc-setupスキルを使用してください」）を残し、フォールバックとしてスクリプト直接実行の案内も含める。

### 5. 移行モード（ケースE）の扱い

**課題**: ケースEは旧形式からの移行であり、今後新規発生する可能性は低い（旧バージョンユーザーの減少に伴い）。

**影響**: 移行処理をスキルに移管する投資対効果が低い。

**対策**: ケースEはsetup-prompt.mdに残存させ、将来的に非推奨化・削除を検討する。

## 設計原則: 依存方向

移管後のレイヤー構造と依存方向:

```
setup-prompt.md（判定器）
  ├── ケースA/E → 初回セットアップフロー（setup-prompt.md内で完結）
  ├── ケースB/D → inception.mdへリダイレクト
  └── ケースC → aidlc-setupスキルに委譲
                    └── SKILL.md（薄い入口: 起動条件と委譲のみ）
                          └── aidlc-setup.sh（単一オーケストレータ: フロー制御）
                                ├── sync-package.sh（ファイル同期）
                                ├── migrate-config.sh（設定マイグレーション）
                                ├── setup-ai-tools.sh（AIツール設定）
                                └── その他スクリプト（実処理）
```

- **setup-prompt.md**: 判定の正本。全ケースの分岐判定を担当
- **SKILL.md**: 起動条件の定義と委譲のみ。ドメインロジックを持たない
- **aidlc-setup.sh**: アップグレードフローの単一オーケストレータ。フロー制御を一元管理
- **個別スクリプト**: 実処理のみ担当。フロー制御を持たない
