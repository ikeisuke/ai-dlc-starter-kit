# ドメインモデル: progress.md更新タイミング修正

## 概要

Construction Phaseの「Unit完了時の必須作業」セクション内の3つのステップにおける進捗状態管理の明確化。

## エンティティ

### Unit完了フロー

Construction PhaseのUnit完了時の必須作業は以下の順序で実行される:

```text
ステップ0-0.6 (検証系)
  → ステップ1: Unit定義ファイルの実装状態更新 ← 変更対象
  → ステップ1.5-3: Issue更新・履歴・Lint
  → ステップ4: Gitコミット ← 変更対象（チェックリスト）
  → ステップ5: Unit PR作成・マージ ← 変更対象（注意事項追加）
  → ステップ6: コンテキストリセット
```

### 変更対象のステップ

| ステップ | 現在の責務 | 追加する責務 |
|----------|-----------|-------------|
| ステップ1 | Unit定義ファイルの実装状態を「完了」に更新 | 「完了 = PR準備完了」の解釈を明記 |
| ステップ4 | 変更ファイルのコミット | チェックリストから不要な参照を削除 |
| ステップ5 | Unit PRの作成・マージ | PR作成後の変更禁止ルールを追加 |

## 値オブジェクト

### 実装状態の解釈

| 状態値 | 意味（通常） | 意味（Unitブランチ使用時） |
|--------|-------------|-------------------------|
| 未着手 | 未開始 | 未開始 |
| 進行中 | 実装作業中 | 実装作業中 |
| 完了 | 実装完了 | **PR準備完了**（Operations Phase ステップ6.4.5と同一解釈） |

## 集約ルート

**Unit完了時の必須作業セクション**（`prompts/package/prompts/construction.md`内）がこの変更の集約ルート。3つの変更はすべてこのセクション内で完結する。

## 不変条件

1. ステップ1の実装状態更新は、ステップ4のGitコミットより前に実行される（既存の順序を維持）
2. ステップ4のGitコミットは、ステップ5のPR作成より前に実行される（既存の順序を維持）
3. Unitブランチを使用しない場合の既存動作に影響を与えない
