# ドメインモデル: バージョンファイル更新のプロジェクト固有移動

## 概要

Operations Phaseプロンプトの「バージョンファイル更新」ロジックを、標準プロンプト（operations.md）からプロジェクト固有ルール（rules.md）に移動する。これにより、標準プロンプトからプロジェクト固有ロジックを除去し、責務を適切に分離する。

**重要**: このドメインモデル設計ではコードは書かず、構造と責務の定義のみを行います。

## エンティティ（プロンプト構成要素）

### operations.md（標準プロンプト）

- **責務**: 全プロジェクト共通のOperations Phaseワークフローを定義
- **属性**:
  - ステップ6「リリース準備」: サブステップ一覧とその詳細手順
- **変更内容**:
  - ステップ6.0「バージョンファイル更新（AI-DLCスターターキットのみ）」を削除
  - 削除後もステップID（6.1, 6.2, 6.3, 6.4, 6.4.5, 6.5, 6.6, 6.6.5, 6.7）は維持

### rules.md（プロジェクト固有ルール）

- **責務**: プロジェクト固有のカスタムワークフローと制約を定義
- **属性**:
  - カスタムワークフローセクション: プロジェクト固有の手順を記載
- **変更内容**:
  - 「バージョンファイル更新」ワークフローを追加
  - 実行タイミング: `/upgrading-aidlc` の後、CHANGELOG更新の前

## 値オブジェクト

### バージョンファイル更新ロジック

- **保証する規則**: サイクル番号と一致するバージョン値がversion.txtとaidlc.tomlに反映されること
- **構成要素**:
  - version.txt の更新（サイクル番号からvプレフィックスを除去した値）
  - docs/aidlc.toml の `starter_kit_version` 更新（同値）
  - 更新結果の妥当性確認
- **不変性**: ロジック自体は移動前後で同一（`project.name` の条件判定のみ除去）

## ユビキタス言語

- **標準プロンプト**: `prompts/package/` 配下の全プロジェクト共通テンプレート
- **プロジェクト固有ルール**: `docs/cycles/rules.md` に記載されるプロジェクト固有の制約・ワークフロー
- **カスタムワークフロー**: rules.mdに定義される、Operations Phase実行時にAIが参照するプロジェクト固有手順
- **ステップID**: サブステップの識別子（6.1, 6.2 等）。外部から参照されるため安定化させる
