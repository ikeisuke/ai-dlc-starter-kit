# 論理設計: バージョンファイル更新のプロジェクト固有移動

## 概要

operations.mdのステップ6.0「バージョンファイル更新」セクションを削除し、同等のロジックをrules.mdのカスタムワークフローに追加する。

**重要**: この論理設計ではコードは書かず、変更箇所とインターフェース定義のみを行います。

## コンポーネント構成

### 変更対象ファイル

```text
prompts/package/prompts/
└── operations.md          # ステップ6.0セクション削除

docs/cycles/
└── rules.md               # カスタムワークフローにバージョンファイル更新追加
```

### コンポーネント詳細

#### prompts/package/prompts/operations.md

- **責務**: 全プロジェクト共通のOperations Phaseフロー定義
- **変更箇所1**: サブステップ一覧（577行目）
  - 「1. 6.0 バージョンファイル更新（AI-DLCスターターキットのみ）」行を削除
  - 残りのMarkdownリスト番号を繰り上げ（2→1, 3→2, ...）
  - ステップID（6.1, 6.2, 6.3, 6.4, 6.4.5, 6.5, 6.6, 6.6.5, 6.7）は変更しない
- **変更箇所2**: ステップ6.0セクション本体（591-623行目）
  - `#### 6.0 バージョンファイル更新（AI-DLCスターターキットのみ）` セクション全体を削除
  - 削除範囲: 591行目（`#### 6.0`）から624行目（`#### 6.1` の直前の空行）まで

#### docs/cycles/rules.md

- **責務**: プロジェクト固有のルールとカスタムワークフロー定義
- **変更箇所**: カスタムワークフローセクション
  - 既存の「Operations Phase 完了時の必須作業」の後に新セクション追加
  - 見出し: `### バージョンファイル更新【重要】`
  - 実行タイミングの明示: `/upgrading-aidlc` 実行後、ステップ6.1（CHANGELOG更新）の前
  - 内容: version.txt更新、docs/aidlc.toml更新、更新確認の3ステップ
  - `project.name` の条件判定は不要（rules.mdはプロジェクト固有のため）

## 処理フロー概要

### Operations Phase ステップ6 の実行フロー（変更後）

**ステップ**:

1. AIがrules.mdを読み込み、カスタムワークフローを確認
2. `/upgrading-aidlc` を実行（既存）
3. **バージョンファイル更新を実行（新規: rules.mdから読み取り）**
4. ステップ6.1 CHANGELOG更新へ進む（既存）

**関与するコンポーネント**: operations.md, rules.md
