# ユーザーストーリー

## Epic 1: コミット/squash管理の強化

### ストーリー 1: Inception Phase完了時のsquash (#234)
**優先順位**: Must-have

As a AI-DLC開発者
I want to Inception Phase完了時に中間コミットを1つのfeatコミットにsquashしたい
So that コミット履歴がクリーンになり、サイクルの変更追跡が容易になる

**受け入れ基準**:
- [ ] `commit-flow.md` にInception Phase完了時のsquash統合フローが追加されている
- [ ] squash設定確認（`rules.squash.enabled`）→ VCS判定 → ユーザー確認 → squash実行のフローが記載されている
- [ ] `inception.md` の完了時手順（ステップ5: Gitコミット）がsquash統合フローを参照している
- [ ] squash実行時の起点コミットはサイクルブランチ分岐点（`git merge-base origin/main HEAD`等）を使用する
- [ ] squash後のコミットメッセージは `feat: [{{CYCLE}}] Inception Phase完了 - {DESCRIPTION}` 形式
- [ ] squash失敗時は既存のエラーリカバリフロー（recoveryコマンド提示）を踏襲する
- [ ] `rules.squash.enabled = true` の場合にsquashが実行され、`false` の場合は通常コミットにフォールバックする
- [ ] squash成功後、`git log --oneline $(git merge-base origin/main HEAD)..HEAD` で `feat: [{{CYCLE}}] Inception Phase完了` コミットが1件のみ存在することを確認できる

**技術的考慮事項**:
- squash-unit.sh を `--base` 明示指定で呼び出す（スクリプト側の変更は最小限）
- Unit型のsquash統合フローと手順を共通化してDRY原則を維持

### ストーリー 2: squash-unit.sh 事後squash対応 (#228)
**優先順位**: Must-have

As a AI-DLC開発者
I want to 過去のUnit（HEAD以外）に対して事後squashを実行したい
So that squash忘れや後から追加した修正コミットを正しく統合できる

**受け入れ基準**:
- [ ] `squash-unit.sh` に `--retroactive` オプションが追加されている
- [ ] `--retroactive` 指定時は `GIT_SEQUENCE_EDITOR` + `GIT_EDITOR` を使った非対話的rebase方式でsquashが実行される
- [ ] 対象Unitのコミット範囲がコミットメッセージのUnit番号から特定される
- [ ] squash前後で対象Unit以降の `git diff <base>..HEAD` が一致し、後続コミットの内容が保持される
- [ ] `--dry-run` と `--retroactive` の組み合わせが動作し、対象コミット一覧とrebase計画が表示される
- [ ] Linux/macOSの `sed -i` 差異に対応している（`sed -i ''` vs `sed -i`）
- [ ] squash成功時は `squash:success:{新ハッシュ}` が出力される
- [ ] `commit-flow.md` のSquash統合フローに事後squashの手順が追記されている
- [ ] 異常系: Unit番号に該当するコミットが見つからない場合、`squash:error:unit-not-found` を出力して終了する
- [ ] 異常系: 作業ツリーがdirtyな場合、`squash:error:dirty-worktree` を出力して終了する
- [ ] 異常系: rebase衝突が発生した場合、`squash:error:conflict` と `git rebase --abort` によるリカバリ手順を出力する

**技術的考慮事項**:
- 既存の `git reset --soft` 方式はデフォルト動作として維持（後方互換性）
- `--retroactive` は明示指定時のみ有効
- jj環境での事後squash対応はスコープ外（gitのみ）

## Epic 2: レビューフローの品質向上

### ストーリー 3: AIレビュー未完了時のユーザーレビュー移行制限 (#231)
**優先順位**: Must-have

As a AI-DLC開発者
I want to AIレビュー開始後はレビューが完了するまで続行を強制したい
So that AIレビューの品質ゲートが確実に機能する

**受け入れ基準**:
- [ ] `review-flow.md` ステップ5（反復レビュー後）の継続確認で、AIレビュー開始後にユーザーレビューへ遷移する選択肢が存在しない
- [ ] `review-flow.md` ステップ5.5（セルフレビュー反復後）の継続確認で、AIレビュー開始後にユーザーレビューへ遷移する選択肢が存在しない
- [ ] AIレビュー開始後は、指摘0件または指摘対応判断フロー完了まで続行される
- [ ] ステップ4（AIレビュー開始前）の「いいえ」選択肢はそのまま残る（レビュー開始前の判断は許容）
- [ ] ステップ6（外部AIレビュー続行不能時）のフォールバック選択肢はそのまま残る

**技術的考慮事項**:
- 変更対象はreview-flow.mdの2箇所（ステップ5の継続確認、ステップ5.5の継続確認）
- ストーリー7（用語変更）との実装順序: ストーリー7を先に実施し、用語変更後の状態でストーリー3を適用する

### ストーリー 4: セルフレビューでのサブエージェント活用 (#226)
**優先順位**: Must-have

As a AI-DLC開発者
I want to セルフレビューフォールバック時にサブエージェント（コンテキスト分離）でレビューを実行したい
So that 同一コンテキスト内のバイアスが低減し、レビュー精度が向上する

**受け入れ基準**:
- [ ] `review-flow.md` ステップ5.5のセルフレビュー実行手順が、サブエージェント起動方式に変更されている
- [ ] サブエージェントに渡す入力: (1) レビュー対象ファイルパスの一覧、(2) レビュー種別（code/architecture/security/inception）、(3) レビュー観点（対応するSKILL.mdの内容）
- [ ] サブエージェントが統一フォーマット（指摘#N / 重要度 / 内容 / 推奨修正 / 判定）でレビュー結果を返す
- [ ] メインエージェントが結果を反映し、既存の反復レビューフローに接続する
- [ ] サブエージェント起動失敗時は従来のインラインセルフレビューにフォールバックし、履歴に「フォールバック: サブエージェント起動失敗」と記録される
- [ ] 履歴記録に「サブエージェント方式」であることが記載される（`【レビューツール】self-review (sub-agent)`）

**技術的考慮事項**:
- Claude Code環境: `Task` ツール（subagent_type指定）を使用
- その他のAIツール環境: サブプロセスまたはサブセッションで同等のコンテキスト分離を実現する旨を記載

### ストーリー 5: レビューサマリファイル生成（新規）
**優先順位**: Must-have

As a コードレビュアー
I want to AIレビュー結果がファイルとして蓄積されていてほしい
So that PRレビュー時にAIレビューの観点と対応内容を確認できる

**受け入れ基準**:
- [ ] AIレビュー完了時にレビューサマリファイル（`docs/cycles/{{CYCLE}}/construction/units/{NNN}-review-summary.md`）が生成される
- [ ] レビューサマリには以下が含まれる: レビュー種別、使用ツール、指摘内容一覧、各指摘への対応（修正済み/先送り+理由）
- [ ] レビューサマリファイルが存在しない場合（AIレビュー未実施）は生成をスキップする
- [ ] `review-flow.md` のレビュー完了ステップ（ステップ5、ステップ5.5）にサマリファイル生成手順が追加されている
- [ ] Inception Phaseのレビューサマリは `docs/cycles/{{CYCLE}}/inception/{artifact}-review-summary.md` に保存される

**技術的考慮事項**:
- レビューサマリファイルはAIレビュー実施時にリアルタイム生成（後から生成すると精度低下）
