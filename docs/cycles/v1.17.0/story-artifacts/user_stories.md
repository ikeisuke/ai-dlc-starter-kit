# ユーザーストーリー

## Epic 1: コミット/squash管理の強化

### ストーリー 1: Inception Phase完了時のsquash (#234)
**優先順位**: Must-have

As a AI-DLC開発者
I want to Inception Phase完了時に中間コミットを1つのfeatコミットにsquashしたい
So that コミット履歴がクリーンになり、サイクルの変更追跡が容易になる

**受け入れ基準**:
- [ ] `commit-flow.md` にInception Phase完了時のsquash統合フローが追加されている
- [ ] squash設定確認（`rules.squash.enabled`）→ VCS判定 → ユーザー確認 → squash実行のフローが記載されている
- [ ] `inception.md` の完了時手順（ステップ5: Gitコミット）がsquash統合フローを参照している
- [ ] squash実行時の起点コミットはサイクルブランチ分岐点（`git merge-base origin/main HEAD`等）を使用する
- [ ] squash後のコミットメッセージは `feat: [{{CYCLE}}] Inception Phase完了 - {DESCRIPTION}` 形式
- [ ] squash失敗時は既存のエラーリカバリフロー（recoveryコマンド提示）を踏襲する

**技術的考慮事項**:
- squash-unit.sh を `--base` 明示指定で呼び出す（スクリプト側の変更は最小限）
- Unit型のsquash統合フローと手順を共通化してDRY原則を維持

### ストーリー 2: squash-unit.sh 事後squash対応 (#228)
**優先順位**: Must-have

As a AI-DLC開発者
I want to 過去のUnit（HEAD以外）に対して事後squashを実行したい
So that squash忘れや後から追加した修正コミットを正しく統合できる

**受け入れ基準**:
- [ ] `squash-unit.sh` に `--retroactive` オプションが追加されている
- [ ] `--retroactive` 指定時は `GIT_SEQUENCE_EDITOR` + `GIT_EDITOR` を使った非対話的rebase方式でsquashが実行される
- [ ] 対象Unitのコミット範囲がコミットメッセージのUnit番号から特定される
- [ ] 後続コミット（対象Unit以降のコミット）が破損しない
- [ ] `--dry-run` と `--retroactive` の組み合わせが動作する
- [ ] Linux/macOSの `sed -i` 差異に対応している
- [ ] squash成功時は `squash:success:{新ハッシュ}` が出力される
- [ ] `commit-flow.md` のSquash統合フローに事後squashの手順が追記されている

**技術的考慮事項**:
- 既存の `git reset --soft` 方式はデフォルト動作として維持（後方互換性）
- `--retroactive` は明示指定時のみ有効
- jj環境での事後squash対応はスコープ外（gitのみ）

### ストーリー 4: セルフレビューでのサブエージェント活用 (#226)
**優先順位**: Must-have

As a AI-DLC開発者
I want to セルフレビューフォールバック時にサブエージェント（コンテキスト分離）でレビューを実行したい
So that 同一コンテキスト内のバイアスが低減し、レビュー精度が向上する

**受け入れ基準**:
- [ ] `review-flow.md` ステップ5.5のセルフレビュー実行手順が、サブエージェント起動方式に変更されている
- [ ] サブエージェントにはレビュー対象ファイルとレビュー観点（SKILL.mdの内容）のみが渡される
- [ ] サブエージェントがレビュー結果を返し、メインエージェントが結果を反映する
- [ ] サブエージェント起動失敗時は従来のインラインセルフレビューにフォールバックする
- [ ] 履歴記録に「サブエージェント方式」であることが記載される

**技術的考慮事項**:
- Claude Code環境では `Task` ツールを使用
- 他のAIツール環境でも同等の機能が利用可能な方式を記載（汎用的な記述）

### ストーリー 5: PRレビュアビリティ向上（新規）
**優先順位**: Must-have

As a コードレビュアー
I want to PRにAIレビュー結果と対応内容が記載されていてほしい
So that レビュー観点の共有とレビュー負荷の軽減ができる

**受け入れ基準**:
- [ ] AIレビュー完了時にレビューサマリファイル（`docs/cycles/{{CYCLE}}/construction/units/{NNN}-review-summary.md`）が生成される
- [ ] レビューサマリには以下が含まれる: レビュー種別、指摘内容一覧、各指摘への対応（修正済み/先送り+理由）
- [ ] Unit PR本文に要件・受け入れ基準とレビューサマリ内容が記載される
- [ ] サイクルPR本文にIntent概要・受け入れ基準・全体修正概要とレビューサマリファイルへのリンクが記載される
- [ ] レビューサマリファイルが存在しない場合（AIレビュー未実施）はPR本文のサマリセクションが省略される
- [ ] `review-flow.md` のレビュー完了ステップにサマリファイル生成手順が追加されている
- [ ] `construction.md` のUnit PR作成テンプレートが更新されている
- [ ] `operations.md` のサイクルPR作成テンプレートが更新されている

**技術的考慮事項**:
- レビューサマリファイルはAIレビュー実施時にリアルタイム生成（後から生成すると精度低下）
- サイクルPRは情報量が多いためファイルリンク方式を採用
