# 論理設計: レビューサマリファイル生成

## 概要

review-flow.md にレビューサマリファイル生成の共通手順を追加し、ステップ5（AIレビューフロー）とステップ5.5（セルフレビューフロー）から参照する構成を設計する。併せてテンプレートファイルとindex.mdの変更を定義する。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。

## アーキテクチャパターン

既存のreview-flow.mdの構造に準拠し、共通手順セクション参照パターンを適用する。review-flow.md内の既存パターン（「指摘対応判断フロー」がステップ5/5.5の両方から参照されている）と同様に、「レビューサマリ更新手順」を共通セクションとして定義する。

## コンポーネント構成

### 変更対象の構成

```text
prompts/package/
├── prompts/common/
│   └── review-flow.md          ← 変更（共通セクション追加 + ステップ5/5.5から参照追加）
└── templates/
    ├── index.md                ← 変更（エントリ追加）
    └── review_summary_template.md  ← 新規作成
```

### review-flow.md の変更箇所

#### 変更箇所1: 共通セクション追加

**挿入位置**: ステップ7（ユーザーレビューフロー）の直後、「外部入力検証ルール」セクションの直前に新セクションを追加。

**理由**: 指摘対応判断フローと同じレベル（処理フロー内のサブセクション）だが、step 5/5.5の両方から参照されるため、独立したセクションとして配置する方が視認性が高い。

**セクション名**: `## レビューサマリファイル更新手順`

**セクション内容**:
- 目的と前提条件
- ファイルパス決定ルール（フェーズ別マッピング）
- 追記フォーマット
- 処理手順（ファイル存在チェック → 新規作成 or 追記）

#### 変更箇所2: ステップ5（AIレビューフロー）への参照追加

**挿入位置**: ステップ5の項目5（全種別で指摘がゼロになった時点）内の、履歴への記録の直後に1行追加。

**追加内容**: 「レビューサマリファイル更新手順」セクションに従いサマリファイルを更新する旨の参照指示。

**注**: 指摘対応判断フロー経由のサマリ更新は、変更箇所3のステップ5.5側の挿入位置2で説明する通り、共通の指摘対応判断フロー内に1箇所追加する（DRY化）。

#### 変更箇所3: ステップ5.5（セルフレビューフロー）への参照追加

**挿入位置**: ステップ5.5の項目7（全種別で指摘がゼロになった時点）内の、履歴への記録の直後に1行追加。

**追加内容**: ステップ5と同一の参照指示。

**挿入位置2**: 指摘対応判断フロー → 項目7（次のアクション決定）内の「全て先送りの場合: ユーザーレビューフローへ進む」の直前に、サマリ更新参照を追加。この位置は判断フロー完了後であり、結果が確定した後になる。ステップ5/5.5の両方から呼び出される共通の指摘対応判断フロー内に1箇所追加するため、DRY原則にも合致する。

### テンプレートファイル構造

`review_summary_template.md` の構造:

```markdown
# レビューサマリ: [対象名]

## 基本情報

- **サイクル**: {{CYCLE}}
- **フェーズ**: [Construction / Inception]
- **対象**: [Unit名 / ステップ名]

---

## Set [N]: [日時]

- **レビュー種別**: [code, security 等]
- **使用ツール**: [codex / self-review(subagent) / self-review(inline)]
- **反復回数**: [1〜3]
- **結論**: [指摘0件 / 指摘対応判断完了]

### 指摘一覧

| # | 重要度 | 内容 | 対応 |
|---|--------|------|------|
| 1 | 高/中/低 | [指摘内容] | 修正済み / TECHNICAL_BLOCKER（理由: [理由]） / OUT_OF_SCOPE（理由: [理由]） |

（指摘0件の場合: 「指摘なし」と記載）

---
```

**セパレータ**: 各セット間は `---`（水平線）で区切る。

**追加内容**:

## 処理フロー概要

### レビューサマリ更新フロー

**起動条件**:
1. 全種別で指摘0件に到達した時点
2. 指摘対応判断フロー完了後、全て先送りでユーザーレビューへ進む場合

**注**: 条件2は「修正する（RESOLVE）」を選択した指摘がある場合は対象外（反復レビューに戻り、最終的に条件1で記録される）。

**ステップ**:
1. フェーズと呼び出し元ステップからファイルパスを決定
2. ファイルの存在を確認
   - 存在しない場合: テンプレートから新規作成（基本情報セクションを埋める）
   - 存在する場合: ファイル末尾にセパレータ（`---`）を追記
3. 新しいセットのヘッダーを追記（セット番号、日時、レビュー種別、ツール、反復回数、結論）
4. 指摘一覧テーブルを追記
   - 指摘0件の場合: 「指摘なし」
   - 指摘ありの場合: 各指摘の重要度・内容・対応を記録

**関与するコンポーネント**: review-flow.md（共通セクション）、テンプレートファイル
