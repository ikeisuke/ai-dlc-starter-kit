# Intent（開発意図）

## プロジェクト名
AI-DLC Starter Kit v1.0.1 - バグ修正と継続的改善の仕組み構築

## 開発の目的
AI-DLC Starter Kit v1.0.0で発見された2つのバグを修正し、スターターキットの継続的な改善と利用プロジェクトでの運用を効率化するための仕組みを構築する。

### 主な目的
1. **バグ修正**: 実運用で発見された以下の問題を解決
   - 実行記録の日付コマンドが展開されない問題
   - セットアップ時に必要なディレクトリ（inception/）が作成されない問題

2. **スターターキットのバージョンアップ対応**: 利用プロジェクト側での最新化を容易にする
   - 本スターターキット（ai-dlc-starter-kit）がバージョンアップした際に、利用プロジェクト側でキットを最新化する手順の確立
   - バージョンアップ時の認知負荷を減らす仕組みの構築
   - テンプレート/プロンプトの表記揺れ問題の解決

3. **サイクル間のタスク引き継ぎ管理**: サイクルをまたいだタスク管理の明確化
   - 各サイクルのOperations Phase完了時に生成される次サイクルへのタスクリストの管理方法
   - どのサイクルの何を見ればいいかを明確にする仕組み

4. **AI-DLCサイクル全体へのIssue駆動開発の統合**: より効率的な開発フローの確立
   - Inception → Construction → Operations の全フェーズにIssue駆動な仕組みを組み込む方法の検討
   - GitHubやGitLabのIssue管理との連携方法

5. **Operations Phaseでのテストとバグ対応フローの明確化**: テスト実施と問題発見時の対応を標準化
   - テスト実施時の記録フォーマット（型）の提供
   - バグ発見時の対応手順の明確化（どのフェーズに戻るか、どう修正するか）
   - バグ記録と追跡の方法

6. **既存プロンプトの活用徹底**: プロンプト生成時の一貫性確保
   - 独自プロンプトを作らず、既存のフェーズプロンプト（inception.md、construction.md、operations.md）を活用する仕組み
   - プロンプト参照ガイドの作成

## ターゲットユーザー
- AI-DLC Starter Kitを使用する開発者
- AI-DLC手法を実践するチーム
- 本プロジェクトのメンテナー

## ビジネス価値
- **信頼性向上**: バグ修正により、実行記録が正確に残り、セットアップが確実に完了する
- **運用効率化**: スターターキットのバージョンアップが容易になり、利用プロジェクト側の認知負荷が軽減される
- **メンテナンス性向上**: サイクル間のタスク引き継ぎが明確になり、長期的なプロジェクト運営が容易になる
- **開発効率向上**: Issue駆動開発の統合により、タスク管理とAI-DLCサイクルがシームレスに連携する
- **品質保証の標準化**: テストとバグ対応フローが明確になり、問題発見時の対応が迅速になる
- **一貫性の確保**: 既存プロンプトの活用により、フェーズ間の一貫性が保たれ、混乱が減少する

## 成功基準
- バグ1（日付コマンド展開）が修正され、実行記録に正しい日時が記録される
- バグ2（inception/ディレクトリ作成）が修正され、セットアップ時に必要なディレクトリが全て作成される
- スターターキットのバージョンアップ手順が文書化され、利用プロジェクト側での適用が容易になる
- 表記揺れ問題の解決策が確立される
- サイクル間のタスク引き継ぎ管理方法が文書化され、実装される
- Issue駆動開発の統合方法が検討され、設計が完了する（実装は次サイクル以降も可）
- Operations Phaseでのテスト記録フォーマットが提供され、バグ対応フローが文書化される
- 既存プロンプト活用ガイドが作成され、独自プロンプト作成を防ぐ仕組みが確立される

## 期限とマイルストーン
特に明示的な期限は設定しない（AI-DLC手法に従い、短サイクルで完了を目指す）

## 制約事項
- 既存のv1.0.0の構造を大きく変更しない
- 技術スタック: Markdown、Bash、Git（必要に応じてプログラム言語使用可）
- ドキュメント読み込み制限: `docs/cycles/v1.0.1/` 配下のファイルのみを読み込む

## 不明点と質問（Inception Phase中に記録）

### 対話履歴

[Question] 今回のサイクル v1.0.1 で実現したい主な目的や狙いは何ですか？
[Answer] 2つあります。1つはバグ修正、2つ目はバージョンアップの方法の確立です。

[Question] 1つ目のバグ修正について、具体的にどのようなバグがありますか？
[Answer] 実行記録を残す時に日付コマンドが展開されず、コマンド文字列がそのまま残ってしまう。

[Question] 他にもバグはありますか？
[Answer] セットアップ時点で必要なディレクトリ（inception/、construction/など）が作成されていない。

[Question] バージョンアップの方法の確立について、具体的にどのような方法を確立したいですか？
[Answer] アップデートの検知と反映の仕組み、差分更新の仕組み、テンプレートやプロンプトを動的に構成すると表記揺れが発生し差分管理が困難になる問題の解決。

[Question] このプロジェクトは greenfield（新規プロジェクト）と brownfield（既存プロジェクト）のどちらですか？
[Answer] brownfield

[Question] 技術スタックについて、v1.0.1で変更や追加はありますか？
[Answer] 基本は既存のまま（Markdown、Bash、Git）で、必要に応じてプログラム言語を使用可能。

[Question] v1.0.1の成功基準や、完了したと判断できる状態はどのようなものですか？
[Answer] 2つのバグが修正され、バージョンアップ手順が文書化されている。

[Question] バージョンアップの方法について、より詳しく教えてください。主語は何ですか？
[Answer] 本スターターキット（ai-dlc-starter-kit）がバージョンアップした際に、利用プロジェクト側でこのキットを最新にするための手順と、その際の認知負荷を減らす仕組み。

[Question] 他に追加したい目的はありますか？
[Answer] 2つあります。1つは各サイクルのOperations Phase終了後の次サイクルへのタスク引き継ぎ管理。どのサイクルの何を見ればいいかわからなくなる問題の解決。もう1つは、AI-DLCサイクル全体（Inception → Construction → Operations）にIssue駆動な仕組みを組み込む検討。

[Question] 他に追加したい目的はありますか？
[Answer] 2つあります。1つはOperations Phaseでのテストの型（フォーマット）の追加。バグ発見時にどのように対応するのがいいかわからなくなる問題の解決。もう1つは、プロンプト作成時に独自プロンプトを作ってしまわないよう、既存のフェーズプロンプト（inception.md、construction.md、operations.md）を活用する仕組み。
