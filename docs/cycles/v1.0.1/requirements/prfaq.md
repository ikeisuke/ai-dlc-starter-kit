# PRFAQ: AI-DLC Starter Kit v1.0.1

## Press Release（プレスリリース）

**見出し**: AI-DLC Starter Kit v1.0.1 リリース - より確実で持続可能な開発サイクルを実現

**副見出し**: バグ修正と継続的改善の仕組みを強化し、長期的なプロジェクト運営を支援

**発表日**: 2025年12月上旬（予定）

**本文**:

### 背景

AI-DLC Starter Kit v1.0.0のリリース以降、多くの開発者に利用いただき、実運用を通じて貴重なフィードバックを得ることができました。その中で、以下の課題が明らかになりました：

1. **セットアップ時のバグ**: ディレクトリ作成の不備により、各フェーズ実行時にエラーが発生することがありました
2. **バージョンアップの難しさ**: スターターキット本体が更新された際に、利用プロジェクト側でどのように最新化すればよいかが不明確でした
3. **サイクル管理の課題**: 複数のサイクルを経た後、どのサイクルのタスクを見ればよいかわからなくなる問題がありました
4. **Issue管理との乖離**: 多くのプロジェクトで使用されているIssue駆動開発との連携が不足していました

### プロダクト

AI-DLC Starter Kit v1.0.1は、これらの課題に対処し、より確実で持続可能な開発サイクルを実現します：

**1. セットアップの確実性向上**
- セットアップ時に必要なすべてのディレクトリ（inception/を含む）が確実に作成されます
- 履歴記録の日付展開が正しく動作することを保証します

**2. スターターキットのバージョンアップ対応**
- 利用プロジェクト側で、スターターキット本体の新バージョンを簡単に検知できます
- バージョンアップ手順が明確に文書化され、認知負荷を最小限に抑えて最新化できます
- テンプレートやプロンプトの表記揺れ問題が解決され、差分が明確になります

**3. サイクル管理基盤の構築**
- 各サイクル完了時のタスクが明確に記録され、次サイクルへスムーズに引き継がれます
- 複数サイクルのタスクを横断的に確認できる仕組みにより、「どこを見ればいいか」が一目瞭然です

**4. Issue駆動開発との統合設計**
- GitHubやGitLabのIssue管理とAI-DLCサイクルがシームレスに連携する設計が完成します
- Epic → Unit → 実装 → クローズのフローが明確になり、タスク管理が一元化されます

v1.0.1は、AI-DLC Starter Kitの継続的改善の第一歩です。今後のサイクルでは、以下を予定しています：

## FAQ（よくある質問）

### Q1: v1.0.0からv1.0.1への主な変更点は何ですか？
A: 主な変更点は4つあります：
1. セットアップバグの修正（inception/ディレクトリ作成、日付展開）
2. スターターキットのバージョンアップ対応（検知、手順書、表記揺れ対策）
3. サイクル間のタスク引き継ぎ管理の仕組み
4. Issue駆動開発統合の設計（実装は次サイクル以降）

詳細は `docs/cycles/v1.0.1/requirements/intent.md` を参照してください。

### Q2: v1.0.0からv1.0.1へのバージョンアップは必須ですか？
A: 必須ではありませんが、以下の理由から推奨します：
- セットアップバグが修正されるため、新しいサイクルを開始する際のエラーが減少します
- バージョンアップ手順が文書化されるため、今後の更新が容易になります
- サイクル管理が改善され、長期的なプロジェクト運営が楽になります

既に v1.0.0 で開発を進めている場合は、次のサイクル開始時に v1.0.1 を適用することをお勧めします。

### Q3: 既存プロジェクト（v1.0.0を使用中）への影響はありますか？
A: v1.0.1 は既存のプロジェクト構造を大きく変更しません。主な影響は以下の通りです：

**プロンプトファイルの更新**:
- `docs/aidlc/prompts/inception.md`
- `docs/aidlc/prompts/construction.md`
- `docs/aidlc/prompts/operations.md`
- `prompts/setup-prompt.md`

**新規ファイルの追加**:
- `docs/aidlc/upgrade-guide.md`（バージョンアップ手順書）
- サイクル管理スクリプト（該当する場合）

既存のサイクル成果物（`docs/cycles/{CYCLE}/`）には影響しません。バージョンアップ手順書に従って、プロンプトファイルを更新するだけで適用できます。

### Q4: Issue駆動開発の統合は v1.0.1 で実装されますか？
A: v1.0.1 では設計のみを行い、実装は次サイクル以降となります。

v1.0.1 で完成する成果物：
- Issue駆動開発統合の設計文書
- Issueテンプレート（Epic用、Unit用、バグ用）
- Issue → Unit → 実装 → クローズのフロー定義
- 実装の優先順位と範囲の決定

これにより、次サイクルで確実に実装を進めることができます。設計段階で利用者の意見を取り入れることで、より実用的な統合を目指します。

### Q5: サイクル管理の仕組みはどのように使いますか？
A: サイクル管理の仕組みは、以下の2つの機能を提供します：

**1. タスクリストの記録**:
各サイクルのOperations Phase完了時に、`docs/cycles/{CYCLE}/operations/next_cycle_tasks.md` にタスクを記録します。これにより、次サイクルで何を実施すべきかが明確になります。

**2. サイクル横断タスクの集約**:
複数サイクルのタスクを集約して表示するスクリプトまたはサマリーファイル（`docs/cycles/task_summary.md`）を提供します。これにより、「どのサイクルの何を見ればいいか」が一目でわかります。

使用例：
```bash
# タスクサマリーを表示
bash scripts/show_tasks.sh

# または、サマリーファイルを確認
cat docs/cycles/task_summary.md
```

### Q6: 表記揺れ問題とは何ですか？どのように解決されますか？
A: 表記揺れ問題とは、テンプレートやプロンプトを動的に生成する際に、同じ内容でも微妙な表現の違いが発生し、バージョン間の差分として認識されてしまう問題です。

v1.0.1 では、以下のアプローチを検討し、最適な解決策を選定します：
1. **静的テンプレート化**: 動的生成をやめ、静的なテンプレートを配布
2. **ハッシュ値管理**: ファイルのハッシュ値を記録し、内容の同一性を判定
3. **差分ツールのカスタマイズ**: 空白・改行を無視した差分表示

選定した解決策を実装し、バージョンアップ時の差分が明確になります。

### Q7: 次のサイクル（v1.0.2以降）では何が予定されていますか？
A: v1.0.1の成果を基に、以下を予定しています：

**短期（v1.0.2）**:
- Issue駆動開発統合の実装
- サイクル管理の自動化強化

**中期（v1.0.3以降）**:
- CI/CD連携の深化
- マルチプラットフォーム対応の拡充
- コミュニティテンプレートの提供

**長期**:
- AIエージェントとの連携強化
- リアルタイムコラボレーション機能
- メトリクス分析とレコメンデーション

具体的な計画は、v1.0.1のOperations Phase完了時に策定します。
