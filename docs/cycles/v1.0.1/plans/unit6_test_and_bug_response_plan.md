# Unit 6: テストとバグ対応基盤 - 実行計画

## 概要

Operations Phaseでのテスト実施時に使用する記録フォーマットを提供し、バグ発見時の対応フローを明確化する。

---

## 成果物一覧

| 成果物 | パス | 説明 |
|--------|------|------|
| テスト記録テンプレート | `docs/aidlc/templates/test_record_template.md` | テスト項目、結果、バグ記録を統一形式で記録 |
| バグ対応フロー文書 | `docs/aidlc/bug-response-flow.md` | バグ種類別の対応フローと判断基準 |
| Operations Phase更新 | `docs/aidlc/prompts/operations.md` | テスト記録指示の追加 |
| 設計ドキュメント | `docs/cycles/v1.0.1/design-artifacts/` | ドメインモデル、論理設計 |
| 実装記録 | `docs/cycles/v1.0.1/construction/units/` | 実装記録 |

---

## 実行フロー

### Phase 1: 設計【対話形式、コードは書かない】

#### ステップ1: ドメインモデル設計
- テスト記録とバグ対応の概念モデルを定義
- エンティティ: テスト項目、テスト結果、バグレポート
- 成果物: `unit6_domain_model.md`

#### ステップ2: 論理設計
- テンプレート構成とフロー構成を定義
- バグ種類別の対応フロー設計
- 成果物: `unit6_logical_design.md`

#### ステップ3: 設計レビュー
- ユーザーに設計内容を提示し承認を得る

---

### Phase 2: 実装【設計を参照してドキュメント生成】

#### ステップ4: テスト記録テンプレート作成
- `docs/aidlc/templates/test_record_template.md` を作成
- チェックリスト形式（`- [ ]`）で進捗を可視化
- テスト項目、結果、バグ記録欄を含む
- 再現手順、期待結果、実際の結果を記録

#### ステップ5: バグ対応フロー文書作成
- `docs/aidlc/bug-response-flow.md` を作成
- バグ種類別の対応フロー（設計バグ、実装バグ、環境バグ）
- どのフェーズに戻るかの判断基準
- バグ修正の記録方法（history.mdへの追記）
- 再テストの手順

#### ステップ6: Operations Phaseプロンプト更新
- テスト記録テンプレートの使用指示を追加
- バグ対応フローへの参照を追加

#### ステップ7: 整合性確認とレビュー
- 各フェーズプロンプトのバックトラックセクションとの整合性確認
  - Construction Phase: 「Operations Phaseからバグ修正で戻ってきた場合」
  - Operations Phase: 「Constructionに戻る必要がある場合」
- 実装記録作成
- ユーザーレビュー
