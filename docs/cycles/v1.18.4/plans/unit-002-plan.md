# Unit 002 計画: エラー時バックログ記録支援

## 概要
Construction Phase中のビルド/テストエラー発生時に、ユーザーに確認の上でバックログ登録を提案するフローを追加する。

## 変更対象ファイル
- `prompts/package/prompts/construction.md` - 「#### ステップ6: 統合とレビュー」の「2. テスト実行」直後に挿入（既存項目3〜6は繰り下げ）

## 実装計画

### 現状の問題
ステップ6「統合とレビュー」でビルド/テスト実行後にエラーが発生した場合、エラー修正は行うが、再発防止や改善のためのバックログ記録フローがない。

### 修正方針
ステップ6の「2. テスト実行」の後に、ビルド/テストエラー検出時のバックログ登録提案フローを追加する。

### フロー設計

1. ビルドまたはテストでエラーが発生した場合
2. **ユーザーに「バックログに記録しますか？」と確認**（エラー発生時はcommon/rules.mdのフォールバック条件 `reason_code=error` に該当するため、`automation_mode` に関わらず常にユーザー確認を行う）
3. ユーザーが「はい」の場合（バックログ登録を実行）:
   - バックログモード取得: `docs/aidlc/bin/check-backlog-mode.sh`
   - **mode=issue / mode=git**:
     - `issue`: GitHub Issue作成（`bugfix-` prefix、`gh:available` 確認要）。失敗時は `git` にフォールバック
     - `git`: `docs/cycles/backlog/bugfix-{slug}.md` にファイル作成
   - **mode=issue-only / mode=git-only**（排他モード）:
     - `issue-only`: GitHub Issue作成のみ。失敗時は警告表示、手動対応依頼
     - `git-only`: ファイル作成のみ
4. ユーザーが「いいえ」の場合: スキップ
5. 選択結果（登録/スキップ）を履歴に記録
6. エラー修正を実施してビルド/テストを再実行

### review-flow.md 5a準拠の最小要件
- **mode解決手順**: `check-backlog-mode.sh` で取得。失敗時は `git` として扱う
- **gh利用可否判定**: Issue作成前に `gh:available` を確認（ステップ2.5の結果を参照）
- **slug生成**: エラー内容から短い識別子を生成（英数字・ハイフン、`^[a-z0-9][a-z0-9-]{0,63}$`）
- **重複回避**: 同名ファイル/Issue存在時はサフィックス付与（`-2`, `-3`...）
- **失敗時の挙動**: Issue作成失敗時のフォールバック（mode=issueのみ）、git方式失敗時は警告表示
- **テンプレート**: `docs/aidlc/templates/backlog_item_template.md` に準拠
- **安全規則**: heredoc終端トークン拒否、slug制約、引数の二重引用符

### 履歴記録フォーマット
```text
【エラー時バックログ登録】{登録 / スキップ}
【エラー種別】{ビルドエラー / テストエラー}
【バックログモード】{mode}
【登録先】{Issue番号 / ファイルパス / なし}
```
