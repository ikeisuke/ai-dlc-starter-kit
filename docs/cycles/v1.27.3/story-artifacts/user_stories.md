## Epic 1: エクスプレスモード再設計

### ストーリー 1: エクスプレスモードのフロー制御純化
**優先順位**: Must-have

As a AI-DLC利用者
I want to depth_level に関係なくエクスプレスモード（フェーズ連続実行）を有効化したい
So that フェーズ間のコンテキストリセットを省略して効率的に開発できる

**受け入れ基準**:
- [ ] `depth_level=standard` の状態で `start express` を実行すると、depth_level を minimal に変更せずにエクスプレスモードが起動する
- [ ] `depth_level=minimal` での `start express` は従来通り動作する（後方互換）
- [ ] エクスプレスモード起動時に「フェーズ連続実行モードを適用します」と表示される
- [ ] エクスプレスモード適用時、Inception 完了後にコンテキストリセットをスキップし Construction Phase に自動遷移する
- [ ] エクスプレスモードが無効の場合、Inception 完了時にコンテキストリセットが従来通り提示される

**技術的考慮事項**:
- `rules.md` のエクスプレスモード仕様セクションの前提条件を `depth_level=minimal` から削除
- `inception.md` ステップ4b のスキップ条件を変更
- `start express` コマンドの意味を「minimal オーバーライド」から「フェーズ連続実行の有効化」に変更

### ストーリー 2: Unit 複雑度に基づくエクスプレスモード適格性判定
**優先順位**: Must-have

As a AI-DLC利用者
I want to Unit の複雑度に基づいてエクスプレスモードの適用可否が自動判定されたい
So that 複雑なタスクでは通常フロー、シンプルなタスクでは連続実行と適切に使い分けられる

**受け入れ基準**:
- [ ] Unit 定義完了後に、各 Unit の複雑度が以下の4項目で評価される
- [ ] 全 Unit が `eligible` と判定された場合のみエクスプレスモードが適用される
- [ ] 1つでも `ineligible` な Unit がある場合、「エクスプレスモード適用不可: [Unit名] が複雑度条件を満たしません（理由: [項目名]）」と表示され通常フローに切り替わる
- [ ] Unit 数 2-3 でも全 Unit が `eligible` ならエクスプレスモードが適用される
- [ ] Unit 数のハードコーディングされた上限がない
- [ ] Unit 数が0の場合はエクスプレスモード対象外（従来通り）
- [ ] 判定結果と理由が Unit 定義ファイルの実装状態セクションに記録される

**複雑度判定ルール**:

| 評価項目 | eligible 条件 | ineligible 条件 |
|----------|--------------|----------------|
| 受け入れ基準の明確さ | 全基準が具体的で検証可能。曖昧な表現（「適切に」「十分に」等）がない | 曖昧な基準が1つ以上ある、または基準が未定義 |
| 依存関係の複雑さ | Unit 間依存が線形（A→B→C）。循環依存・多段分岐依存なし | 循環依存、3つ以上の Unit からの同時依存、または外部システムとの双方向依存あり |
| 技術的リスク | 使用技術が既知。プロジェクト内に類似実装あり | 未使用の技術・フレームワーク導入、外部API連携の新規追加、またはアーキテクチャ変更を伴う |
| 変更影響範囲 | 変更対象ファイルが特定可能で限定的 | 影響範囲が不明確、または横断的変更（共通基盤の改修等）を伴う |

**技術的考慮事項**:
- 判定ロジックはプロンプト内でAIが実行（シェルスクリプトではない）
- 判定はUnit定義完了後のステップ4bで実行

### ストーリー 3: depth_level に応じた成果物品質の維持
**優先順位**: Must-have

As a AI-DLC利用者
I want to エクスプレスモード適用時でも depth_level に応じた成果物が生成されること
So that 連続実行の利便性と成果物の質を両立できる

**受け入れ基準**:
- [ ] `depth_level=standard` でエクスプレスモード適用時、standard 相当の成果物（Intent、ユーザーストーリー、Unit定義、ドメインモデル、論理設計）が生成される
- [ ] `depth_level=minimal` でエクスプレスモード適用時、minimal 相当の成果物が生成される（従来通り）
- [ ] `depth_level=comprehensive` でエクスプレスモード適用時、comprehensive 相当の成果物が生成される
- [ ] エクスプレスモードが depth_level の成果物要件を変更しない（エクスプレス = フロー制御のみ）

**技術的考慮事項**:
- エクスプレスモード = フロー制御（連続実行）、depth_level = 成果物詳細度、automation_mode = 承認制御。3つの軸が独立して動作する設計

---

## 横断的な非機能要件: 後方互換性

以下の回帰確認を全ストーリー完了後に検証する:

- [ ] `automation_mode=manual` でエクスプレスモードが有効でも、フェーズ内の承認ポイントはユーザー確認が必要
- [ ] `automation_mode=semi_auto` でエクスプレスモードが有効の場合、フェーズ内の承認は自動通過しフェーズ間も自動遷移する
- [ ] `depth_level=minimal` かつ Unit 数1の場合（従来のエクスプレスモード条件）、従来と同じ動作（成果物簡略化 + 連続実行）になる

### ストーリー 5: post-merge-cleanup.sh の警告時 exit 2 追加
**優先順位**: Should-have

As a AI-DLCスクリプト利用者
I want to post-merge-cleanup.sh が警告状態のときに exit 2 を返すこと
So that 呼び出し元が正常完了と警告ありを区別できる

**受け入れ基準**:
- [ ] クリーンアップが完全に成功した場合は `exit 0` で終了する
- [ ] 以下の非致命的な問題が発生した場合は `exit 2` で終了する:
  - 一時ファイル・ディレクトリの削除失敗
  - リモートブランチの削除失敗（ローカル削除は成功）
  - 古い worktree エントリのクリーンアップ失敗
- [ ] 以下の致命的エラーの場合は `exit 1` で終了する（現状維持）:
  - Git リポジトリ外での実行
  - メインブランチの操作失敗（pull、checkout等）
  - ローカルブランチの削除失敗
- [ ] `OVERALL="warning"` 状態が正しく `exit 2` にマッピングされる

**技術的考慮事項**:
- `migrate-config.sh` のゴールドスタンダード実装（`_has_warnings` フラグ）を参考にする
- 最終 exit 前に `OVERALL` 変数を判定する分岐を追加
