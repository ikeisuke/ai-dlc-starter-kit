# 論理設計: フェーズ内操作順序の明示化

## 概要

commit-flow.mdに追加する操作順序ルールセクションの構成とフォーマットを定義する。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。

## アーキテクチャパターン

ルールベース方式: 順序制約をテーブル形式で列挙し、違反時の自己修正フローを統一的に記述する。commit-flow.mdの既存構造（「コミットポリシー」→各サブセクション）に従い、新規サブセクションとして追加する。

## コンポーネント構成

### セクション構成

```text
コミットポリシー（既存）
├── コミットタイミング（既存）
├── コミットメッセージフォーマット一覧（既存）
├── Co-Authored-By 設定（既存）
├── プレースホルダ定義（既存）
└── 操作順序ルール（新規追加）★
    ├── 順序制約テーブル
    └── 順序違反検知時の自己修正フロー
```

### コンポーネント詳細

#### 順序制約テーブル

- **責務**: commit-flow.mdスコープ内の操作順序制約を一覧化
- **依存**: なし（自己完結）
- **フォーマット**: Markdownテーブル（先行操作、後続操作、前提条件、違反時アクション）

#### 順序違反検知時の自己修正フロー

- **責務**: AIが順序違反を検知した際の対応手順を定義
- **依存**: 順序制約テーブル
- **フォーマット**: 番号付きステップリスト

## インターフェース設計

### ドメインモデルとの対応

本Unitはプロンプトドキュメント（Markdown）への追記であり、プログラム的なAPIは存在しない。ドメインモデルで定義した概念は、以下の通りMarkdownの構造要素にマッピングされる:

| ドメインモデル概念 | Markdown実装 |
|------------------|-------------|
| OperationOrderRule | 順序制約テーブルの1行 |
| OperationOrderRuleSet | 順序制約テーブル全体 |
| OrderViolationDetector.detect() | 自己修正フローのステップ1-2（AIによる判定） |
| PredecessorResolver.resolve() | 自己修正フローのステップ3（先行操作の特定） |

## データモデル概要

### 順序制約テーブルのスキーマ

| カラム | 説明 |
|--------|------|
| 先行操作 | 先に完了すべき操作 |
| 後続操作 | 先行操作完了後に実行すべき操作 |
| 前提条件 | ルール適用条件（常時 / 設定依存） |
| 違反時アクション | 停止→先行操作を実行 |

### 順序制約の具体ルール

| # | 先行操作 | 後続操作 | 前提条件 | 違反時アクション |
|---|---------|---------|---------|---------------|
| 1 | コミット完了 | PR作成 | 常時 | 停止→未コミット変更をコミット |
| 2 | スカッシュ完了 | Unit完了コミット | squash.enabled=true | 停止→スカッシュフローを先に実行 |
| 3 | AIレビュー完了 | ユーザーレビュー依頼 | review mode != disabled | 停止→AIレビューフローを先に実行 |

### スコープ外の順序制約（参照リンク）

- 全Unit完了 → Operations Phase移行: `construction.md` の完了基準セクションを参照
- 設計承認 → 実装開始: `construction.md` のPhase 1/Phase 2分離を参照

## 処理フロー概要

### 順序違反検知→自己修正フロー

**ステップ**:

1. AIが操作実行前に順序制約テーブルを参照
2. 先行操作が未完了の場合、違反を検知
3. 実行すべき先行操作を特定（ドメインサービスの責務）
4. 先行操作を実行（アプリケーションフローの責務）
5. 先行操作完了後、元の後続操作を再開
