# Unit 002 計画: Self-Healingテストループ

## 概要

Construction Phase Step 6（統合とレビュー）にビルド/テスト失敗時の自動修正ループ（最大3回）を導入する。現行の手動ループ（「エラー修正を実施し、ビルド/テストを再実行」）を構造化し、attempt番号・失敗要因の出力仕様を定義する。非回復系エラー（認証失効、ツール制約等）は即時フォールバックとしリトライ対象外とする。

## 変更対象ファイル

- `prompts/package/prompts/construction.md` — Step 6 セクション（L539-593付近）

## 実装計画

### 1. Step 6 の現行フロー分析

現行:
1. ビルド実行
2. テスト実行
3. エラー時バックログ登録提案 → エラー修正 → 再実行（暗黙のループ、回数制限なし）
4. AIレビュー
5-7. 承認・記録

### 2. Self-Healingループの責務分離設計

Step 6 の項目 1-3 を以下の責務分離に基づき再構成:

**オーケストレータ（試行管理）**:
- attempt回数の管理（最大3回）
- エラー分類の呼び出し
- attempt結果の出力
- 収束判定（成功/最大回数到達/非回復系エラー検出）

**エラー分類器（ErrorClassifier）**:
- エラーを3カテゴリ（`recoverable` / `non_recoverable` / `transient`）に分類
- 分類結果をオーケストレータに返却

**実行・修正（Executor/Fixer）**:
- ビルド/テストの実行
- エラー分析と修正の実施

**フォールバックハンドラ**:
- 非回復系エラー検出時のユーザー判断フロー
- 3回失敗時のユーザー判断フロー（バックログ登録提案を含む）

**再構成後のフロー**:

1. ビルド実行
2. テスト実行
3. **エラー発生時 → エラー分類器で判定**
   - 非回復系 → 即時フォールバックハンドラへ
   - 回復可能 → Self-Healingループ開始（オーケストレータ管理）
4. attempt 1〜3: AI自動修正 → 再ビルド/テスト → エラー分類 → 判定
5. 3回失敗時: フォールバックハンドラ（ユーザー判断フロー）
6. AIレビュー（既存の項目4以降）

### 3. 非回復系エラーの分類と判定基準

エラーを以下の3カテゴリに分類し、カテゴリに応じた対応を行う:

**即時中断（ユーザー介入必須）**:
| エラー種別 | 判定基準 | 対応 |
|-----------|---------|------|
| 認証失効・トークン期限切れ | 認証関連エラーメッセージ（401, 403, auth, token expired等） | 即時フォールバック |
| ツール制約（リソース不足） | disk full, out of memory, ENOMEM, ENOSPC等 | 即時フォールバック |
| 環境設定エラー（依存未解決） | command not found, module not found, パッケージ未インストール等 | 即時フォールバック |

**一時的障害（transient — 待機後再試行の可能性あり）**:
| エラー種別 | 判定基準 | 対応 |
|-----------|---------|------|
| ネットワーク接続不可 | connection refused, timeout, DNS resolution failed等 | 1回のみ再試行（attempt回数を消費する）。再試行後も失敗ならフォールバック |

**回復可能（recoverable — Self-Healingループ対象）**:
| エラー種別 | 判定基準 | 対応 |
|-----------|---------|------|
| コード・テストのロジックエラー | 上記のいずれにも該当しないエラー | Self-Healingループでattempt回数を消費しながら自動修正 |

**判定の優先順位**: 即時中断（non_recoverable） > 一時的障害（transient） > 回復可能（recoverable、デフォルト）

**attemptカウントルール**: transient・recoverableの両カテゴリとも、再試行時にattemptカウントを1消費する。最大3回のattempt上限はカテゴリ横断で共有する（例: transientで1回消費 → recoverableで残り2回）。

### 4. attempt出力仕様

各attemptで以下のフィールドを出力（全フィールド必須）:

| フィールド | 型 | 説明 |
|-----------|---|------|
| attempt | `{N}/3` | 現在の試行番号/最大試行数 |
| error_type | `ビルドエラー` / `テストエラー` | エラーの種別 |
| error_category | `recoverable` / `non_recoverable` / `transient` | エラー分類結果 |
| cause | 文字列 | エラーの要約 |
| fix_summary | 文字列 | 実施した修正の要約 |

```text
【Self-Healing】attempt {N}/3
【エラー種別】{ビルドエラー / テストエラー}
【エラー分類】{recoverable / non_recoverable / transient}
【失敗要因】{エラーの要約}
【修正内容】{実施した修正の要約}
```

### 5. 3回失敗時のユーザー判断フロー

```text
【Self-Healing失敗】3回の自動修正で解決できませんでした。
【エラー種別】{ビルドエラー / テストエラー}
【最終エラー】{エラーの要約}
【試行履歴】
  attempt 1: {失敗要因の要約}
  attempt 2: {失敗要因の要約}
  attempt 3: {失敗要因の要約}

どのように対応しますか？
1. 手動で修正を継続する
2. バックログに記録してスキップする
3. 処理を中断する
```

### 6. 非回復系エラー検出時のユーザー判断フロー

```text
【非回復系エラー検出】自動修正の対象外です。
【エラー種別】{ビルドエラー / テストエラー}
【エラー分類】{non_recoverable / transient}
【エラー内容】{エラーの要約}
【判定理由】{該当した判定基準}

どのように対応しますか？
1. 手動で修正を継続する
2. バックログに記録してスキップする
3. 処理を中断する
```
