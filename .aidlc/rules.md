# 追加ルール

このファイルは、AI-DLC Starter Kit プロジェクト固有の追加ルールや制約を記載します。

**重要**: セットアップ完了後、このファイルをプロジェクトに合わせてカスタマイズしてください。

---

## メタ開発の意識【最重要】

このプロジェクトは「AI-DLCスターターキットを使って、AI-DLCスターターキット自体を開発している」というメタ構造を持つ。

**この意識が薄れると起きる問題**:
- スターターキットの改善なのか、利用プロジェクトの実装なのか混同する
- テンプレートの変更が「次回セットアップ時に反映」されることを忘れる
- 自分自身のプロンプトを編集していることの影響範囲を見誤る

**常に確認すべきこと**:
- 今触っているファイルは「ツール側（`skills/aidlc/`）」か「成果物側（`.aidlc/cycles/`）」か
- スキル内リソースの修正は `skills/aidlc/` 配下を編集すること（メタ開発時の編集対象パス）
- 変更が即時反映されるか、次回セットアップ時に反映されるか

**2つの参照方式の区別**:
- **メタ開発時（ファイル編集）**: `skills/aidlc/**`（プロジェクトルート相対パス、例外META-001として許可）
- **スキル実行時（AI-DLCフロー内）**: スキルベース相対パス（`templates/`, `steps/`, `scripts/` 等）
- 通常ルールやUnit記述ではスキルベース相対参照を使用すること

**プラグイン前提の構成原則**:
- v2.0.5以降、スターターキットは `skills/aidlc/` プラグインとして構成される
- プロジェクトルート相対パスでのスキル内リソース参照（`skills/aidlc/templates/xxx.md` 等）は禁止（メタ開発時の編集を除く）
- 外部プロジェクトには `skills/` ディレクトリが存在しない場合があるため、スキル実行時は常にスキルベース相対パスを使用する

### スキル間依存ルール

- **禁止**: 他スキルの内部実装（`scripts/`, `steps/`, `templates/` 等の内部ファイルパス）への依存
- **許可**: スキルの呼び出し名（`/aidlc`, `/reviewing-construction-code` 等）とSKILL.mdで定義された入出力引数への依存
- 内部ファイルパス参照はAPIと見なさない。スキル更新時の波及範囲を限定するため、公開された呼び出しインターフェイスのみに依存すること

### ファイル参照境界ルール【重要】

AIエージェントが参照・作成してよいパスをホワイトリスト形式で定義する。**許可リスト・例外リストに該当しないパスは全て禁止**（デフォルト禁止）。

#### 許可パス一覧

| カテゴリ | パスパターン | アクセス方法 | 許可操作 | 説明 |
|---------|-------------|-------------|---------|------|
| AIDLC設定 | `.aidlc/config.toml`, `.aidlc/config.local.toml` | 直接参照 | read | プロジェクト設定（編集はsetupスキル経由） |
| サイクルデータ | `.aidlc/cycles/{{CYCLE}}/**` | 直接参照 | read, write, create | 現サイクルの成果物・履歴・設計ドキュメント |
| AIDLC追加ルール | `.aidlc/rules.md` | 直接参照 | read, write | プロジェクト固有ルール |
| スキル内リソース | `templates/**`, `scripts/**`, `steps/**`, `guides/**`, `config/**`, `references/**` | スキルベースディレクトリ相対 | read, execute | SKILL.mdの「パス解決」ルールに従う |
| エージェント設定 | `CLAUDE.md`, `AGENTS.md` | 直接参照 | read | エージェント設定ファイル |
| 開発者ドキュメント | `docs/development/**` | 直接参照 | read | メタ開発向け開発者ドキュメント |

**注意**:
- スキル内リソースは必ずスキルベースディレクトリからの相対パスで参照すること（SKILL.mdの「パス解決」ルール参照）。`skills/aidlc/templates/xxx.md` のようなプロジェクトルート相対パスでの参照は禁止
- **デプロイ検証の例外**: `ls skills/aidlc/SKILL.md` のようなスキルデプロイ状況の存在確認は、コンテンツ参照ではなく運用上の確認のため許可

#### デフォルト禁止（代表的な違反例）

| 違反パターン | 禁止理由 | 正しい参照方法 |
|-------------|---------|--------------|
| `skills/aidlc/templates/xxx.md` | プロジェクトルート相対パスでのスキル内リソース参照。メタ開発外プロジェクトには `skills/` が存在しない | `templates/xxx.md`（スキルベースディレクトリ相対） |
| `skills/aidlc/scripts/xxx.sh` | 同上 | `scripts/xxx.sh`（スキルベースディレクトリ相対） |
| `skills/aidlc/steps/xxx.md` | 同上 | `steps/xxx.md`（スキルベースディレクトリ相対） |
| `prompts/xxx` | 廃止済みパス。v2.0.5以降はスキルプラグイン構成に移行済み | スキルベース相対パスで参照（メタ開発時の編集は `skills/aidlc/**`、例外META-001参照） |
| 任意のプロジェクトファイルの作成 | AIDLC管理外のファイルをAIエージェントが勝手に作成すべきでない | 本ルールの対象外（ユーザーの明示的指示による作成はAIDLCの参照境界ルールではなくユーザー承認ポリシーで管理） |
| `~/.claude/plugins/marketplaces/**` | プラグインキャッシュの直接編集。キャッシュはスキル実行時に読み取り専用で使用されるものであり、編集してもgit管理されない。メタ開発時の編集対象はプロジェクト内の `skills/aidlc/**`（META-001） | `skills/aidlc/**`（プロジェクトルート相対パス、META-001） |

#### 例外リスト（メタ開発固有）

以下はデフォルト禁止の対象だが、特定条件下で許可される:

| ID | 対象パス | 許可操作 | 理由 | 適用範囲 | 注意事項 |
|----|---------|---------|------|---------|---------|
| META-001 | `skills/aidlc/**` | read, write, create | メタ開発時のスキル内リソース編集 | メタ開発リポジトリのみ | スキル実行時はスキルベースディレクトリ相対パスで参照すること |
| META-002 | （廃止） | - | `docs/aidlc/` はv2.0.5で廃止済み。IDは安定識別子として欠番維持 | - | - |
| META-003 | `bin/**` | execute | プロジェクト固有スクリプト | カスタムワークフロー実行時 | 本ファイルの「カスタムワークフロー」セクションで明示指定されたコマンドのみ |

## カスタムワークフロー

### バージョンファイル更新【重要】

Operations Phase ステップ7（リリース準備）の 7.1（バージョン確認）完了後、7.2（CHANGELOG更新）の前にリポジトリルートで以下を実行すること：

```bash
bin/update-version.sh --version {{CYCLE}}
```

dry-runで事前確認する場合:

```bash
bin/update-version.sh --version {{CYCLE}} --dry-run
```

**理由**: AI-DLCスターターキット自体のリリース時にバージョン番号を更新するため。

### パーミッション管理【重要】

Operations Phase ステップ7（リリース準備）の 7.7（Gitコミット）完了後に以下の2ステップを順に実行すること：

**ステップ1: セッション履歴分析**

```
/tools:suggest-permissions
```

- **新規ルール提案あり**: 提案内容を確認し、適切なものを設定に反映する。反映した場合は次のカスタムワークフロー前に追加コミットする
- **新規ルール提案なし**: 「パーミッション提案: 変更なし」と表示して次へ進む
- **コマンド実行失敗**: 以下を表示し、ステップ2へ進む

  ```text
  【警告】suggest-permissionsの実行に失敗しました。
  手動でパーミッション設定を確認してください。
  ```

**ステップ2: 既存設定の監査**

```
/tools:suggest-permissions --review all
```

- **HIGH/CRITICAL指摘あり**: 指摘内容を確認し、対応を実施する（ask/denyガードの追加等）。変更した場合は次のカスタムワークフロー前に追加コミットする
- **指摘なし（またはLOW/MEDIUM のみ）**: 「パーミッション監査: 合格」と表示して次へ進む
- **コマンド実行失敗**: 以下を表示し、次のステップへ進む

  ```text
  【警告】suggest-permissions --review allの実行に失敗しました。
  手動で既存のパーミッション設定に危険な構成がないか確認してください。
  ```

**理由**: リリース前にパーミッション設定のセッション分析と監査を行い、新たな自動承認パターンの検討と危険な設定の検出をリリースごとに実施するため。

### Bash Substitution Check実行【重要】

Operations Phase ステップ7（リリース準備）の 7.5（Markdownlint実行）完了後、7.6（progress.md更新）の前に以下を実行すること：

```bash
bin/check-bash-substitution.sh
```

- **違反なし（終了コード0）**: 次のステップへ進む
- **違反あり（終了コード1）**: 違反箇所を修正してから次のステップへ進む。コマンド置換の代わりにリテラル文字列やプレースホルダーを使用すること
- **スクリプトエラー（終了コード2）**: スクリプトの状態を確認し、問題を解決してから再実行

**理由**: Bashコードブロック内のコマンド置換（`$()` やバッククォート）はAIエージェントが意図せず展開する可能性があるため、リリース前に検出・修正する。このチェックはGitHub Actions CIでも実行される。

## コーディング規約

### コマンド置換（`$()`）使用禁止【重要】

すべてのBashコマンド実行において、`$()` およびバッククォート（`` ` ``）によるコマンド置換を使用しない。

**理由**:
- プロンプトファイル内: AIエージェントがプロンプトを読み込む際に意図せず展開・実行される可能性がある
- gitコマンド等の実行時: ツール出力に `$()` が表示され、ユーザーの混乱を招く

**対応方法**:
- 動的な値はAIエージェントがコンテキスト変数から直接置換する（例: `{{CYCLE}}`, `{{project.name}}`）
- コマンド出力が必要な場合は別ステップで実行し、結果を変数として保持する
- 複数行のgitコミットメッセージは `-m` フラグを複数回使用する（例: `git commit -m "title" -m "body"`）

（例）
- 命名規則: lowerCamelCase、UPPER_SNAKE_CASE等
- フォーマットルール
- コメント記述ルール

## AIレビューツールの使用ルール【重要】

AI-DLCプロンプトで「AIレビュー」を実行する際は、以下のルールに従うこと。

### 使用するスキル

**レビュー種別に対応するSkillを使用する**:

| 呼び出し元 | Skills呼び出し |
|-------------|----------------|
| 計画承認前 | `skill="reviewing-construction-plan"` |
| 設計レビュー | `skill="reviewing-construction-design"` |
| コード生成後 | `skill="reviewing-construction-code"` |
| 統合とレビュー | `skill="reviewing-construction-integration"` |
| Intent承認前 | `skill="reviewing-inception-intent"` |
| ストーリー承認前 | `skill="reviewing-inception-stories"` |
| Unit定義承認前 | `skill="reviewing-inception-units"` |
| デプロイ計画承認前 | `skill="reviewing-operations-deploy"` |
| PRマージ前 | `skill="reviewing-operations-premerge"` |

### 呼び出し方法

```text
skill="reviewing-[stage]", args="[レビュー対象] 優先ツール: [codex|claude|gemini]"
```

- レビュー種別はreview-flow.mdの「レビュー種別の決定」に従い選択する
- 優先ツールは `.aidlc/config.toml` の `[rules.reviewing].tools` の先頭を使用する
- ツール選択の最終決定はスキル内部の責務（優先ツールはヒントのみ）

## ローカルレビュー必須ルール【重要】

外部レビューツール（Codex、Claude等）を呼び出す**前に**、必ずローカルで差分レビューを実施すること。

**手順**:

1. `git diff` または `git diff --cached` で変更内容を確認する
2. 変更の妥当性・漏れ・意図しない変更がないかを目視確認する
3. 問題があれば修正・コミットしてから外部レビューに進む

**対象タイミング**:

- Construction Phase: AIレビュー実施前（コード生成後、統合とレビュー時）
- Operations Phase: PRマージ前レビュー（7.12）でCodexレビューをトリガーする前

**理由**: 外部ツールに依存する前にセルフチェックすることで、明らかな問題を早期に発見し、レビューの往復回数を減らす。

## Worktree運用ルール【重要】

このプロジェクトでは `.worktree/dev` で開発を行う。

### ブランチ運用フロー

1. **サイクル開始時**: dev worktree で `cycle/vX.X.X` ブランチを作成して作業
2. **アップグレード時**: `upgrade/vX.X.X` ブランチを作成（aidlc-setupスキルが案内）
3. **PR作成・マージ**: dev worktree からpush → PRマージ
4. **マージ後**: `bin/post-merge-sync.sh` を実行（main pull + detached HEAD化 + マージ済みブランチ削除を自動化）
5. **次サイクル開始時**: dev worktree で最新の `origin/main` から新ブランチを作成

```bash
# マージ後（dev worktreeで実行）
bin/post-merge-sync.sh

# dry-runで事前確認する場合
bin/post-merge-sync.sh --dry-run

# リモートブランチ削除の確認をスキップする場合
bin/post-merge-sync.sh --yes

# 次サイクル開始（dev worktreeで実行）
git fetch origin
git checkout -b cycle/vX.X.X origin/main
```

### 注意事項

- dev worktree から `main` への checkout は不可（メインリポジトリが使用中のため）
- main の更新は `bin/post-merge-sync.sh` が親リポジトリ側で自動実行する
- `cycle/` および `upgrade/` プレフィックスのマージ済みブランチのみ削除対象（安全制約）

## 設計レビュー時のガイド照合ルール【重要】

Construction Phase 1（設計フェーズ）の設計レビュー時に、既存ガイド文書との整合性を確認すること。

**手順**:

1. 設計対象に関連するガイドを特定する（メタ開発時は `skills/aidlc/guides/` を参照、メタ開発時の例外参照）
2. 設計内容がガイドの規約に違反していないか確認する
3. 違反がある場合: 設計を修正するか、ガイド側の改訂を提案する
4. 対象ガイドが存在しない場合: スキップ（問題なし）

**対象ガイド例**:

- シェルスクリプトの終了コードに関する設計 → `guides/exit-code-convention.md`
- エラーハンドリングに関する設計 → `guides/error-handling.md`
- バックログ管理に関する設計 → `guides/backlog-management.md`

**理由**: v1.27.3 で終了コード規約に反する設計（警告付き完了を exit 2 にする）がユーザー指摘まで検出できなかった。既存ガイドとの照合を設計レビューの観点に含めることで、既知の規約違反を早期に検出する。
