# PRFAQ: AI-DLC Starter Kit v2.5.0（自己改善ループ導入）

## Press Release（プレスリリース）

**見出し**: AI-DLC v2.5.0、開発プロセスを「自分で振り返る」AI に進化

**副見出し**: 4 階層設定整理と振り返りステップ導入で、AI-DLC を使うほど skill 自体が改善されるループを実装

**発表日**: 2026 年 5 月（v2.5.0 タグ作成日）

**本文**:

[背景]

AI-DLC スターターキットを使った開発では「Codex マージ前レビューで指摘 5 反復」「skill 仕様の曖昧さで毎回違う実装結果」など、利用者の暗黙知として蓄積されるが skill 本体には還元されない構造的課題がありました。empirical-prompt-tuning による試行で、課題の本質は「skill 仕様書は明確でも、過去事例のショートカット推論で誤読する」ことだと判明。これを継続的に補正する経路が必要です。

[プロダクト]

v2.5.0 は AI-DLC に 2 つの仕組みを追加します:

1. **振り返りステップ**: 各サイクルの Operations 完了時に `retrospective.md` が自動生成され、「何が起きたか / なぜ起きたか / skill 起因か（3 問自問）」を記録します。skill 起因と判定された項目は `feedback_mode = "mirror"` 設定下で「下書き → AskUserQuestion で承認 → upstream Issue 起票」のフローを実行し、安全に upstream への改善提案を発行できます。

2. **4 階層設定の整理**: `rules.reviewing.mode` などの「個人好み」設定 7 キーが project 共有の `.aidlc/config.toml` に固定書き込みされる問題を解消します。skill 内蔵 `defaults.toml` + user-global `~/.aidlc/config.toml` で個別管理できるようになり、チーム開発時の選好混入事故を防ぎます。`aidlc-migrate` が既存プロジェクトに対して移行を提案します（強制移動はしない非破壊運用）。

[顧客の声]

> 「Codex に同じことを 5 回指摘されるサイクルが終わるかも。retrospective に残せば次のサイクルで同種の指摘を Inception 冒頭に思い出せる。」（個人開発者）

> 「個人好み設定が project 共有に勝手に書き込まれないので、チームメンバーごとの環境で setup できる。スターターキットを安心して導入できるようになった。」（チームリード）

> 「メタ開発側で `mirror` モードを使えば、利用者から Issue が上がる前に自分たちで欠陥を補正できる。フィードバックループが内部化される。」（メタ開発者）

[今後の展開]

v2.5.0 で導入する `silent` / `mirror` モードに加え、運用データが蓄積した v2.6.x 以降で `on` モード（自動起票）と「優先度フィルタ」「過去サイクル横断の重複検出」を追加予定です。AI-DLC 自身が AI-DLC で改善されるループを次の段階へ進めます。

## FAQ（よくある質問）

### Q1: v2.5.0 にアップグレードすると既存プロジェクトの設定が変わりますか?

A: いいえ、変わりません。既存 `.aidlc/config.toml` の値は 4 階層マージ仕様で引き続き優先されます。`aidlc-migrate` が「個人好みキーを user-global に移動するか」を提案しますが、ユーザーが「そのまま残す」を選択すれば設定ファイルは一切変更されません。

### Q2: 振り返り機能は強制ですか? 嫌な場合はオフにできますか?

A: `feedback_mode = "disabled"` で完全に無効化できます。デフォルトは `silent`（ローカル `retrospective.md` 生成のみ、upstream 送信なし）で、最も控えめな挙動です。upstream にフィードバックしたい人は `mirror` に切り替え、対話で承認したものだけ送信します。

### Q3: 「skill 起因」と判定される条件は何ですか?

A: 3 問のうち 1 問以上に「Yes（10 文字以上の具体的な引用付き）」と回答できた項目のみが skill 起因候補となります。「該当」「あり」だけの空虚な引用は警告 + 自動的にローカル記録のみへダウングレードされ、upstream への誤起票を防ぎます。

### Q4: 振り返り Issue が氾濫しませんか?

A: 重複検出（同一サイクル内で skill 引用箇所一致 / タイトル類似度しきい値超過の項目をグルーピング）と、サイクル毎上限（`feedback_max_per_cycle` のデフォルト 3）の 2 段階で氾濫を緩和します。上限超過分はローカル記録のみで AskUserQuestion 表示も抑制されます。

### Q5: v2.5.0 で対応しない範囲は何ですか?

A: 以下は v2.6.x 以降に分離しています:

- `feedback_mode = "on"`（自動起票、メタ開発向け）
- 優先度フィルタ（自動起票候補と Mirror 推奨の振り分け）
- 過去サイクル横断の AI 類似判定による重複検出
- A1-3 パッチ提案（upstream PR 自動生成）
- review-flow.md の指摘対応判断フローと連動した過去指摘クラスタリング

### Q6: 開発者は何から始めればいいですか?

A: v2.5.0 リリース後、`/aidlc-setup` または `/aidlc-migrate` を実行してください。新規プロジェクトは個人好みキーが含まれない `.aidlc/config.toml` が生成されます。既存プロジェクトには移動提案ダイアログが表示されます。`feedback_mode` を試したい場合は `~/.aidlc/config.toml` に以下を追加します:

```toml
[rules.retrospective]
feedback_mode = "mirror"
feedback_max_per_cycle = 3
```

次回 Operations Phase 完了時に retrospective.md が生成され、skill 起因候補について承認ダイアログが表示されます。
