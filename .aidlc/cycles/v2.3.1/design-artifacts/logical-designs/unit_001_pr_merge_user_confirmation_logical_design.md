# 論理設計: PRマージのユーザー判断化

## 概要

Operations Phase ステップ 7.13 にマージ実行前のユーザー確認ゲートを追加する。変更対象ファイルと影響範囲を定義する。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。

## アーキテクチャパターン

Operations Phase のステップファイルは4層構造:

1. **インデックス層**（`index.md`）: 分岐ロジック・ゲート判定の一元管理
2. **集約層**（`02-deploy.md`）: サブステップ一覧と遷移制御
3. **詳細手順層**（`operations-release.md`）: 各ステップの具体的手順
4. **完了基準層**（`03-release.md`）: 実行ルール・完了基準・完了時の確認

本変更は主に詳細手順層に実装し、インデックス層にインタラクション種別マトリクスを追加する。

### 影響分析: `03-release.md`（変更不要）

`03-release.md` は完了基準として「PRマージ完了」を列挙し、7.8-7.13 を「PR準備完了後のレビュー・マージ作業」と参照している（L19, L31）。この記述は「PRマージが完了していること」を要求するのみであり、マージ前のユーザー確認有無には依存しない。

- `03-release.md` L19: 「PRマージ完了」→ 完了基準はマージの結果状態を参照するため、確認ゲート追加後も基準は変わらない

## インターフェース設計

### `index.md` §2.6 に追加するインタラクション種別マトリクス

現在の §2.6 はゲート発生箇所の一覧のみだが、「ゲート承認」と「ユーザー選択」を区別するマトリクスに拡張する:

| ステップ | インタラクション種別 | manual | semi_auto |
|---------|-------------------|--------|-----------|
| ステップ1（変更確認）の選択 | ゲート承認 | ユーザー確認 | auto_approved（§2.3） |
| 各対話形式ステップ計画承認 | ゲート承認 | ユーザー確認 | auto_approved |
| リリース準備計画承認 | ゲート承認 | ユーザー確認 | auto_approved |
| PR Ready 化承認 | ゲート承認 | ユーザー確認 | auto_approved |
| **PRマージ実行（7.13）** | **ユーザー選択** | **ユーザー確認** | **ユーザー確認** |

**注記**: 「ユーザー選択」は SKILL.md「AskUserQuestion使用ルール」に定義されたインタラクション種別であり、`automation_mode` に関わらず常にユーザー確認が必要。PRマージは破壊的・不可逆操作であるためこの分類に該当する。

### `operations-release.md` ステップ 7.13 の変更後インターフェース

#### 入力（MergeReadinessContext）

| 項目 | 型 | 取得元 | 説明 |
|-----|---|--------|------|
| `pr_number` | Integer | ステップ 7.8 で取得済み | 対象PR番号 |
| `resolved_merge_method` | MergeMethod (`merge`\|`squash`\|`rebase`) | `merge_method` 設定（`ask` 時は先行の選択結果） | 確定済みマージ方法 |
| `gh_availability` | GhAvailabilityStatus (`available`\|その他) | プリフライトの `gh_status` | GitHub CLI 利用可否 |

#### 出力（MergeExecutionResult）

| 結果 | 説明 | 後続処理 |
|-----|------|---------|
| `merged` | マージ成功 | 完了処理へ |
| `auto-merge-set` | 自動マージ設定済み | 待機案内 |
| `error:<code>` | エラー発生 | エラー対処フロー |
| `user_rejected` | ユーザーがマージを拒否 | マージ中断・ユーザー判断 |

## 処理フロー概要

### 変更後のステップ 7.13 処理フロー

**ステップ**:

1. PR本文の `Closes #XX` を最終確認（既存、変更なし）
2. `gh_availability` チェック（既存、変更なし）
   - `!= available` → 手動案内して終了
3. `merge_method` 判定（既存、変更なし）
   - `ask` → マージ方法をユーザーに質問（ユーザー選択）
   - その他 → 設定値を `resolved_merge_method` として使用
4. **【新規追加】マージ実行確認**（ユーザー選択）
   - 提示情報: `MergeReadinessContext`（PR番号、マージ方法）
   - 質問: 「PR #{pr_number} を {resolved_merge_method} マージしてよろしいですか？」
   - ユーザー承認 → 次へ / 拒否 → `user_rejected` として中断
5. マージスクリプト実行（既存、変更なし）
6. 結果処理（既存、変更なし）

**関与するコンポーネント**: `operations-release.md`（手順変更）、`index.md`（マトリクス追記）、`02-deploy.md`（参照注記）

## 実装上の注意事項

- `merge_method=ask` の場合、マージ方法の選択（既存）とマージ実行確認（新規）が連続する。2回のユーザー確認が発生するが、これは意図的な設計（方法選択と実行可否は別の判断）
- `automation_mode=manual` 時の既存動作に退行がないこと。既存フローでは `merge_method=ask` 時のみユーザー確認が存在し、その他は確認なくマージ実行していた。変更後は全ケースでマージ実行前に確認が入る
- `operations-release.sh merge-pr` スクリプト自体は変更しない。呼び出し前の制御フローのみ変更
