# ユーザーストーリー

## Epic: プリフライトチェック・ワークフロー改善

### ストーリー 1: defaults.toml 存在チェックの正確化
**優先順位**: Must-have

As a AI-DLC利用者
I want to プリフライトチェックで `config/defaults.toml` がSKILL.mdのパス解決ルールに従って正しく検出される
So that 偽陰性の「defaults.toml: 不在」警告が表示されなくなる

**受け入れ基準**:
- [ ] プリフライトチェック実行時に `config/defaults.toml` がスキルベースディレクトリから解決される
- [ ] スキルベースディレクトリ配下に `config/defaults.toml` が存在する場合、「✓ defaults.toml: 存在」と表示される
- [ ] `config/defaults.toml` が実際に不在の場合のみ、「⚠ defaults.toml: 不在」警告が表示される

**技術的考慮事項**:
- SKILL.mdの「パス解決」ルールに従い、`config/` をスキルベース相対で解決

### ストーリー 2: 推奨・提案メッセージの応答確保（共通基盤）
**優先順位**: Must-have

As a AI-DLC利用者
I want to 推奨・提案メッセージ表示後に応答する機会が確保される
So that 重要な推奨事項を見落とさずに対応できる

**受け入れ基準**:
- [ ] SKILL.mdの「実行判断・対話規約」セクションに推奨・提案メッセージの応答確保ルールが追記されている
- [ ] ルールに以下の使い分け基準が定義されている: (a) ユーザーアクションが必要な推奨 → 選択肢を表示しユーザー応答を待つ (b) 情報提供のみの推奨 → 表示して進行、履歴に記録
- [ ] `steps/inception/01-setup.md` ステップ9-3: mainがbehind状態の場合、「取り込む / スキップして続行」の選択肢が表示され、ユーザー応答を待つ
- [ ] `steps/common/preflight.md`: warn件数が1件以上の場合、「続行する / 問題を解決してから再チェック」の選択肢が表示され、ユーザー応答を待つ（blockerがない場合のみ）
- [ ] `automation_mode=semi_auto` のゲート自動承認条件（`rules-automation.md`のフォールバック条件テーブル）が変更されていない
- [ ] 応答待ち中にユーザーが中断した場合、progress.mdに判断待ち状態が記録され、再開時に判断待ち項目を参照できる

**対象外の異常系**:
- エクスプレスモード・full_autoモードでの動作変更

**技術的考慮事項**:
- semi_autoモードの対話フローを阻害しないバランスが必要
- 既存の3種別分類（ゲート承認、ユーザー選択、情報収集）に「推奨・提案」カテゴリを追加する方向
- 実装にはAskUserQuestionツールを使用する

### ストーリー 3: Operations Phaseのリモート同期チェック追加
**優先順位**: Must-have

As a AI-DLC利用者
I want to Operations Phase開始時にリモートブランチとの差分が確認される
So that 古いコードベースで作業を進めるリスクを低減できる

**受け入れ基準**:
- [ ] Operations Phase開始時（01-setup.md）に `git fetch` + 現在ブランチの追跡先との差分件数チェックが追加されている
- [ ] behind件数が1以上の場合、「取り込む / スキップして続行」の選択肢が表示され、ユーザー応答を待つ
- [ ] behind件数が0の場合、「リモートと同期済みです」と表示して自動続行する
- [ ] `git fetch` 失敗時（オフライン等）、「⚠ リモート確認失敗（オフライン等）」と警告表示してスキップする
- [ ] upstream未設定時、「⚠ 追跡ブランチ未設定のため同期チェックをスキップします」と警告表示してスキップする
- [ ] 既存のステップ7（03-release.md）の `verify-git` チェックと判定基準が一貫している（同じgit操作ベース）

**対象外**:
- ahead/diverged 状態の判定と対応（本ストーリーではbehindのみ対象）

**調査タスク（設計フェーズで実施）**:
- 取り込み漏れの原因調査（原因、再現条件、対策方針）を設計ドキュメントに記録

**技術的考慮事項**:
- Operations Phase ステップ7に既存の `operations-release.sh verify-git` がある（再利用可能か検討）
- Operations Phase開始時のチェックはブロッカーではなく推奨レベル（warn severity）
- 実装にはAskUserQuestionツールを使用する
