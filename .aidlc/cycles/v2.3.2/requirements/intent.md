# Intent（開発意図）

## プロジェクト名

ai-dlc-starter-kit

## 開発の目的

**主目的**: v2.3.1 までに蓄積した3件のワークフロー改善・バグ修正を実施する。

1. **#561 semi_autoモードの不要ユーザー確認修正（bugfix）**: semi_auto設定時にフォールバック条件非該当のゲート承認でユーザー確認が発生する問題を修正する。修正対象は「ゲート承認」に分類されるポイントのみであり、「ユーザー選択」「情報収集」は対象外
2. **#556 対話UIに「設定に保存」選択肢追加（feature）**: config.tomlで変更可能なデフォルト動作の質問に、選択を設定ファイルへ永続保存するオプションを追加する
3. **#546 PR Closes記載時の部分対応Issue自動判別（feature）**: Unit定義ファイルの「関連Issue」セクションに部分対応注記を記法として導入し、`get-related-issues` で部分対応と完全対応を区別してCloses/Relatesの自動判定を行う

## ターゲットユーザー

AI-DLC Starter Kitを使用する開発者（主に自身のメタ開発）

## ビジネス価値

- semi_autoモードの信頼性向上（#561）: 自動承認が期待通りに動作し、不要な中断が減少する
- 設定カスタマイズの利便性向上（#556）: 対話フロー内で設定を変更でき、config.toml手動編集が不要になる
- リリース品質の向上（#546）: 部分対応Issueの誤Closeを防止し、Issue管理の正確性が向上する

## 成功基準

- semi_autoモードでフォールバック条件非該当のゲート承認が自動承認される（#561）
- manualモードの確認フローが不変であること（#561 非回帰条件）
- semi_autoモードでもユーザー選択・情報収集は従来どおりAskUserQuestion必須であること（#561 非回帰条件）
- 対象質問に「設定に保存」選択肢が表示され、選択時に指定層のconfig.tomlが更新される（#556）
- 既存の設定読込優先順位（local > project > user > defaults）が不変であること（#556 非回帰条件）
- PR Closes記載時に部分対応Issueが自動検出され、完全対応のみClosesに含まれ、部分対応はRelatesとして記載される（#546）
- 注記のない既存Unit/Issueは「完全対応」として扱われ、既存運用を壊さないこと（#546 非回帰条件）

## 含まれるもの

### #561 semi_autoゲート承認修正

- ステップファイル内の全ゲート承認ポイントで「セミオートゲート判定」への参照を明確化
- 対象: SKILL.mdの「AskUserQuestion使用ルール」テーブルで「ゲート承認」に分類されるポイント
- 非対象: 同テーブルで「ユーザー選択」「情報収集」に分類されるポイント（これらは変更しない）

### #556 設定保存機能

- 「設定に保存」対象の質問: `rules.git.merge_method`（ask時）、`rules.git.branch_mode`（ask時）、`rules.git.draft_pr`（ask時）のデフォルト動作系質問
- 保存先選択ルール: ユーザーに `config.toml`（プロジェクト共有）/ `config.local.toml`（個人設定）を選択させる。デフォルトは `config.local.toml`
- config.toml/config.local.toml安全書き込み機能（`write-config.sh` の新規作成）
- 初回保存時の挙動: ファイル未存在時は新規作成、セクション未存在時は安全に追加、既存コメント・他キーは維持
- 既存の `read-config.sh` の4層読込ロジックには変更を加えない

### #546 部分対応Issue自動判別

- Unit定義ファイルの「関連Issue」セクションに部分対応記法を導入: `#NNN（部分対応）`
- ソースオブトゥルース: Unit定義ファイルの「関連Issue」セクション
- 判定ルール: `#NNN` のみ → 完全対応（Closes）、`#NNN（部分対応）` → 部分対応（Relates）
- 記法欠落・不正時のデフォルト動作: 注記なしは「完全対応」扱い（既存互換）
- `get-related-issues` の出力に Closes/Relates 区別を追加
- PR Closes記載確認ステップでの自動検証追加

## 含まれないもの

- full_autoモードの実装（#398は別スコープ）
- Construction復帰時のステップレベルチェックポイント（#554は別スコープ）
- サイクルスコープ妥当性チェック強化（#552は別スコープ）
- AskUserQuestionツール自体の拡張（ツールのオプション追加等）
- 既存PRの自動修正・過去サイクル成果物への注記追加

## 既存機能への影響

### #561

- **manualモード**: 確認フロー不変。全ゲートでユーザー確認必須のまま
- **semi_autoモード**: ゲート承認のみ影響。ユーザー選択（PRマージ等）と情報収集は従来どおり

### #556

- **設定読込優先順位**: `read-config.sh` の4層ロジック不変。書き込み機能は独立した `write-config.sh` として追加
- **手動編集**: 既存の手動config.toml編集との共存。書き込みはファイル未存在時の新規作成、セクション・キーの追加・更新に対応。既存コメントや他キーは維持する

### #546

- **既存Unit**: 注記なしの `#NNN` は「完全対応」扱い（後方互換）
- **PR生成フロー**: 自動判定結果をPR本文に反映する前に確認ステップを追加

## 期限とマイルストーン

パッチリリースのため、1サイクルで完了予定

## 制約事項

- メタ開発プロジェクトのため、スキル内リソース編集は `skills/aidlc/` 配下で行う
- config.toml書き込みはTOMLフォーマットを壊さない安全な方法で実施する
- 既存のautomation_mode仕様との後方互換性を維持する
- 部分対応記法は最小限の規約追加とし、既存成果物のフォーマットを壊さない
