# PRFAQ: AI-DLC Starter Kit v2.3.2

## Press Release（プレスリリース）

**見出し**: AI-DLC Starter Kit v2.3.2 — ワークフロー品質パッチ

**副見出し**: semi_auto自動承認の信頼性修正、設定保存UI、部分対応Issue判別の3改善を一括リリース

**発表日**: 2026-04-11

**本文**:

v2.3.1までの運用で蓄積された3件のワークフロー改善要望に対応するパッチリリースです。

semi_autoモードでフォールバック条件非該当にもかかわらずユーザー確認が発生する問題（#561）を修正し、自動承認の信頼性を向上させました。対話中の設定選択を設定ファイルに保存する機能（#556）を追加し、同じ選択の繰り返しを削減しました。PR Closes記載時に部分対応Issueを自動判別する機能（#546）を追加し、Issueの誤Close防止を実現しました。

## FAQ（よくある質問）

### Q1: semi_autoモードの修正はどのような影響がありますか？
A: ステップファイル内のゲート承認ポイントの記述を明確化し、LLMが「ゲート承認」「ユーザー選択」「情報収集」を正しく区別できるようにします。manualモードやユーザー選択（PRマージ等）の動作は一切変更されません。

### Q2: 「設定に保存」機能はどの質問で使えますか？
A: 初期リリースでは `merge_method`、`branch_mode`、`draft_pr` の3設定（いずれもask設定時）が対象です。保存先は `config.local.toml`（個人設定）がデフォルトで、`config.toml`（プロジェクト共有）も選択可能です。

### Q3: 部分対応Issueの判別はどのように機能しますか？
A: Unit定義ファイルの「関連Issue」セクションで `#NNN（部分対応）` と記載すると、PR作成時にClosesではなくRelatesとして記載されます。注記なしの `#NNN` は従来どおり完全対応（Closes）として扱われるため、既存の運用は変更されません。

### Q4: write-config.shとread-config.shの関係は？
A: `write-config.sh` は設定書き込み専用の新規スクリプトで、`read-config.sh` の4層読込ロジックには一切変更を加えません。既存の設定優先順位（local > project > user > defaults）は維持されます。
