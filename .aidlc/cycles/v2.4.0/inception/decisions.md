## DR-001: #596 を Unit 002（スクリプト挙動変更）と Unit 003（ドキュメント・周知）に分割

- **ステップ**: ユーザーストーリー / Unit 定義
- **日時**: 2026-04-23

### 背景

#596（`update-version.sh` の `starter_kit_version` 上書き bug）を 1 ストーリー（当初ストーリー 6）に集約していたが、AI レビュー Set 1 P2 #4 で「変更対象が広すぎ Small/Independent を満たしにくい」と指摘された。スクリプト修正、出力仕様変更、テンポラリ削除、存在チェック維持、CHANGELOG、README、テスト、dry-run 実証、設計ドキュメント更新を 1 ストーリーに混載していたため、見積もり粒度が粗く Done 条件がぶれやすい。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 1 ストーリー / 1 Unit のまま維持 | 関連性の高い変更を一括実施できる | 見積もり粒度が粗い、Construction で変更競合しやすい、レビューでも完了条件が曖昧 |
| 2 | スクリプト挙動変更（6a/Unit 002）とドキュメント・周知（6b/Unit 003）に分割 | 各 Unit の Done 条件が明確、Unit 003 は Unit 002 完了後に着手と依存関係も明示できる | Unit 数が増える（4→5）、CHANGELOG など共有ファイルの所有境界を別途調整が必要 |

### 決定

**選択肢 2（分割）** を採用

### トレードオフと判断根拠

- **得たもの**: ストーリー / Unit 単位の Done 条件明確化、Construction Phase での編集競合リスク低減、レビュー時の完了条件確認の容易さ
- **犠牲にしたもの**: Unit 数増加（5→7、最終的にも 7 で収まる）、CHANGELOG の節単位所有分離（`#596` 節は Unit 003 / `#597` 節は Unit 007）の追加調整工数
- **判断根拠**: Unit 003 完了報告で Unit 002 実装結果に整合した CHANGELOG 記述ができる依存順が明確になることが、Construction Phase の生産性向上に直結する。Unit 数増加のオーバーヘッドより、Done 条件明確化のメリットが大きい

## DR-003: #595 を「純削除」固定（代替判定条件追加は別 Issue）

- **ステップ**: ユーザーストーリー / Unit 定義
- **日時**: 2026-04-23

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 純削除のみ（書き換えしない） | 最小修正、テスト・リスク最小、Done 条件明確 | メタ開発判定の代替手段が失われる（現状も常に不成立だったため実質無影響） |

| 3 | ストーリーで「どちらか」と曖昧化（Construction で判断） | 設計時の柔軟性 | Done 条件不確定、レビュー時に完了判定不能、Independent/Small/Testable 弱化 |

### トレードオフと判断根拠

- **得たもの**: ストーリー 5 / Unit 004 の Done 条件明確化、最小修正で v2.4.0 リリース内に収まる、検証コスト低減

- **判断根拠**: 現状でもメタ開発判定は常に不成立で実害がないため、純削除で混乱の根本原因（v1 遺物言及）を消す方が優先度が高い。代替判定条件の必要性は v2.5.0 以降のバックログ Issue で別途議論し、必要時に独立ストーリー化する

## DR-004: Operations Phase の Milestone 不在判定を 4 段階優先順位に固定

- **ステップ**: ユーザーストーリー / Unit 定義
- **日時**: 2026-04-23

### 背景

Operations Phase の Milestone close ストーリー（ストーリー 2 / Unit 006）で、Milestone 不在検出時の fallback 作成と既存 Milestone（open / closed の各状態）の扱いが、AI レビュー Set 1 P2 #3 と Set 2 P2 #1 で「異常系が具体化されていない、判定順次第で挙動が分かれる」と指摘された。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | open Milestone のみ判定し、closed の存在は無視 | シンプル | 同名 closed の意図せぬ再利用や誤再オープンのリスク |
| 2 | 4 段階優先順位で評価（closed 1+→停止 / open 2+→停止 / open 1→再利用 / 両方 0→fallback 作成） | 全ケースの動作が一意に決定、誤再オープン防止 | 実装が複雑（4 ケース判定）、Markdown ステップ内で擬似コード or 表形式の明示が必要 |
| 3 | open / closed どちらか 1 件以上あれば既存利用、それ以外で fallback | 簡素 | closed が存在するときに warn なしで紐付けると意図確認なしで進行してしまう |

### 決定

**選択肢 2（4 段階優先順位で一本化）** を採用

### トレードオフと判断根拠

- **得たもの**: 実装者が解釈で補う余地のない判定規則、誤再オープン・重複紐付け・複数候補時の安全停止
- **犠牲にしたもの**: Markdown ステップ内に判定規則の明示記述（擬似コード or 表）が必要、実装複雑度の増加（4 ケース分岐）
- **判断根拠**: Milestone 操作はサイクル進捗の正確性に直結するため、安全側に倒した判定を一意に固定する方が、後年の運用混乱コストを抑えられる。重複や逆転を防ぐため「この優先順位で評価し最初に該当した規則のみを適用」と明示

## DR-005: `cycle-label.sh` / `label-cycle-issues.sh` を v2.4.0 で deprecated 残置（物理削除は後続サイクル）

- **ステップ**: Intent / ユーザーストーリー / Unit 定義
- **日時**: 2026-04-23

### 背景

#597 Milestone 運用本採用にあたり、サイクルラベル系スクリプト（`cycle-label.sh` / `label-cycle-issues.sh`）の扱いを確定する必要があった。一括物理削除も技術的には可能だが、blast radius が大きい（参照箇所の見落とし、後方互換ブランチでの利用残留など）。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | v2.4.0 で物理削除（Unit E の前倒し） | 一気にクリーンアップ、deprecated 残置による混乱なし | blast radius 大、残参照を全部見つけきれないリスク、本採用フロー（A-C）の検証と削除が同サイクルに混在 |
| 2 | v2.4.0 で deprecated 化（スクリプト先頭コメントと CHANGELOG 明記）、物理削除は後続サイクル（Unit E） | 本採用フロー検証に集中できる、削除前に deprecated 期間で残参照を発見できる | スクリプトファイルが残置されるため一見「現役」と誤認される可能性（コメントで明示するため大きくはない） |
| 3 | スクリプトをそのまま残し、フローからの呼び出しのみ削除（deprecation 注記なし） | 最小変更 | 後続サイクルで物理削除する判断材料が残らない、新規利用の追加リスク |

### 決定

**選択肢 2（deprecated 残置、物理削除は Unit E）** を採用

### トレードオフと判断根拠

- **得たもの**: 本採用フロー（Unit A-C）の検証に集中、deprecation 期間で残参照や利用者の追従状況を観察可能、後続サイクル Unit E での物理削除を安全実施できる
- **犠牲にしたもの**: スクリプトファイルが残置されるため、リポジトリ閲覧時に「現役か deprecated か」の判別がコメント参照に依存する（CHANGELOG と先頭コメントで明示するため致命的でない）
- **判断根拠**: blast radius の大きい変更を本採用フロー検証と同時に行わない原則。Intent でも Unit D-F は本サイクル外と明示済みであり、本判断はその境界線の具体化

## DR-006: v2.4.0 サイクル自身の Milestone を Inception 完了時に手動で作成（自己参照回避）

- **ステップ**: Intent / ユーザーストーリー
- **日時**: 2026-04-23

### 背景

v2.4.0 で Inception Phase に Milestone 作成手順（Unit B）を追加するが、その手順を v2.4.0 自身に適用するか（自己参照）が論点となった。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | v2.4.0 自身にも Unit B の手順を適用（更新中の手順を実使用） | 手順の動作検証が同サイクルで完結 | 更新前後の手順が混在、検証と実使用の二重化、レビュー時に「どの版の手順が動いたか」が曖昧 |
| 2 | v2.4.0 自身は Inception 完了時に手動で Milestone を作成し、Unit B の手順は v2.5.0 以降から標準適用 | 更新版手順の検証は次サイクル以降に純粋に行える、本サイクルの混乱回避 | v2.4.0 専用の手動運用タスク（T1）が必要、Unit 数とは別に運用追跡が必要 |

**選択肢 2（手動作成 + Unit B の標準適用は v2.5.0 以降）** を採用

### トレードオフと判断根拠

- **得たもの**: 更新版手順の自己参照混乱回避、v2.3.6 → v2.4.0 → v2.5.0 と Milestone 連続性を維持、Unit B のステップ手順が「v2.4.0 自身に適用しない」というサイクル固有注記なしの恒久手順として書ける
- **犠牲にしたもの**: 運用タスク T1 として user_stories.md 末尾に追加追跡が必要（Inception 完了処理タスクで必ず実施するチェックリスト化）
- **判断根拠**: 共有プロダクト手順にサイクル固有の例外注記を入れない原則（DR-005 の Unit 005 責務境界とも整合）。手動運用は T1 として追跡可能なら追加コスト最小

## DR-007: #595 で代替判定条件の追加を本サイクル対象外（別 Issue / 別ストーリー）

## DR-008: 公開ドキュメントは v2.4.0 以降を Milestone 運用に統一、旧サイクル運用は併記しない

- **ステップ**: ユーザーストーリー / Unit 定義
- **日時**: 2026-04-23

### 背景

公開ドキュメント（`docs/configuration.md` / `README.md` / `skills/aidlc/guides/` / `skills/aidlc/rules.md`）のサイクル運用記述を Milestone 運用に書き換える際、旧サイクル（v2.3.6 以前）のラベル運用との併記を残すか単一表現にするかが論点となった。AI レビュー Set 2 P3 #3 で「強い設計判断であり DR 一覧に入れるべき」と指摘された。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | Milestone 運用と旧ラベル運用を併記（v2.4.0 以降は Milestone、それ以前はラベル） | 旧サイクル参照時の混乱回避、移行期の利用者向け配慮 | 公開ドキュメントが冗長化、旧運用が「現役オプション」と誤認されるリスク |
| 2 | Milestone 運用に統一（旧運用は併記なし）、過去サイクル追跡は CHANGELOG / `.aidlc/cycles/v*/` ディレクトリ / `cycle:v*` ラベル（deprecated 物理残置）で代替 | 公開ドキュメント簡潔化、現役と deprecated の区別が明確、新規利用者が迷わない | 旧サイクル参照時に CHANGELOG / ディレクトリ / 残置ラベルを併用する必要 |
| 3 | 公開ドキュメントは触らず、Markdown ステップ手順だけ更新 | 最小変更 | 公開ドキュメントとフローの乖離、利用者周知不足 |

### 決定

**選択肢 2（Milestone 運用に統一、旧運用は併記なし）** を採用

### トレードオフと判断根拠

- **得たもの**: 公開ドキュメントの簡潔性と一貫性、新規利用者の理解容易、現役と deprecated の境界明確化
- **犠牲にしたもの**: 旧サイクル参照時に CHANGELOG / `.aidlc/cycles/v*/` ディレクトリ / `cycle:v*` ラベル残置を併用する手間（CHANGELOG にこの方針を明記することで利用者周知）
- **判断根拠**: 過去サイクルの追跡手段は既に十分（CHANGELOG / ディレクトリ / ラベル残置）。公開ドキュメントに旧運用を残すと「Milestone と ラベル どちらが推奨か」の誤解を生み、本採用の意図が薄れる。CHANGELOG に方針を明記すれば旧サイクル参照時の戸惑いも防げる
