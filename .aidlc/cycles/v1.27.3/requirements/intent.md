# Intent（開発意図）

## プロジェクト名
AI-DLC Starter Kit v1.27.3

## 開発の目的
エクスプレスモードの質改善を主目的とし、併せてシェルスクリプトの終了コード規約統一（#397）および外部依存の文書化（#396）を行う。

**主目的: エクスプレスモード再設計**
エクスプレスモードは現在 `depth_level=minimal` と結合しており、成果物の簡略化がフェーズ連続実行の前提になっている。これにより質が下がるリスクがある。エクスプレスモードを「フェーズ連続実行（フロー制御）」に純化し、depth_level とは独立して動作するよう再設計する。判定基準は Unit 数の固定制限ではなく、Unit の複雑度に基づく適格性判定とする。

**副次対応: 終了コード規約統一（#397）**
シェルスクリプトの終了コード（0: 成功 / 1: バリデーションエラー / 2: システムエラー）の使い分けが統一されていない問題を修正する。

**副次対応: 外部依存の文書化（#396）**
README に claude-skills リポジトリの依存関係とインストール方法を記載する。

## ターゲットユーザー
AI-DLC Starter Kit を使用する開発者（AI エージェント含む）

## ビジネス価値
- エクスプレスモードが depth_level=standard でも利用可能になり、成果物の質を維持したまま連続実行の恩恵を得られる
- 終了コード規約の統一によりエラーハンドリングの信頼性が向上する
- 外部依存の文書化により初期導入のハードルが下がる

## スコープ内（変更対象）
- **エクスプレスモード**: 起動条件の再定義、複雑度判定ロジックの新設、関連プロンプト（`common/rules.md` エクスプレスモード仕様、`inception.md` ステップ4b、`construction.md` エクスプレス関連）の改修
- **終了コード統一**: `squash-unit.sh`、`migrate-config.sh`、`update-version.sh`、`post-merge-sync.sh`、`post-merge-cleanup.sh`、`pr-ops.sh`、`issue-ops.sh`、`check-issue-templates.sh`、`migrate-backlog.sh`、`setup-ai-tools.sh`、`aidlc-setup.sh` の終了コード修正、規約ガイド文書の新設
- **外部依存文書化**: `README.md` に claude-skills リポジトリの説明セクション追加

## 成功基準
- エクスプレスモードが depth_level に依存せず、Unit の複雑度で適用可否を判定する
- 複雑度判定ロジックが定義・実装され、判定結果に基づいてエクスプレスモードの適用可否が制御される（判定項目・区分・条件は下記「複雑度判定の最小要件」参照）
- Unit 数 2-3 でも判定基準を満たせばエクスプレスモードが適用される（検証シナリオで確認）
- `depth_level=standard` でエクスプレスモード適用時に、standard 相当の成果物が生成されること（質の維持）
- 既存の `depth_level=minimal` 利用時の挙動および `start express` コマンドの互換性が維持されること
- manual/semi_auto モードで回帰がないこと
- 全シェルスクリプトで終了コード規約（0: 成功 / 1: バリデーションエラー / 2: システムエラー）が統一されている
- README に claude-skills リポジトリの依存関係とインストール方法が記載されている

## 期限とマイルストーン
1サイクルで完了

## 制約事項
- エクスプレスモードの再設計は既存の manual/semi_auto モードの動作に影響を与えない
- 終了コード修正はプロンプト側の判定ロジックとの整合性を維持する
- `docs/aidlc/` は直接編集禁止（`prompts/package/` を編集）

## スコープ外
- full_auto モードの新設（#398 として別サイクルで対応）
- depth_level=minimal の成果物要件自体の変更

## 不明点と質問（Inception Phase中に記録）

[Question] エクスプレスモードの複雑度判定基準をどう定義するか？
[Answer] AIが Unit の受け入れ基準・依存関係・技術的リスクを評価し、自律実行の適格性を判定する。詳細設計は Construction Phase で行うが、Intent時点での最小要件は以下の通り。

### 複雑度判定の最小要件（Intent時点の確定事項）

**評価対象項目**:
1. Unit の受け入れ基準の明確さ（曖昧な基準がないか）
2. Unit 間の依存関係の複雑さ（循環依存・多段依存がないか）
3. 技術的リスクの有無（未知の技術・外部API連携等）
4. 変更影響範囲の大きさ（既存コードへの影響が限定的か）

**判定結果の区分**:
- `eligible`: エクスプレスモード適用可（全項目が低複雑度）
- `ineligible`: エクスプレスモード適用不可（1項目でも高複雑度）

**適用条件**:
- 全 Unit が `eligible` と判定されること
- Unit 数の上限はハードコーディングしない（複雑度判定で制御）

**Construction Phase で詳細化する範囲**:
- 各評価項目のスコアリング方法
- 閾値の具体値
- エッジケースの取り扱い
