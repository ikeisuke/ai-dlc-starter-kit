# ユーザーストーリー - v1.16.3

## Epic: 設定基盤の改善

### ストーリー 1: read-config.sh のデフォルト値統一 (#221-1)
**優先順位**: Must-have

As a AI-DLCを利用するAIエージェント
I want to read-config.sh を --default オプションなしで呼び出しても正常値を取得したい
So that 設定読み取り時のエラーリスクを排除し、プロンプトでの --default 指定漏れを防止できる

**受け入れ基準**:
- [ ] defaults.toml にプロンプト内で参照される全設定キー（現在の17キー + 新規追加キー `rules.commit.ai_author`, `rules.branch.mode`）のデフォルト値が定義されている
- [ ] `read-config.sh rules.reviewing.mode` のように --default なしで呼び出した場合、defaults.toml の値が返り終了コード0となる
- [ ] defaults.toml にもプロジェクト設定にもキーが存在しない場合、終了コード1が返る
- [ ] 既存の --default オプションは引き続き動作する（後方互換）
- [ ] プロジェクト設定やlocal設定がある場合、defaults.toml の値を上書きする（既存の優先順位を維持）

**技術的考慮事項**:
- defaults.toml は既にread-config.shの階層0で読み込まれているため、キー追加のみで対応可能
- 現在不足しているキー: `rules.commit.ai_author`, `rules.branch.mode` 等を洗い出して追加

### ストーリー 2: read-config.sh の一括読み取り (#221-2)
**優先順位**: Must-have

As a AI-DLCを利用するAIエージェント
I want to 複数の設定キーを1コマンドで一括取得したい
So that フェーズ開始時の冗長な個別呼び出しを削減し、処理効率を向上できる

**受け入れ基準**:
- [ ] `read-config.sh --keys rules.reviewing.mode rules.reviewing.tools rules.history.level` で3キーを一括取得できる
- [ ] 出力は `rules.reviewing.mode:required` のような key:value 形式で、1行に1キーずつ出力される
- [ ] キーが存在しない場合、その行は出力されない（他のキーの出力には影響しない）
- [ ] 全キーが存在しない場合、何も出力せず終了コード1を返す
- [ ] 1件以上のキーが取得できた場合、終了コード0を返す
- [ ] 既存の単一キー指定（`read-config.sh <key>`）は引き続き動作する
- [ ] `--keys` と単一キー指定（位置引数）を同時使用した場合、エラーメッセージを出力し終了コード2を返す

**技術的考慮事項**:
- `--keys` と単一キー指定は排他的（同時使用はエラー）
- env-info.sh と同じ key:value 形式で統一

### ストーリー 4: リセット告知にサマリ追加 (#220)
**優先順位**: Should-have

As a AI-DLCを利用する開発者
I want to リセット告知時に完了作業・リポジトリ状態・次アクションのサマリを確認したい
So that 次のセッションですぐに作業を再開でき、状況把握の時間を短縮できる

**受け入れ基準**:
- [ ] Unit完了時のリセット告知に、完了したUnit名と作業内容の要約が1行で表示される
- [ ] Phase完了時のリセット告知に、完了したPhaseと残Unit/次Phaseの情報が1行で表示される
- [ ] リポジトリ状態（現在のブランチ、未マージPR有無）が1行で表示される
- [ ] 次にやるべきアクション（次フェーズ名または次Unit名を含む）が1行で表示される
- [ ] サマリは3行で構成され、各行に以下の必須項目を含む: (1) サイクル番号+完了作業名、(2) ブランチ名+PR状態、(3) 次のアクション指示
- [ ] サマリの各行はセッション固有の実際の値（Unit名、ブランチ名、PR番号等）を含み、テンプレートの変数がそのまま残っていないこと

**依存関係**:
- ストーリー3（リセット指示の省略防止）の完了後に実装可能（リセットメッセージの構造変更が前提）

**技術的考慮事項**:
- AIが動的に生成する部分のため、プロンプトでサマリ生成の指示を追加
- git/ghコマンドでリポジトリ状態を取得する指示を含める

## Epic: ブランチ管理の改善

### ストーリー 5: ブランチ作成方式の設定固定化 (#214)
**優先順位**: Should-have

As a AI-DLCを繰り返し利用する開発者
I want to サイクルブランチの作成方式を設定で固定したい
So that 毎サイクル同じ選択をする手間を省ける

**受け入れ基準**:
- [ ] `docs/aidlc.toml` に `[rules.branch].mode` を設定できる
- [ ] `mode = "branch"` の場合、ステップ7で質問なしにブランチが自動作成される
- [ ] `mode = "worktree"` の場合、ステップ7で質問なしにworktreeが自動作成される
- [ ] `mode = "ask"` または未設定の場合、現行通りユーザーに質問する
- [ ] defaults.toml に `[rules.branch].mode = "ask"` がデフォルト値として定義されている
- [ ] 無効な値が設定された場合、警告を表示して "ask" として扱う
- [ ] `mode = "worktree"` かつ `[rules.worktree].enabled = false` の場合、警告を表示し "branch" にフォールバックする

**技術的考慮事項**:
- inception.md ステップ7の分岐ロジックを変更
- `[rules.worktree].enabled` との整合性チェックをステップ7の先頭に追加

## Epic: レビューフローの改善

### ストーリー 6: セルフレビューフォールバック (#216)
**優先順位**: Should-have

As a 外部レビューツールを設定していない環境のAIエージェント
I want to 外部ツールが利用できない場合に自身でセルフレビューを実施したい
So that レビューなしでマージされるリスクを低減し、品質を一定水準で維持できる

**受け入れ基準**:
- [ ] 外部レビューツール（Skills）が利用不可の場合、セルフレビューの選択肢が提示される
- [ ] セルフレビューでは、対応するレビュー種別（code/architecture/security/inception）の観点が適用される
- [ ] セルフレビュー結果は通常のAIレビューと同じ形式（指摘番号・重要度付き）で出力される
- [ ] 履歴記録に「セルフレビュー」であることが明記される
- [ ] `mode = "required"` の場合、セルフレビューが最優先の選択肢として提示される
- [ ] `mode = "recommend"` の場合、セルフレビューが推奨として提示される
- [ ] 外部ツールが利用可能な場合は従来通り外部ツールを使用する（セルフレビューは使用しない）

**技術的考慮事項**:
- review-flow.md のステップ3（利用可否チェック）とステップ6（AIレビュー不可時）を改修
- セルフレビューの品質は外部ツールに劣る可能性があるため、その旨を履歴に記録
