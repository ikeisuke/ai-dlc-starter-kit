# Unit 001 計画: post-merge-sync.sh修正

## 概要
リモートブランチが自動削除済みの場合にpost-merge-sync.shがexit 1を返す問題を修正する。

## 根本原因
1. 非対話環境で`read -rp`が失敗し、`set -e`によりexit 1で即終了する
2. GitHub自動削除済みブランチのstaleリモート追跡参照が残り、削除対象として列挙される

## 変更対象ファイル
- `bin/post-merge-sync.sh`

## モード優先順位（決定表）

| 条件 | 動作 | 理由 |
|------|------|------|
| `--dry-run` | 非破壊（表示のみ）。fetch --pruneも実行しない | dry-runは常に非破壊最優先 |
| `--yes` | 確認なしで削除実行（非対話環境でも） | --yesは明示的な削除許可 |
| `--yes`なし + 対話環境 | 確認プロンプトを表示 | 従来動作 |
| `--yes`なし + 非対話環境 | リモートブランチ削除をスキップ | 確認不能のため安全デフォルト |

## 実装計画

### 1. 関数分割によるリファクタリング
既存のリモートブランチ削除ロジック（L198-243）を以下の関数に分割:

- `resolve_delete_mode()`: 実行モード判定（dry-run / yes / interactive / non-interactive）
- `check_remote_branch_exists()`: `git ls-remote --exit-code`による読み取り専用の存在確認
- `delete_remote_branch()`: 実際の削除実行

### 2. 非対話環境の検出
`[[ ! -t 0 ]]`で非対話環境を検出。`--yes`なし + 非対話の場合、リモートブランチ削除をスキップしメッセージ表示。

### 3. リモート存在確認の一本化
- 削除直前に`git ls-remote --exit-code`で存在確認（読み取り専用）
- `fetch --prune`は使用しない（dry-run互換性維持、ネットワーク障害伝播の防止）
- stale参照があっても`ls-remote`で実際の存在を確認するため問題なし

### 4. エラーハンドリング
- `ls-remote`失敗時（ネットワーク不通等）: warn表示してスキップ（exit 0維持）
- `git push --delete`失敗時: warn表示、`DELETE_FAILED=true`は維持するが最終的にexit 0（partial-failureをwarnに降格）
