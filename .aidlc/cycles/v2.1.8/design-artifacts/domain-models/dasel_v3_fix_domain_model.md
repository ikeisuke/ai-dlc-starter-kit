# ドメインモデル: dasel v3対応

## 概要

aidlc-setupのプロンプトファイル内でv2形式のみで記述されているdaselコマンド例を、v2/v3両対応の記述に更新する。

**重要**: このドメインモデル設計では**コードは書かず**、構造と責務の定義のみを行います。実装はImplementation Phase（コード生成ステップ）で行います。

## エンティティ（Entity）

### プロンプトファイル: 01-detect.md

- **責務**: aidlc-setup初期検出時のバージョン比較ロジックの記述
- **変更箇所**: 行101 - バージョン取得コマンド
- **現状**: `dasel -f .aidlc/config.toml 'starter_kit_version'`（v2形式のみ）
- **問題**: dasel v3では `dasel -f` サブコマンドなし形式が認識されず `Usage: dasel <command>` エラー

### プロンプトファイル: 02-generate-config.md

- **責務**: 欠落キー追記時のdaselコマンド例の記述
- **変更箇所**: 行417 - キー追記コマンド例
- **現状**: `dasel put -f .aidlc/config.toml -t <type> '<key>' -v '<value>'`（v2のみ）+ 「パイプ入力形式（dasel v3）」という曖昧な記述
- **問題**: v3の具体的なコマンド形式が示されていない

## 値オブジェクト（Value Object）

### daselバージョン互換コマンド形式（実機検証済み）

- **v2 読み取り**: `dasel -f <file> '<key>'`
- **v3 読み取り**: `cat <file> | dasel -i toml '<key>'`（v3では `-f` フラグが廃止、パイプ入力必須）
- **v2 書き込み**: `dasel put -f <file> -t <type> '<key>' -v '<value>'`
- **v3 書き込み**: v3の `put` はパイプ入力＋リダイレクト形式（AIが環境に応じて適切な形式を選択）

**キー形式**: ドット記法（`project.name`）をそのまま使用。ブラケット記法変換はスクリプト側（toml-reader.sh）の責務であり、プロンプト記述では不要。

## ドメインサービス

### 修正方針サービス

- **責務**: プロンプトファイルの記述をv2/v3両対応にする方針決定
- **方針1（01-detect.md）**: AIエージェントが実行するコマンドのため、v2形式とv3形式の両方を併記し、AIが試行で判定する旨を記述。`read-config.sh` 等の外部スクリプトへの新たな依存は追加しない（Unit境界: プロンプトファイルのみ変更）
- **方針2（02-generate-config.md）**: コマンド例にv2/v3両方の具体的な形式を明記し、AIエージェントが環境に応じて選択できるようにする

## ユビキタス言語

- **ブラケット記法**: dasel v3で導入されたキーアクセス形式（`t["v"]`）。v2のドット記法（`t.v`）と対比
- **フェイルセーフ**: daselコマンド失敗時にgrep等の代替手段で続行する既存の安全機構
