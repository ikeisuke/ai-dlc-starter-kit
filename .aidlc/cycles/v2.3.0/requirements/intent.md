# Intent（開発意図）

## プロジェクト名
AI-DLC Starter Kit v2.3.0 — コンテキスト圧縮 Tier 2/3（インデックス集約型）

## 開発の目的

**主目的**: #519 コンテキスト圧縮プロジェクトを完遂する。v2.2.3 までに Tier 1（session-state.md廃止、preflight.md圧縮、設定値条件ロードスキップ）が完了した成果の上に、Tier 2（中工数・中効果）と Tier 3 の一部（S4alt プログレッシブロード）を「インデックス集約型（案D）」で実装する。

**副次目的**: v2.2.3 で顕在化した #553「Inception後半フェーズ誤判定」バグを、案D の現在位置判定ロジック集約の一環として根本解決する。

### 案Dの設計方針

- **フェーズインデックスファイル**（常時ロード、小サイズ）: 全ステップの目次、分岐ロジック、現在位置判定チェックポイントを集約
- **各ステップ詳細ファイル**（必要時ロード）: 現在ステップの詳細手順のみをロード
- **復帰時フロー**: コンパクション後はインデックスのみ読み込み → 成果物ベース判定 → 該当ステップ詳細のみロード
- **分岐ロジック集約**: ステップ間の条件分岐をインデックスに一元化し、変更時の影響範囲を局在化
- **現在位置判定の単一責任点**: フェーズインデックスが現在位置判定の**唯一の正本**。`compaction.md` の判定テーブル機能は廃止し、インデックスに完全移管する（`compaction.md` 自体は `automation_mode` 復元等の他機能を残すため存続、ただし現在位置判定セクションは削除）
- **一意判定の前提条件**:
  1. 参照する成果物: 現サイクル配下の `requirements/`, `story-artifacts/`, `construction/`, `operations/` のディレクトリ構造と主要ファイルの存在
  2. 同点時の優先順位: `operations/` > `construction/` > `inception/`（後フェーズが先勝ち）。ただし各フェーズの `progress.md` の「完了」状態で明示的に補正
  3. 判定不能時の扱い: インデックス側で「判定不能」と表示してユーザーに確認を求め、`automation_mode` に関わらずユーザー選択を必須とする

## ターゲットユーザー

AI-DLC Starter Kit を使用する開発者（メタ開発者含む）。長時間セッションでコンパクションの影響を受けやすいユーザー、および大規模サイクルで多くのステップを実行するユーザー。

## ビジネス価値

- **コンテキスト効率の大幅向上**: Inception 初回ロードを現状 22,972 tok から ~15,000 tok（-35%）まで削減し、セッション中盤でのコンパクションリスクを低減する
- **復帰時ロードの最小化**: コンパクション後はインデックスのみで現在位置を把握できるため、復帰コストが大幅に削減される
- **保守性向上**: 分岐ロジックが一元化され、ステップ仕様変更時の影響範囲が局所化される
- **#553 の根本解決**: 対処療法ではなく、判定ロジック自体を集約することで同種の誤判定バグを構造的に排除する
- **#519 のクローズ**: 長期にわたる圧縮プロジェクトの区切りをつけ、以降は MCP 等の新手法に注力できる状態を作る

## 成功基準

計測条件: `/tmp/anthropic-venv/bin/python3` + `tiktoken cl100k_base` で同一条件比較。比較基準は v2.2.3 時点の実測値とする。

**必須基準**:
- Inception Phase 初回ロードが **15,000 tok 以下**（v2.2.3 実測 22,972 tok 比 -35% 以上）
- Construction Phase / Operations Phase 初回ロードが少なくとも現状維持（17,980 / 17,209 tok 以下）
- フェーズインデックスファイルが全フェーズで作成され、以下3点を集約している:
  1. 全ステップの目次・概要
  2. ステップ間分岐ロジック
  3. 現在位置判定チェックポイント
- コンパクション復帰時、インデックスファイル読み込みのみで現在ステップが一意に判定できる
- #553 の再現ケース（Inception後半でUnit定義ファイル生成後のコンパクション復帰）で Inception として正しく復帰できる

**動作保証基準**:
- 全フェーズ（Inception / Construction / Operations）のステップ実行が現行と同じ結果になる
- 既存の AI レビュー・セミオートゲート・コンパクション復帰・意思決定記録が機能する
- Tier 2 の施策は下記「Tier 2 施策の採否」に従い取り込む
- v2.3.0 新規サイクルで、コンパクション復帰時にインデックスのみで正しく現在位置判定できることを実測で確認する

### Tier 2 施策の採否

| 施策 | 採否 | 理由 |
|------|------|------|
| `operations-release.md` のスクリプト化 | **採用** | 詳細をスクリプト化する方向は案Dの「詳細ファイル化・必要時ロード」と同方向。インデックスからの参照先としてスクリプトを指定する形で統合できる |
| `review-flow.md` 判定ロジックの簡略化 | **採用** | 分岐ロジックをインデックス側に集約する案Dの主旨と完全一致。`review-flow.md` 内の条件分岐表はインデックスに移管、詳細手順だけを残す |
| ステップファイル内 boilerplate 削減 | **採用（自動解消扱い）** | 案D化の過程で「各ステップ共通の繰り返しパターン」は自然に削除される。独立施策としての実装は不要だが、達成状況を完了時に確認する |

**判断条件**: 案Dの構造（インデックス集約・詳細ファイル分離・復帰時最小ロード）を阻害せず、削減効果に寄与するものを採用する。この条件に合わない施策（例: 設計変更を伴わない単なるファイル分割）は今サイクル対象外とする。

## スコープ

### 含まれるもの

- **案D: インデックス集約型プログレッシブロードの実装**
  - Inception / Construction / Operations の3フェーズそれぞれにフェーズインデックスファイルを作成
  - 既存ステップファイルを「詳細手順」と「インデックス側に集約すべき情報（分岐・判定）」に分離
  - SKILL.md の共通初期化フローを「フェーズインデックスのみを読み込む」方式に変更
  - 復帰時フローの再設計（`steps/common/session-continuity.md` / `compaction.md` の更新）
- **Tier 2 施策のうち案Dと整合するもの**
  - `operations-release.md` のスクリプト化（インデックス集約の流れで詳細を外部化）
  - `review-flow.md` 判定ロジックの簡略化（インデックス側に分岐を寄せる）
  - ステップファイル内 boilerplate 削減（インデックス化で自然に解消される部分）
- **#553 の根本解決**
  - 現在位置判定ロジックをインデックスファイルに集約
  - Inception 後半（Unit 定義ファイル生成後）のコンパクション復帰が正しく Inception として扱われることを検証
  - `compaction.md` の現在位置判定テーブルは廃止し、他機能（`automation_mode` 復元等）のみ残す
- **計測と検証**
  - v2.2.3 ベースラインの再計測（Inception / Construction / Operations）
  - 実装後の計測と削減効果の確認
  - 削減目標 15,000 tok 達成の検証

### 含まれないもの

- **MCP / ツールベース検索の設計・実装** — 次サイクル以降に切り出し、#519 からは外出し
- **Tier 3 の他の選択肢（案A の極限圧縮、案C のフェーズインデックス方式単体）** — 案D に統合
- **既存ガイドファイル（`guides/**`）の大規模再構成** — 必要最小限の更新に留める
- **Reviewing スキル（`reviewing-*`）の内部実装変更** — 呼び出しインターフェース維持
- **`v2.3.0` と銘打つマイグレーション支援ツールの追加** — 設定・構造変更は `/aidlc-migrate` の既存機構内で扱える範囲に留める
- **v2.2.x 以前で開始した進行中サイクルの継続利用保証** — 後述「既存サイクル互換性方針」を参照

### 既存サイクル互換性方針

- **v2.2.x 以前に完了したサイクルの成果物**（`.aidlc/cycles/v1.*/`, `.aidlc/cycles/v2.0.*/`, `.aidlc/cycles/v2.1.*/`, `.aidlc/cycles/v2.2.*/`）は **読み取り専用の履歴データとして維持**する。ディレクトリ構造・ファイルフォーマットは変更しない
- **v2.2.x 以前で開始した進行中サイクル**の継続利用は **保証しない**。v2.3.0 の新フロー（フェーズインデックス経由）は新規サイクル（`v2.3.0` 以降）を前提とする
- **スターターキット利用者への影響**:
  - 新規インストール・新規サイクル開始: 新フローが適用される
  - 既存インストールのアップグレード: `/aidlc migrate` 実行で `.aidlc/config.toml` の `starter_kit_version` を更新。移行時点で進行中のサイクルがある場合は「そのサイクルは旧フローで完了し、次サイクルから新フローに移行」を推奨する
- **メタ開発プロジェクト（本プロジェクト自身）**: 現在進行中のサイクルは v2.3.0（本サイクル）のみ。v2.2.3 までは完了・リリース済みのため、移行は自然に行われる

## 期限とマイルストーン

- Inception Phase: 本セッション中に完了
- Construction Phase: 複数 Unit を段階的に実装
  - Unit A: フェーズインデックスの設計と試作（1フェーズで先行実装）
  - Unit B: 残り2フェーズへ展開
  - Unit C: Tier 2 施策の取り込み
  - Unit D: #553 統合検証
  - Unit E: 計測・目標達成確認
- Operations Phase: v2.3.0 リリース

## 制約事項

- **不変ルールの厳守**: SKILL.md の「ステップファイルの読み込みは省略不可」に抵触しない設計が必要。インデックスファイルを「フェーズステップ読み込み」の対象として定義する形で整合させる
- **既存サイクルの成果物との互換性**: `.aidlc/cycles/v*/` 配下の既存成果物フォーマットは維持する
- **外部プロジェクトへの影響**: スターターキット利用プロジェクトに破壊的変更を強制しない（設定ファイル互換性を維持）
- **品質劣化リスクの回避**: #519 body の「触るべきでないセクション」リスト（compaction 復元、review-flow 判断、preflight blocker、rules-automation フォールバック、rules-core スコープ保護）は慎重に扱う
- **計測ツール**: tiktoken (cl100k_base) 参考値。Claude トークナイザーとは±10-20%の差異がある前提で判断する
