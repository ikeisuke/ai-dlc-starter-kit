# PRFAQ: AI-DLC Starter Kit v2.3.0

## Press Release（プレスリリース）

**見出し**: AI-DLC Starter Kit v2.3.0 — インデックス集約型プログレッシブロードで初回コンテキストを35%削減

**副見出し**: フェーズインデックスに目次・分岐・復帰判定を一元集約。コンパクション後もインデックスだけで即座に復帰できる #519 完遂リリース

**発表日**: 2026-04-10（想定）

**本文**:

v2.2.0 から続くコンテキスト圧縮プロジェクト（#519）を完遂するリリースです。Tier 2（`operations-release.md` スクリプト化、`review-flow.md` 判定ロジック簡略化、ステップファイル内 boilerplate 削減）と Tier 3 の一部（S4alt プログレッシブロード）を、ユーザー提案を取り入れた「インデックス集約型（案D）」で実装します。

案D の核心は、各フェーズに新設する「フェーズインデックスファイル」です。全ステップの目次・ステップ間分岐ロジック・現在位置判定チェックポイントを1箇所に集約し、詳細手順は必要時にのみロードします。これにより、Inception 初回ロードを 22,972 tok から 15,000 tok 以下（-35%）まで削減し、セッション中盤のコンパクションリスクを大幅に低減します。

あわせて、v2.2.3 で顕在化した #553「Inception 後半でのコンパクション復帰誤判定バグ」を、案D の現在位置判定ロジック集約の一環として根本解決します。インデックスが判定の「唯一の正本」となるため、`compaction.md` との二重管理問題が解消され、同種の誤判定バグが構造的に排除されます。

## FAQ（よくある質問）

### Q1: 案D とは何ですか？なぜ既存の案 A/B/C ではなく案D を採用したのですか？

A: 案D は「インデックス集約型プログレッシブロード」の名称で、#519 body に記載されていた案A（ステップファイル極限圧縮）・案B（depth_level ベース省略）・案C（フェーズインデックス方式）の3案に、ユーザー提案の「分岐ロジックもインデックスに集約して復帰時も最小限で済むようにする」という観点を加えて発展させたものです。案A-C は単独では「累積ロードの問題」「削減量の小ささ」「復帰最適化の欠如」のいずれかの課題が残りましたが、案D は分岐集約と復帰最小化により、長時間セッションでの実用性を大幅に高めます。

### Q2: コンパクション復帰時の動作はどう変わりますか？

A: v2.3.0 以降、コンパクション復帰時はフェーズインデックスファイル1個のみを読み込めば、現在位置と次のアクションが一意に判定できます。従来 `compaction.md` に散らばっていた判定ロジックはインデックスに集約され、`compaction.md` 自体は `automation_mode` 復元等の他機能のみが残ります。これにより復帰時のロードコストが大幅に削減されます。

### Q3: #553 バグはどう修正されますか？

A: 対処療法ではなく根本解決します。案D で「現在位置判定の単一責任点はフェーズインデックス」と定義したため、`inception/progress.md` の完了状態を考慮した判定が自動的に行われ、Unit 定義ファイル生成後のコンパクション復帰でも Inception として正しく判定されます。再現シナリオ（PRFAQ 未着手ケース・完了処理中ケース）での動作が実測で検証されます。

### Q4: 既存の v2.2.x 以前で開始したサイクルは継続利用できますか？

A: v2.2.x 以前に完了したサイクルの成果物は読み取り専用の履歴データとしてそのまま維持されます。v2.2.x 以前で開始した進行中サイクルの継続利用は保証されません（ディレクトリ構造自体は変わりませんが、復帰フローが新仕様になります）。進行中サイクルがある場合は、そのサイクルを旧フローで完了させてから次サイクルで新フローに移行することを推奨します。

### Q5: 不変ルール「ステップファイル読み込みは省略不可」との整合性は？

A: フェーズインデックスファイルを「フェーズステップ読み込み」の対象として再定義することで整合を取ります。つまり、初回ロードではインデックスのみを読み込み、詳細ステップファイルはインデックスから必要時に参照する形となります。ステップファイル自体の「読み込みを省略する」わけではなく、「いつ読み込むか」を最適化する設計です。

### Q6: Tier 2 施策（operations-release スクリプト化、review-flow 簡略化）はどう実装されますか？

A: `operations-release.md` は `scripts/operations-release.sh` に移管され、Operations フェーズインデックスからスクリプトを呼び出す形になります。`review-flow.md` の条件分岐表は `steps/common/review-routing.md` に移管され、各フェーズインデックスから参照されます。どちらも動作等価性検証（整理前後で出力・引数が一致すること）が受け入れ基準に含まれます。

### Q7: 今回含まれない施策は何ですか？

A: Tier 3 の「MCP / ツールベース検索」は、設計未確定の領域であるため次サイクル以降に切り出します。v2.3.0 は S4alt（案D）までを対象とし、MCP 検討は独立した改善機会として別 issue に分離します。
