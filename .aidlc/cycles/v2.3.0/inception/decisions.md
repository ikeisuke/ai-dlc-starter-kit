# 意思決定記録 - v2.3.0

## DR-001: #519 サイクルスコープの決定

- **ステップ**: Intent明確化（質問1）
- **日時**: 2026-04-09

### 背景

v2.2.3 で #519「AI-DLCスキルのコンテキストサイズ圧縮」の Tier 1（session-state.md廃止、preflight.md圧縮、設定値条件ロードスキップ）が完了。#519 body には Tier 2（中工数・中効果）と Tier 3（高工数・高効果、S4alt プログレッシブロード、MCP/ツールベース検索）が残っており、今サイクルで何処まで扱うかを決める必要があった。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | Tier 2 に着手（Tier 3 は別サイクル） | スコープが明確。中規模サイクルで収まる | #519 のクローズが先延ばしになる |
| 2 | Tier 2 + Tier 3 完遂 | #519 を本サイクルで完遂できる。長期プロジェクトの区切りになる | スコープが広くリスクが高い |
| 3 | Tier 1 完了で #519 をクローズ、Tier 2/3 を新 issue に分割 | 今サイクルは小規模で済む | #519 の当初計画（~10,000 tok 到達）に対する未達感が残る |

### 決定

**選択肢2: Tier 2 + Tier 3 完遂** を採用（ただし Tier 3 の MCP/ツールベース検索は DR-002 で除外決定）

### トレードオフと判断根拠

- **得たもの**: #519 の区切りを明確化でき、以降は MCP 等の新手法に注力できる状態を作れる
- **犠牲にしたもの**: サイクル規模が大きくなり、実装工数と検証負荷が増える
- **判断根拠**: #519 は v2.2.0 から続く長期プロジェクトで、段階的に完了してきた経緯がある。ここで一気にクローズまで持っていく方が、プロジェクト管理上の効率が良い。Tier 3 全体ではなく実行可能な範囲（S4alt）に絞ることで、スコープ膨張を防止する（DR-002 で明確化）

## DR-002: Tier 3 MCP/ツールベース検索の除外

- **ステップ**: Intent明確化（質問1詳細化）
- **日時**: 2026-04-09

### 背景

DR-001 で「Tier 2 + Tier 3 完遂」を決定したが、Tier 3 には S4alt プログレッシブロードと MCP/ツールベース検索の2つが含まれる。MCP は #519 body 上「大幅削減」としか記載されておらず、具体的な設計が存在しない。工数が膨らむリスクがあるため、今サイクルに含めるかを再判断した。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | MCP は含めない（今サイクルは S4alt まで） | スコープが実行可能な範囲に収まる。リスク低減 | MCP の検討が先延ばしになる |
| 2 | MCP の設計案検討まで含める | 次サイクル以降の準備が進む | ドキュメント作業に工数が取られる |
| 3 | MCP の実装まで含める | #519 を完全にクローズできる | 工数大、設計未確定でリスク高 |

### 決定

**選択肢1: MCP は含めない** を採用

### トレードオフと判断根拠

- **得たもの**: 今サイクルは S4alt のみに集中でき、確実に成果を出せる
- **犠牲にしたもの**: MCP の検討開始が次サイクル以降に遅れる
- **判断根拠**: MCP は設計未確定の新領域で、今サイクルで扱うにはリスクが大きすぎる。S4alt の完遂だけでも #519 の目標値（~15,000 tok）に到達可能な見込みがあり、MCP は独立した改善機会として次サイクル以降に切り出すのが合理的

## DR-005: S4alt プログレッシブロードの実装方針

- **ステップ**: Intent明確化（質問2およびユーザー提案を踏まえた案Dの提示）
- **日時**: 2026-04-09

### 背景

#519 body には S4alt の3案（案A: ステップファイル極限圧縮、案B: depth_level ベース省略、案C: フェーズインデックス方式）が記載されていた。AI 側の初回提案に対してユーザーから「案C のインデックス方式なら、分岐もインデックス側に持てば途中で落ちた場合でも最小限で復活できるのでは？」という指摘があり、案C を発展させて「案D: インデックス集約型 + 分岐ロジック集約 + 復帰時最小ロード」を新案として提示。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 案A: ステップファイル極限圧縮 | 既存構造を維持しつつ削減 | 実行中にガイドを都度ロードすると後半で累積する |
| 2 | 案B: depth_level ベース省略 | 実装が軽い | minimal 時限定、削減量が小さい |
| 3 | 案C: フェーズインデックス方式 | 構造整理ができる | 分岐・復帰の最適化が不十分 |
| 4 | 案D: インデックス集約型（案C拡張） | 分岐集約で保守性向上。復帰時最小ロードで #553 根本解決と統合可能 | 設計難易度が高く、既存の各ステップファイルの大幅リファクタが必要 |
| 5 | 案D 限定版（分岐集約のみ） | スコープ縮小 | インデックスなしでは効果が限定的 |

### 決定

**選択肢4: 案D（インデックス集約型）** を採用

### トレードオフと判断根拠

- **得たもの**: 累積しない復帰最適化、分岐の一元化による保守性向上、#553 の根本解決との統合（判定ロジックがインデックスに集約される）
- **犠牲にしたもの**: 設計難易度の高さと実装工数の増加
- **判断根拠**: ユーザー提案の「復帰時も最小限で済む」という観点は、長時間セッションのコンパクション問題の根本解決につながる。案A/B/C 単独ではこの観点が充たされないため、案D の統合的アプローチが最適

## DR-006: Inception 初回ロード削減目標

- **ステップ**: Intent明確化（質問3）
- **日時**: 2026-04-09

### 背景

現状計測で v2.2.3 の Inception 初回ロードが 22,972 tok と判明。目標値の設定が必要だった。#519 body の推定値は Tier1+2 完了時 ~16,000 tok、Tier 3 完了時 ~10,000 tok。

| # | 選択肢 | メリット | デメリット |

## DR-008: Tier 2 施策の必達度（Must-have 扱い）

- **日時**: 2026-04-09

- **判断根拠**: Intent の主目的（#519 Tier 2/3 完遂）を曲げずに貫く方が、プロジェクト管理上一貫性がある。Tier 2 施策は中規模で見積もり可能な範囲であり、完遂は現実的

## DR-011: Unit 分割方針（6 Unit 構成）

- **ステップ**: Unit 定義作成
- **日時**: 2026-04-09

### 背景

9 ストーリーを Unit に分解する際、粒度の決定が必要だった。1 ストーリー = 1 Unit（9 Unit）、責務グルーピング（6 Unit）、機能統合型（3-4 Unit）の選択肢があった。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 9 Unit（1 ストーリー = 1 Unit） | 責務が細かく管理可能 | Unit 数が多く管理コスト増。完了処理の回数が増える |
| 2 | 6 Unit（責務グルーピング） | 粒度がバランス型。ストーリー4+5、ストーリー6+7 などを統合 | - |
| 3 | 3-4 Unit（機能統合型） | Unit 数を抑えられる | 1 Unit が大規模化しすぎる。段階的検証が困難 |

### 決定

**選択肢2: 6 Unit（責務グルーピング）** を採用

### トレードオフと判断根拠

- **得たもの**: Unit 粒度が「基盤構築（001）→ 共通判定（002）→ 横展開（003, 004）→ Tier 2 統合（005）→ 計測・クローズ（006）」と段階的になり、各 Unit が 1-2 日規模で実装・検証可能
- **犠牲にしたもの**: 1 ストーリー = 1 Unit の細粒度管理
- **判断根拠**: ストーリー4とストーリー5は「基盤」と「受け入れ確認」の関係にあり分離すると重複作業。ストーリー6とストーリー7は Tier 2 施策としてひとまとまり。計測とクローズ判断は連続作業。これらをまとめることで管理コストと実装負荷のバランスが取れる
