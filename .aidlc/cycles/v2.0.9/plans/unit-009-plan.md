# Unit 009: セミオートゲート判定修正・レビュー結果保存フォーマット更新 - 計画

## 概要

Unit 008で発生した問題: AIレビューで3件の指摘が検出されたが、自己判断でOUT_OF_SCOPEとしてセミオートゲートで自動承認してしまった。仕様上は指摘が存在する時点でフォールバックすべき。

## 根本原因分析

### 仕様のギャップ

1. **`review_issues` の判定根拠が自然言語で曖昧**: 「AIレビュー指摘が残っている」が機械判定不能な記述
2. **レビュー完了状態の構造化不足**: `指摘が検出されたか` / `未解決指摘が残っているか` / `ユーザー承認が必要か` が同じ概念として混在
3. **遷移の競合**: review-flow.md に「全て先送り → ユーザーレビューフローへ」と「共通処理でセミオートゲート判定」が併存し、優先制御が不明確
4. **バックログ参照が自由記述**: OUT_OF_SCOPE判断のIssue番号がフォーマット揺れしやすい

## 変更対象ファイル

| ファイル | 変更種別 |
|---------|---------|
| `skills/aidlc/steps/common/rules.md` | レビュー結果シグナルの構造化・`review_issues`判定基準の機械可読化 |
| `skills/aidlc/steps/common/review-flow.md` | レビュー完了時のシグナル生成・遷移の一本化 |
| `skills/aidlc/templates/review_summary_template.md` | バックログ列の分離 |

## 実装計画

### 1. `steps/common/rules.md` の修正（判定ルールの正本）

#### 1a. レビュー結果シグナルの導入

フォールバック条件テーブルの近くに、レビュー完了時にreview-flow.mdが生成するシグナルの定義を追加:

```text
### レビュー結果シグナル（review-flow.md が生成）

| シグナル名 | 型 | 説明 |
|-----------|-----|------|
| review_detected | boolean | 当該承認ポイントの全レビュー種別を通じて1件以上の指摘が検出されたか |
| deferred_count | integer | OUT_OF_SCOPE + TECHNICAL_BLOCKER の件数（全種別合算） |
| resolved_count | integer | 修正済みの件数（全種別合算） |
| unresolved_count | integer | 未対応の件数（全種別合算） |
```

**スコープとライフサイクル**:
- **生成タイミング**: レビューステップ完了時の共通処理（ステップ0）で、当該承認ポイントの全レビュー種別の結果を集約して生成
- **有効範囲**: 同一承認ポイント内でのみ有効。ゲート判定後に破棄（次の承認ポイントに持ち越さない）
- **初期化**: 各承認ポイントのレビュー開始前に未設定状態。レビュー未実施の場合は未設定のままとし、`review_not_executed` フォールバックで処理
- **集計単位**: 当該承認ポイントで実施した全レビュー種別（code, architecture, security等）の集約結果。1種別でも指摘があれば `review_detected=true`

#### 1b. `review_issues` 判定基準の機械可読化

フォールバック条件テーブルを更新:
- 現在: `review_issues` / 「AIレビュー指摘が残っている」
- 修正: `review_issues` / 「`review_detected=true`（指摘対応判断の結果に関わらず、指摘が検出された時点でフォールバック）」

`auto_approved` の条件を明記:
- 「`review_detected=false` の場合のみ `auto_approved` を許可」

### 2. `steps/common/review-flow.md` の修正（シグナル生成と遷移）

#### 2a. レビュー完了時のシグナル生成

「レビューステップ完了時の共通処理」セクションの冒頭に、シグナル生成ステップを追加:

```text
0. **レビュー結果シグナル生成**: 当該承認ポイントの全レビュー種別の結果を集約し、以下のシグナルをコンテキスト変数として設定する（rules.md「レビュー結果シグナル」定義に準拠）
   - `review_detected`: 全種別を通じて1件以上の指摘が検出された場合 `true`、0件なら `false`
   - `deferred_count`: OUT_OF_SCOPE + TECHNICAL_BLOCKER の件数（全種別合算）
   - `resolved_count`: 修正済みの件数（全種別合算）
   - `unresolved_count`: 未対応の件数（全種別合算）
   ※ シグナルは同一承認ポイント内でのみ有効。ゲート判定後に破棄する
```

#### 2b. 遷移の一本化

「全て先送り → ユーザーレビューフローへ」の遷移と「共通処理でセミオートゲート判定」の競合を解消:
- 指摘対応判断フローを経由した場合: シグナル `review_detected=true` が設定済みなので、共通処理のセミオートゲート判定で自動的に `fallback(review_issues)` に遷移
- 「全て先送り → ユーザーレビューフローへ」の直接遷移を削除し、共通処理経由に一本化
- 共通処理のセミオートゲート判定が正本（rules.md参照）で制御する

#### 2c. review-flow.md の注記（二重管理防止）

`review-flow.md` のセミオートゲート判定部分には判定ルールを記述せず、`rules.md` への参照のみとする（既存の「`common/rules.md` のセミオートゲート仕様参照」を維持）。

### 3. レビューサマリフォーマットの更新

#### 3a. テンプレートにバックログ列を追加

指摘一覧テーブルに `バックログ` 列を追加:

```markdown
| # | 重要度 | 内容 | 対応 | バックログ |
|---|--------|------|------|-----------|
| 1 | 高 | ... | 修正済み（...） | - |
| 2 | 中 | ... | OUT_OF_SCOPE（理由: ...） | #NNN |
| 3 | 中 | ... | TECHNICAL_BLOCKER（理由: ...） | - |
```

列ルール:
- `OUT_OF_SCOPE` 時: バックログ列に Issue番号を記載（必須）
- `TECHNICAL_BLOCKER` 時: バックログ列は `-`（空可）
- `修正済み` 時: バックログ列は `-`

#### 3b. review-flow.md の列の記述ガイダンス更新

「列の記述ガイダンス」セクションに `バックログ` 列のルールを追加。
