# フェーズ実行方式の設計検討: スキル化 vs サブエージェント化

## 概要

AI-DLCフェーズ実行において、共通処理の抽出方式として「スキル化（Skill tool）」と「サブエージェント化（Task tool）」の2つのアプローチを比較し、推奨方式を策定する。

## 方式定義

### 方式A: スキル化（Skill tool）

**概要**: 共通処理をSKILL.mdファイルとして定義し、Skill toolで呼び出す。v1.16.1で設計されたアプローチ。

**特性**:
- SKILL.md（フロントマター + Markdown指示文）として定義
- Skill tool経由で呼び出し、スキル内容がコンテキストに注入される
- 呼び出し元の会話コンテキストを**共有する**（同一セッション内で実行）
- allowed-toolsでツールアクセスを制限可能
- ユーザーとの対話が可能（AskUserQuestion等）

**制約**:
- スキル内容がコンテキストウィンドウを消費する
- スキル間の直接呼び出し不可（v1.16.1設計ルール）
- ステートレス（セッション間で状態を持たない）

### 方式B: サブエージェント化（Task tool）

**概要**: 共通処理をサブエージェントとして起動し、独立プロセスで実行する。

**特性**:
- Task toolでサブプロセスとして起動
- 呼び出し元のコンテキストを**共有しない**（独立コンテキスト）
- タスク説明に必要な情報をすべて含める必要がある
- バックグラウンド実行が可能
- 結果は単一メッセージで返却

**制約**:
- ユーザーとの直接対話不可（AskUserQuestionが使えない）
- 呼び出し元の進捗状況やUnit定義へのアクセスが限定的
- マルチファイル編集（依存関係あり）の委任は非推奨
- コンテキスト全体を渡す必要があり、プロンプト設計が複雑

## 比較分析

### 比較軸と評価

| 比較軸 | スキル化（A） | サブエージェント化（B） |
|--------|-------------|---------------------|
| **コンテキスト共有** | 共有（同一セッション） | 非共有（独立プロセス） |
| **コンテキスト消費** | スキル内容分を消費 | 消費しない（別ウィンドウ） |
| **ユーザー対話** | 可能 | 不可 |
| **ツールアクセス** | allowed-toolsで制限可能 | 全ツール利用可能（※プラットフォーム制約: Task toolにはallowed-tools相当の制限機構がない） |
| **エラーハンドリング** | 呼び出し元で判断可能 | 結果メッセージのみ |
| **並列実行** | 不可（逐次実行） | 可能（バックグラウンド） |
| **実装コスト** | 低（Markdown定義） | 中（プロンプト設計が複雑） |
| **デバッグ容易性** | 高（同一セッション） | 低（別プロセス） |
| **環境互換性** | Claude Code / KiroCLI 対応 | Claude Code のみ |
| **既存パターン整合** | 6スキル実績あり | 限定的（レビュースキルの外部CLI呼び出し程度） |

### 詳細分析

#### コンテキスト管理

**スキル化**: スキル内容（SKILL.md + references/）がコンテキストウィンドウに注入される。v1.16.1の推定では、3つのHigh優先度スキルで合計~880行追加。大規模スキルはコンテキスト圧迫のリスクがある。ただし、必要な時にのみ読み込まれるため、フェーズプロンプト全体を常時保持するよりは効率的。

**サブエージェント**: 独立コンテキストのため、メインエージェントのウィンドウを消費しない。ただし、タスク説明に十分なコンテキスト（Unit定義、設定値、現在の進捗等）を含める必要があり、プロンプト設計が重要。

#### ユーザー対話

**スキル化**: AskUserQuestion等の対話ツールが利用可能。バックログ優先度の確認、設計承認、レビュー判断など、ユーザー判断が必要な処理に適する。

**サブエージェント**: ユーザー対話不可。完全自動化可能なタスクにのみ適する。判断が必要な場合は事前にメインエージェントが収集して渡す必要がある。

#### 環境互換性

**スキル化**: Claude Code（Skill tool）とKiroCLI（.kiro/skills/ネイティブ発見）の両方で動作。Codex CLIやGemini CLIでも手動読み込みで利用可能。

**サブエージェント**: Claude CodeのTask toolに依存。KiroCLIには同等機能がない（2026年2月現在）。環境依存性が高い。

#### 判定の決定要因

ユーザー対話の必要性と環境互換性が最も重要な決定要因。AI-DLCは複数のAIツール（Claude Code, KiroCLI, Codex CLI, Gemini CLI）での利用を前提としており、**ツール非依存性**が設計原則の一つ。

## 10スキル候補の適性評価

### v1.16.1で特定された10候補

| # | スキル名 | 優先度 | ユーザー対話 | 自動化度 | 推奨方式 | 理由 |
|---|---------|--------|------------|---------|---------|------|
| 1 | issue-management | High | あり | 中 | **スキル** | Issue操作はユーザー確認が必要な場面が多い |
| 2 | backlog-management | High | あり | 中 | **スキル** | 優先度選択・保存先確認にユーザー対話必須 |
| 3 | pr-operations | High | あり | 低 | **スキル** | PR作成・マージはユーザー承認が必要 |
| 4 | version-management | Medium | なし | 高 | **スキル** | 自動化可能だが、KiroCLI互換のためスキル推奨 |
| 5 | setup-initialization | Medium | あり | 低 | **スキル** | セットアップは対話的プロセス |
| 6 | progress-tracking | Medium | なし | 高 | **スキル** | 小規模（~75行）、独立実行のメリット薄い |
| 7 | dialogue-planning | Low | あり | 低 | **現状維持** | 抽象化リスクが利益を上回る（v1.16.1評価） |
| 8 | completion-validation | Low | あり | 中 | **スキル** | 完了条件チェックはフェーズ固有ロジックが多い |
| 9 | code-quality-check | Low | なし | 高 | **サブエージェント候補** | lint実行は完全自動化可能、並列実行メリットあり |
| 10 | unit-squash | Low | なし | 高 | **現状維持** | 既にスクリプト化済み（~12行、抽出不要） |

### 適性判定の基準

1. **ユーザー対話が必要** → スキル化（サブエージェントでは対話不可）
2. **完全自動化可能 + 並列実行メリット** → サブエージェント候補
3. **小規模 + 抽象化リスク > 利益** → 現状維持
4. **環境互換性が重要** → スキル化（KiroCLI対応）

## 推奨方式

### 結論: スキル化を基本方式とし、限定的にサブエージェントを併用するハイブリッド方式

#### 理由

1. **環境互換性**: AI-DLCはツール非依存を設計原則としており、KiroCLI対応が必要。スキル化はすべての対象環境で動作する。

2. **ユーザー対話**: 10候補中7候補でユーザー対話が必要または望ましい。サブエージェントではこれらの対話要件を満たせない。

3. **既存パターン整合**: 6スキルの実績があり、開発者にとって馴染みのあるパターン。v1.16.1の設計をそのまま活用できる。

4. **コンテキスト消費の懸念**: スキル内容がコンテキストを消費する問題は、必要時のみ読み込む設計と、SKILL.mdの簡潔化（references/への詳細分離）で軽減可能。

#### ハイブリッド方式の詳細

| 分類 | 方式 | 対象 | 説明 |
|------|------|------|------|
| Phase A/B 確定 | **スキル化** | 候補#1-6 | ユーザー対話が必要、または環境互換性が重要な処理（High/Medium優先度） |
| Phase C 候補 | **スキル化** | 候補#8（completion-validation） | スキル化方向だが、Phase A/B結果を踏まえ最終判断（Low優先度） |
| Phase C 候補 | **サブエージェント** | 候補#9（code-quality-check） | 完全自動化可能で並列実行メリットがある処理（Low優先度） |
| 現状維持 | - | 候補#7, #10 | 抽象化リスクが利益を上回る、または既にスクリプト化済み |

### サブエージェント適用条件（ガードレール）

サブエージェントを適用する場合の必須条件:

1. ユーザー対話が一切不要であること
2. 単一ファイルまたは完全独立ファイルの操作であること
3. 完了条件が明確に定義可能であること
4. メインエージェントのコンテキストなしで実行可能であること
5. 失敗時のリカバリが容易であること（再実行で解決可能）

#### 並列実行時のタスク分類と排他制御

サブエージェントを並列実行する場合、以下のタスク分類に基づいて排他制御を行う:

| タスク分類 | 定義 | 並列実行 | 例 |
|-----------|------|---------|-----|
| Read-only | ファイル読み取りのみ | 可能（制限なし） | lint実行、静的解析 |
| Single-writer | 特定ファイルへの書き込み | 排他（同一パス不可） | ファイル生成 |

**排他制御ルール**:
- 同一ファイルパスへの書き込みを行うタスクは並列実行禁止（逐次実行にフォールバック）
- Read-onlyタスク同士は無制限に並列実行可能
- 現時点のサブエージェント候補（#9: code-quality-check）はRead-onlyタスクのため、排他制御は不要

### 実行契約（Execution Contract）

スキル化・サブエージェント化のいずれの方式でも、フェーズプロンプト（呼び出し元）と実行方式の間で以下の共通契約を遵守する。

#### 共通契約

| 項目 | 定義 |
|------|------|
| **入力** | スキル: argument-hintフィールド（フロントマター定義）に基づく実行時引数。サブエージェント: promptパラメータ（タスク説明に必要な情報をすべて含む） |
| **出力** | スキル: 呼び出し元コンテキストへの直接書き込み（ファイル作成・編集等）。サブエージェント: 単一メッセージ（結果サマリ） |
| **ステータス** | 成功（処理完了） / 失敗（エラー内容付き） / 中断（ユーザー判断による中止、スキルのみ） |
| **エラー時** | スキル: 呼び出し元で判断・リカバリ。サブエージェント: 再実行で解決（べき等性を前提） |
| **互換性** | v1.16.1のSKILL.mdフロントマター構造（name, description, argument-hint, allowed-tools, compatibility）に準拠 |

**注**: 詳細なI/Oスキーマ（フィールド定義、型、バリデーション等）は、各スキルの実装フェーズ（Phase A/B/C）で個別に定義する。v1.16.1のドメインモデル（`docs/cycles/v1.16.1/design-artifacts/domain-models/skill-design_domain_model.md`）を基礎とする。

## 段階的移行プラン

### Phase A（次サイクル: v1.17.x候補）

**対象**: High優先度スキル3件
- issue-management（~106行削減）
- backlog-management（~137行削減）
- pr-operations（~256行削減）

**推定効果**: ~499行削減（3フェーズプロンプト合計）

**前提**: v1.16.1の論理設計に基づくSKILL.md + references/ 構造

### Phase B（v1.18.x候補）

**対象**: Medium優先度スキル3件
- version-management（~143行削減）
- setup-initialization（~225行削減）
- progress-tracking（~75行削減）

**推定効果**: ~443行削減

### Phase C（v1.19.x候補）

**対象**: Low優先度スキルの再評価
- completion-validation: スキル化を検討
- code-quality-check: サブエージェント化を検討
- dialogue-planning, unit-squash: 再評価（Phase A/Bの結果を踏まえて判断）

### 各Phase共通の実装手順

1. SKILL.md + references/ ファイル作成
2. フェーズプロンプトからの該当セクション置換
3. スキルシンボリンク作成（.claude/skills/）
4. 統合テスト（全フェーズでの動作確認）
5. 行数削減の計測・検証

## リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| スキル内容によるコンテキスト圧迫 | Medium | SKILL.md簡潔化、references/への詳細分離、必要時のみ読み込み |
| KiroCLIでのスキル互換性問題 | Medium | Phase A実装時にKiroCLI環境での動作検証を実施 |
| スキル粒度の不適切さ | Low | v1.16.1の凝集性基準（3点テスト）を遵守 |
| 移行中の後方互換性 | Low | v1.16.1設計のdeprecated field + 1サイクル移行期間を適用 |
