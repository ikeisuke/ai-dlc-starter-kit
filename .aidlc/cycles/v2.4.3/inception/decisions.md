## DR-001: サイクルバージョンを v2.4.3 (patch) に確定

- **ステップ**: 01-setup ステップ8（バージョン提案）
- **日時**: 2026-04-27

### 背景

直近 v2.4.0 (minor) / v2.4.1 (patch) / v2.4.2 (patch) と Issue 集約解消サイクルが続いており、オープン Issue にも patch 親和性の高い bug + refactor が複数存在する。次に minor 機能追加（振り返りステップ追加など）に進むか、patch を継続するかの判断。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | v2.4.3 (patch) | bug + 小規模 refactor を素早く解消、ダウンストリームへの影響小 | 機能追加が遅れる |
| 2 | v2.5.0 (minor) | 振り返りステップ等の機能追加を含められる | スコープ拡大、レビュー負荷増 |
| 3 | v3.0.0 (major) | 破壊的変更を含められる | リリースリスク大、現状で破壊的変更要件は薄い |

### 決定

**v2.4.3 (patch)** を採用

### トレードオフと判断根拠

- **得たもの**: 4 件の patch 親和性 Issue（#612 / #611 / #610 / #609）を 1 サイクルで集約解消。日常運用の摩擦を即時除去
- **犠牲にしたもの**: 機能追加（#590 等）は次 minor サイクルへ持ち越し
- **判断根拠**: 直近 patch リリース継続中で、ダウンストリーム導入者・メタ開発者双方が踏むバグ・紛らわしい運用が複数残っていたため、機能追加より先に整地する価値が高い

## DR-003: #612 解決方針 — 文言整合 + 役割対比明示（案 B 系）

- **ステップ**: 03-intent ステップ1（Intent 明確化）
- **日時**: 2026-04-27

### 背景

Issue #612 では命名衝突解消のため A/B/C 3 案が提示。A=スターターキット自身で upgrade フロー抑止 / B=ブランチ命名変更 + ドキュメント明示 / C=ドキュメントのみ更新。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | A（フロー抑止） | dogfooding 中の混乱を完全排除 | 実装側の判定ロジックが必要、保守コスト |
| 2 | B（命名変更 + ドキュメント明示） | 両用途の役割が一目で分かる、ダウンストリームにも明示的 | 命名変更が必要（ただし実装は v2.4.2 で完了済みと判明） |
| 3 | C（ドキュメントのみ） | 最小修正 | 命名衝突が残り、再発リスクあり |
| 4 | A + B 両方 | 二重保険 | スコープ過大 |

### 決定

**B（命名変更 + ドキュメント明示）** を採用。実態確認後「実装は v2.4.2 で `chore/aidlc-v<version>-upgrade` に統一済み」と判明したため、本サイクルでは `.aidlc/rules.md` の旧表記更新と SKILL.md 系の役割対比追加に最小化（命名のリネーム作業は不要）。

### トレードオフと判断根拠

- **得たもの**: ダウンストリーム / スターターキット自身の役割対比が文言で明確化、新規利用者が迷わない
- **犠牲にしたもの**: A 案の自動抑止ロジックは導入せず（Construction で検討するなら別 Issue 化）
- **判断根拠**: 実装現状を Inception 中に確認したところ、B 案の核心（命名分離）は既に達成済みで、残る差分は `.aidlc/rules.md` L274 の旧表記のみ。文言整合 + 対比節追加で十分目的を達成できる

## DR-004: #611 既存設定の後方互換性 — 暗黙シムを今サイクルで追加

- **ステップ**: 03-intent ステップ1（Intent 明確化）
- **日時**: 2026-04-27

### 背景

`[rules.reviewing].tools` に `"self"` を正式統合する際、既存設定（`tools = ["codex"]` / `tools = []` / 未設定）をどう扱うか。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 暗黙シム（末尾 self 補完）を今サイクルで追加 | 既存設定を壊さず透過的に動作、ダウンストリーム影響なし | コードに小規模な後方互換層が増える |
| 2 | 後方互換なし + aidlc-migrate で一括書き換え | 設定が明示的になる | ダウンストリーム利用者にソース変更負担、migrate の信頼性が必要 |
| 3 | 後方互換を永久維持 | 互換性最大 | コードの暗黙挙動が永続化 |

### 決定

**選択肢1（暗黙シムを今サイクルで追加）** を採用

### トレードオフと判断根拠

- **得たもの**: 既存設定の透過的後方互換、ダウンストリーム移行コストゼロ
- **犠牲にしたもの**: 後方互換層がコードに残る（将来の major サイクルで剥がす可能性）
- **判断根拠**: patch サイクルとして「既存挙動を壊さない」が最優先。明示記述を強制するのは minor 以上の節目で実施する方が穏当

## DR-005: #609 markdownlint hook 化方針 — 案1（§7.5 完全削除）

- **ステップ**: 03-intent ステップ1（Intent 明確化）
- **日時**: 2026-04-27

### 背景

CI（`pr-check.yml` の `markdownlint-cli2-action@v18`）が必須 Check として動作する状況下で、ローカルの hook 化と Operations §7.5 の関係をどうするか。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 案1（§7.5 完全削除） | 手順書最大簡素化、CI と hook の二層に集約 | ローカル hook 設定漏れ環境では検出が CI 任せになる |
| 2 | 案1（§7.5 を簡素化してヘルパーとして残す） | hook 不在環境向け fallback、二重保険 | 手順書に残置物が出る |
| 3 | 案2（hook + §7.5 併存） | 既存ワークフロー破壊なし | 重複 |
| 4 | 案3（PreToolUse hook） | 編集前防止 | 差分計算が複雑、スコープ超過 |

### 決定

**案1（§7.5 完全削除）** を採用

### トレードオフと判断根拠

- **得たもの**: Operations 手順書の簡素化、CI と hook の二層構造で機能上は十分
- **犠牲にしたもの**: ローカル hook 設定漏れ環境ではローカル検出が CI 任せになる（ただし CI が必須 Check なので最終ゲートは確保）
- **判断根拠**: ユーザーから「CI で必須 Check が走るなら案1で良い」との明示的同意があり、二重保険を捨てて簡素化することで手順全体の保守コストを下げる方が合理的

## DR-006: PRFAQ スキップ — patch サイクル慣習踏襲

- **ステップ**: 04-stories-units ステップ5（PRFAQ作成）
- **日時**: 2026-04-28

### 背景

`depth_level=standard` では原則 PRFAQ を作成するが、直近 patch サイクル（v2.4.2）では同様に「patch サイクルのため depth_level=standard でも省略」とスキップされている。本サイクルでも同じ判断が成立するか確認。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | スキップ（v2.4.2 慣習踏襲） | 4 件統合 patch でプレスリリース対象となる新機能なし、作成コスト削減 | 慣習が暗黙のまま継続 |
| 2 | 通常作成 | depth_level の規定遵守 | patch サイクル特有の作成価値が薄い |

### 決定

**選択肢1（スキップ）** を採用

### トレードオフと判断根拠

- **得たもの**: bug + refactor の patch サイクルでプレスリリース価値が薄い成果物作成を回避
- **犠牲にしたもの**: depth_level=standard の規定上は作成必須を厳格化していない（バックログ #590 等の振り返り対象になりうる）
- **判断根拠**: v2.4.2 で同じ判断が下されており、本サイクルも 4 Issue の bug/refactor 解消で外部発表対象となる新機能なし。慣習を踏襲して同質性を保つ

## DR-007: AI レビューフォールバック — セルフレビュー（パス2）採用

- **ステップ**: 03-intent / 04-stories-units 各 AI レビュー時
- **日時**: 2026-04-27〜2026-04-28

### 背景

Codex CLI の usage limit に到達（復帰: Apr 29 7:56 AM）。`review-routing.md §6` のフォールバック対応では `cli_runtime_error (required)` で `retry_1_then_user_choice`、再試行も同結果のためユーザー選択へ。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | セルフレビューにフォールバック（パス2） | スケジュール遅延なし、Issue #611 の必要性そのものを実証 | レビュー品質が外部 CLI より劣る可能性 |
| 2 | ユーザーレビューに直行（パス3） | レビュー責任が明確 | スキップ理由記録が必要 |
| 3 | Codex 復帰まで中断 | 品質最大 | スケジュール遅延（>12h） |

### 決定

**選択肢1（セルフレビューにフォールバック）** を採用

### トレードオフと判断根拠

- **得たもの**: スケジュール継続、Issue #611（self 統合）の現実的必要性を実証する appropriate な証跡
- **犠牲にしたもの**: 外部 CLI レビューより品質が劣る可能性（同一エージェント内のため検証性低下）
- **判断根拠**: 反復レビュー最大3回 + 各回の指摘解消検証で品質保証を行えば、patch サイクル成果物のレビューとしては十分。後続 Operations Phase の PR レビューは Codex 復帰後に再評価する
