# 論理設計: rules.md ブランチ運用文言の実装整合（Unit 001）

## 概要

`.aidlc/rules.md` 「Worktree 運用ルール」セクションに、ダウンストリーム消費プロジェクト向けとスターターキット自身向けのブランチ命名ポリシー対比節を追加し、稼働実装と整合する命名（`chore/aidlc-v<version>-upgrade`）に文言を修正する。

**重要**: 本論理設計では**コードは書かず**、改訂内容（差分・文言案）と挿入位置のみを定義する。実装（rules.md への反映）は Phase 2 で行う。

## アーキテクチャパターン

ドキュメント改訂のため適用なし。改訂対象は `.aidlc/rules.md` のみ（単一ファイル）であり、レイヤード化やモジュール分割は不要。

## 改訂内容詳細（差分仕様）

### 1. L274 の文言整合

**現行（before）**:

```markdown
2. **アップグレード時**: `upgrade/vX.X.X` ブランチを作成（aidlc-setupスキルが案内）
```

**改訂後（after）**:

```markdown
2. **アップグレード時**: スターターキット自身（メタ開発）では通常使用しない。ダウンストリーム消費プロジェクトでは `aidlc-setup` スキルが `chore/aidlc-v<version>-upgrade` ブランチを案内する（詳細は冒頭「ブランチ命名と用途対比」参照）
```

**確定理由**:

- 実装は v2.4.2 で `chore/aidlc-v<version>-upgrade` 命名で稼働済み（`skills/aidlc-setup/steps/03-migrate.md` 参照）
- L271〜L277 の番号付きリストは「スターターキット自身（メタ開発）のブランチ運用フロー」が前提（冒頭の対比節と注記で明示）。よって項目 2「アップグレード時」はメタ開発では「該当しない」が正解で、項目自体を削除すると番号体系が連続せず読みにくくなるため、**項目を残しつつ「メタ開発では通常使用しない」と明示し、ダウンストリーム側の正しい命名を案内する形** に整える
- これにより「メタ開発フローの中にダウンストリーム命名が混在する」誤読を構造的に抑止する

### 2. 対比節の新規追加

**挿入位置**: 「Worktree 運用ルール」セクション冒頭（L269 の「このプロジェクトでは `.worktree/dev` で開発を行う。」直後、「### ブランチ運用フロー」サブセクションの前）

**選択された案**: ドメインモデル設計の不明点 [Answer] に従い (a) 採用。

**追加内容（後述する Markdown ブロック）**:

```markdown
### ブランチ命名と用途対比

このプロジェクト（AI-DLC Starter Kit 自身）と、本キットを依存として取り込むダウンストリーム消費プロジェクトでは、目的別に異なるブランチ命名を使用する。混同を避けるため対比表を以下に示す:

| 命名パターン | 用途 | 作成主体 | 削除契機 | 適用文脈 |
|--------------|------|----------|----------|----------|
| `cycle/vX.X.X` | スターターキット自身のサイクル開発・リリース | 開発者（手動） | `bin/post-merge-sync.sh` | スターターキット自身（メタ開発） |
| `chore/aidlc-v<version>-upgrade` | ダウンストリーム消費プロジェクトの追従用一時ブランチ | `aidlc-setup` スキル | `aidlc-setup` のマージ後フォローアップ | ダウンストリーム消費プロジェクト |
| `aidlc-migrate/v2` | ダウンストリーム消費プロジェクトの v1→v2 マイグレーション専用 | `aidlc-migrate` スキル | `aidlc-migrate` のマージ後フォローアップ | ダウンストリーム消費プロジェクト |
| `upgrade/v*` | （過去サイクル名残）新規作成はされない | （現スキルは作成しない） | `bin/post-merge-sync.sh`（過去ブランチの安全削除のため残置） | ダウンストリーム消費プロジェクト（過去名残） |

以下「ブランチ運用フロー」はスターターキット自身（メタ開発）が `cycle/vX.X.X` を中心に運用する場合の手順である。`aidlc-setup` / `aidlc-migrate` の動作はダウンストリーム消費プロジェクト側で完結し、スターターキット自身では通常使用しない。
```

**確定理由**:

- 集約の不変条件（用途と適用文脈の対応）を Markdown 表で可視化し、読者が冒頭で全体像を把握できるようにする
- 「以下『ブランチ運用フロー』は…」の注記により、続くサブセクションが**スターターキット自身向け**であることを構造的に明示し、ダウンストリーム向けフローとの混同を抑止する

### 3. L298 の文言整合（L298-A 採用）

**現行（before）**:

```markdown
- `cycle/` および `upgrade/` プレフィックスのマージ済みブランチのみ削除対象（安全制約）
```

**改訂後（after）**:

```markdown
- `cycle/` および `upgrade/` プレフィックスのマージ済みブランチのみ削除対象（安全制約）。`upgrade/*` は過去サイクル名残として削除対象に残置されており、現行のアップグレード走行は `chore/aidlc-v<version>-upgrade` を使用する（削除契機は `aidlc-setup` のマージ後フォローアップでの対話的削除案内）
```

**確定理由（L298-A 採用 / L298-B 不採用）**:

- L298-A 採用理由 1: `chore/aidlc-v*-upgrade` の削除は `skills/aidlc-setup/steps/03-migrate.md` §10 マージ後フォローアップで対話的に行われており、`bin/post-merge-sync.sh` で重複対応すると安全側の挙動が二重化する
- L298-A 採用理由 2: Unit 定義「境界: `bin/post-merge-sync.sh` の対応プレフィックス追加は最小限とし、未対応プレフィックスがある場合のみ最小修正」に従い、必要不可欠でなければ追加しない
- L298-A 採用理由 3: 文言で「現行は `chore/aidlc-v<version>-upgrade` を使用」「削除契機は `aidlc-setup` のマージ後フォローアップ」を明示することで、`bin/post-merge-sync.sh` の挙動と現行命名の対応関係を読者が誤解しないようにする
- L298-B 不採用理由: `bin/post-merge-sync.sh` への `chore/aidlc-v*-upgrade` 追加は本サイクルでは見送り、必要性が確認された別サイクルで対応する（バックログ候補だが、現状で運用上の問題は発生していないため即時実装も登録も不要と判断。問題顕在化時に Issue 化）

### 改訂手順（Phase 2 実装の論理ステップ）

1. `.aidlc/rules.md` の L267〜L298 領域を Read で取得
2. L269（「このプロジェクトでは `.worktree/dev` で開発を行う。」）の直後に「### ブランチ命名と用途対比」サブセクションを挿入（上記§2 の Markdown ブロック）
3. L274（旧）の文言を §1 の改訂後文言（「スターターキット自身では通常使用しない…」）に置換
4. L298（旧）の文言を L298-A の改訂後文言に置換
5. grep 検証（2 系統に分けて期待結果を明示）:
   - `grep -rn "upgrade/v" .aidlc/rules.md` → 残存は L298 改訂後の「過去サイクル名残として `upgrade/*` は…」注記のみであること
   - `grep -rn "upgrade/v" skills/aidlc-setup/ skills/aidlc-migrate/` → 残存ゼロであること
   - `grep -rn "chore/aidlc-v" skills/aidlc-setup/` → 既存 `steps/03-migrate.md` 内の現行記述のみが検出されること
   - `grep -rn "chore/aidlc-v" skills/aidlc-migrate/` → 残存ゼロ（aidlc-migrate は v1→v2 マイグレーション専用で `chore/aidlc-v` との直接関係なし）
   - 4 系統の grep 結果すべてを `history/construction_unit01.md` に記録（`design.md` には記録しない理由: 設計時点では未実行の検証ステップのため成果物は履歴側に保存）
6. markdownlint 実行（`markdown_lint=true` のため）

## 実装上の注意事項

- **編集境界の遵守**: `.aidlc/rules.md` の編集は META-001 例外で許可されているが、対象は L267〜L298 周辺のみ。他セクションへの波及修正は本 Unit のスコープ外
- **対比節の節レベル**: 既存 H2「Worktree運用ルール【重要】」（`.aidlc/rules.md` 現行表記をそのまま使用、半角空白なし／強調付き）の直下の H3（`###`）として配置する
- **表記の一貫性**: 対比表内の `<version>` は実バージョン番号のプレースホルダーであり、`x.y.z` 形式と混在させない
- **過去サイクル名残の表現**: 「`upgrade/*` は過去サイクル名残として削除対象に残置」は、安全制約の維持として記述する（廃止予告ではない）
- **Markdownlint 違反予防**: 対比表前後に空行を挿入し、見出し前後の空行ルールを満たす
- **コミット粒度**: Phase 2 でのコミットは「rules.md 文言整合と対比節追加」を 1 コミットに統合し、後続 grep 検証結果記録は別コミット（または同一コミット内）に追加

## 不明点と質問

[Question] 対比表の「適用文脈」列でメタ開発時とダウンストリームを混在させて表記して良いか
[Answer] 良い。読者は「適用文脈」を見て自分の状況に該当する命名を選択でき、対比節の主目的（混同抑止）と整合する

[Question] L298 の文言を「過去サイクル名残として残置」とする場合、将来のクリーンアップ予告は含めるか
[Answer] 含めない。本 Unit のスコープは「現状の安全制約を文言として明示する」ことに限定し、将来計画は別サイクル / 別 Issue で扱う。今回は文言での「残置の理由（過去サイクル名残）」のみ明示する

[Question] 対比節の挿入により「ブランチ運用フロー」サブセクション以降の H3 番号付けは影響を受けるか
[Answer] 受けない。既存サブセクション（「ブランチ運用フロー」「注意事項」）はすでに番号付けされていない `### サブ見出し` 形式であり、新規 `### ブランチ命名と用途対比` を冒頭に追加しても番号変更は発生しない
