# ユーザーストーリー

## Epic: 開発支援ツール基盤のパーミッション管理強化と堅牢性向上

### ストーリー 1: Kiroエージェント許可設定の自動セットアップ
**優先順位**: Must-have
**関連Issue**: #385

As a Kiro CLIを使うAI-DLC利用者
I want to setup-ai-tools.sh実行時にKiroエージェント設定に許可設定が自動的に含まれる
So that AI-DLCワークフローのコマンド実行時に手動で許可を設定する必要がなくなる

**受け入れ基準**:
- [ ] `setup_kiro_agent`関数が`.kiro/agents/aidlc.json`に許可設定（allowedTools等）を含むJSON を生成すること
- [ ] 少なくとも以下のコマンドパターンが許可対象として含まれること: `docs/aidlc/bin/*`（AI-DLCスクリプト群）、`mktemp /tmp/aidlc-*`（一時ファイル生成）
- [ ] 既存の`.kiro/agents/aidlc.json`がユーザーによりカスタマイズされている場合、ユーザー追加済みの`allowedTools`/`toolsSettings`エントリは失われないこと。テンプレート既定値と衝突するキーはユーザー設定を優先すること
- [ ] `.kiro/agents/aidlc.json`が不正なJSONの場合、上書きせず警告メッセージを出力すること
- [ ] Kiroエージェント設定テンプレート（`docs/aidlc/kiro/agents/aidlc.json`）が許可設定を含む形で更新されること
- [ ] `setup-ai-tools.sh`を再実行しても、既存の許可設定が二重追加されないこと

**技術的考慮事項**:
- `translate-permissions`スキルの変換ロジックを参考にKiro JSON形式（tools, allowedTools, toolsSettings）で許可設定を構成する
- 現在のシンボリックリンク方式を維持しつつ、リンク先のテンプレートファイルに許可設定を含める

### ストーリー 2: ワイルドカードルール検出による重複ルール追加防止
**優先順位**: Must-have
**関連Issue**: #384

As a AI-DLC利用者
I want to setup-ai-tools.shがワイルドカードルールの存在を検出して包含される個別ルールの追加をスキップする
So that aidlc-setup再同期時にユーザーが統合したワイルドカードルールが個別ルールに分解されて戻らない

**受け入れ基準**:
- [ ] `.claude/settings.json`に`Bash(docs/aidlc/bin/:*)`のようなワイルドカードルールが存在する場合、`Bash(docs/aidlc/bin/read-config.sh:*)`のような包含される個別ルールの追加をスキップすること
- [ ] ワイルドカード検出はプレフィックスマッチで判定すること（ルール文字列のツール名とパスプレフィックスを比較）
- [ ] ワイルドカード検出が新規ルールの追加前に実行され、追加不要なルールがフィルタリングされること
- [ ] ワイルドカードルールが存在しない場合は従来通り個別ルールが追加されること
- [ ] スキップされたルール数がログに表示されること
- [ ] ツール名が異なるルール（例: `Skill(...)` vs `Bash(...)`）はワイルドカード包含の判定対象にしないこと
- [ ] パスプレフィックスが類似でも包含関係にないルール（例: `Bash(docs/aidlc/bin-extra/:*)` は `Bash(docs/aidlc/bin/:*)` に含まれない）はスキップしないこと
- [ ] 複数のワイルドカードルールが存在する場合、いずれかに包含されればスキップすること

**技術的考慮事項**:
- `_merge_permissions_jq()`と`_merge_permissions_python()`の両方に検出ロジックを追加する必要がある
- ワイルドカードパターンの判定: `Tool(path/:*)` 形式のルールに対し、`Tool(path/specific:*)` が包含されるか

### ストーリー 3: post-merge-cleanup.shのローカルブランチ不在時の継続実行
**優先順位**: Must-have
**関連Issue**: #381

As a 通常ブランチモードでAI-DLCを利用する開発者
I want to post-merge-cleanup.shがローカルブランチ削除済みの状態でもクリーンアップを完走する
So that マージ後にローカルブランチを先に削除してからクリーンアップを実行するワークフローが正常に動作する

**受け入れ基準**:
- [ ] step_0aでローカルのcycleブランチが存在しない場合、`fatal_error`ではなく警告メッセージを出力すること
- [ ] 警告出力後、step_4（ローカルブランチ削除）をスキップし、step_5（リモートブランチ削除判定）まで到達すること
- [ ] ローカルブランチが存在する場合は従来通りの動作（変更なし）であること
- [ ] スクリプト全体が正常終了（終了コード0）すること
- [ ] `--dry-run`モードでもローカルブランチ不在時に警告を表示し、終了コード0で完了すること（削除は実行せず、スキップ理由をログに出力）

**技術的考慮事項**:
- step_0aの`fatal_error`呼び出しを条件付き警告に変更
- step_4はローカルブランチ不在時に既に非致命的処理を行っているため、step_0aの変更と整合性を確認
