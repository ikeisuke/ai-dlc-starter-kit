# 意思決定記録 - v2.3.5

## DR-001: #579 Operations復帰判定の対策方針

- **ステップ**: Intent明確化（対話）
- **日時**: 2026-04-16

### 背景

Operations Phase の復帰判定チェックポイント（`release_done` / `completion_done`）が `history/operations.md` の「PR Ready化」「PRマージ」記録を参照している。これらの記録は 7.8 / 7.13 タイミングでしか発生しないため、ステップファイル上の「マージ前完結」設計（7.7 までに全記録完了）と矛盾している。v2.3.4 のマージで AIエージェントが独自判断で history に追記し、worktree に未コミット変更が残る問題が実際に発生した。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | Option A: 進捗源移行（復帰判定を `operations/progress.md` へ） | 構造的に「マージ前完結」を保証。CI 再実行コストなし。AIエージェントが誤判断する動機を排除できる | `progress.md` の構造変更・判定ロジック書き換えが必要（実装量中） |
| 2 | Option B: ガード追加（「7.8 以降 history 追記禁止」の明示的ガード + 判定ソース移行） | 構造変更が最小 | 人間の規律依存になりやすく、AI エージェントが誤判断する余地が残る |
| 3 | Option C: post-merge 自動処理（`bin/post-merge-sync.sh` で stash / discard） | ステップファイル変更は最小 | 未コミット変更の漏れ素因は残る |
| 4 | Option D: pre-merge コミット同期（7.13 直前の自動コミット＋push） | ロジック追加だけで済む | push により GitHub Actions CI が再実行され、7.12 のグリーン確認が無効化される |
| 5 | Option A + B: 進捗源移行 + ガード追加 | 確実性最高 | Unit 分解が大きくなり工数も増加 |

### 決定

**Option A（進捗源移行）** を採用

### トレードオフと判断根拠

- **得たもの**: 7.7 コミット時点で復帰判定ソースが確定する構造的保証。CI 再実行コストの回避。AIエージェントの誤動作を構造的に抑止
- **犠牲にしたもの**: `progress.md` の構造変更と判定ロジックの書き換え工数
- **判断根拠**: Option D（pre-merge コミット同期）を評価した結果、CI 再実行コストと 7.12 マージ前レビューのグリーン確認が無効化される難点が致命的と判明。Option B/C は規律依存で根本解決にならない。Option A は実装量中だが、構造変更が本質的な解決をもたらす。A+B は工数が嵩むため A 単独で十分と判断

## DR-002: #574 リモート同期チェック誤検知対策の実装範囲

- **ステップ**: Intent明確化（対話）
- **日時**: 2026-04-16

### 背景

Operations Phase 開始時のリモート同期チェックで、squash 後の history rewrite による divergence が `git rev-list HEAD..@{u} --count` で behind として誤検知される。対策案は3つあり、排他的ではない。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | (1) + (2): merge-base ancestor 先行チェック + diverged ステータス区別 | 誤検知解消と適切な案内が得られる | squash 後 push 忘れの根本原因は残る |
| 2 | (1) + (2) + (3): 上記 + squash 直後の自動 push 案内 | 根本原因（squash 後 push 忘れ）を Construction 側で抑止できる | Construction Phase のステップ修正も必要 |
| 3 | (1) のみ: merge-base ancestor 先行チェックのみ | 最小修正 | diverged と behind の区別がないため、ユーザーへの案内は従来通り曖昧 |

### 決定

**(1) + (2) + (3) 全て** を採用

### トレードオフと判断根拠

- **得たもの**: 誤検知解消（(1)(2)）に加え、根本原因（squash 後 push 忘れ）を Construction 側で抑止する仕組み
- **犠牲にしたもの**: Construction Phase のステップファイル（`04-completion.md`）への追記が追加で必要
- **判断根拠**: (3) の案内は自動実行しないため安全性リスクが低く、案内文言の追加だけで実装コストは限定的。根本原因を抑止する構造的改善は価値が高いため、3点をまとめて実装する

## DR-003: #575 merge-pr --skip-checks オプションの実装範囲

- **ステップ**: Intent明確化（対話）
- **日時**: 2026-04-16

### 背景

CIチェックが設定されていないリポジトリで `operations-release.sh merge-pr` を実行すると `checks-status-unknown` エラーで中断し、ユーザーはスクリプトをバイパスするしかない。改善提案は3つあり、排他的ではない。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | (a) + (b) + (c) 全て: メッセージ改善 + `--skip-checks` + ドキュメント記載 | 改善提案3点を完全にカバー | 変更範囲はやや広い |
| 2 | (a) + (b) のみ: スクリプトの挙動改善に絞る | スクリプト本体の問題解決を優先 | ドキュメント更新は別途必要 |
| 3 | (a) のみ: エラーメッセージだけ改善 | 最小修正 | ワークアラウンド（手動 `gh pr merge`）が残る |

### 決定

**(a) + (b) + (c) 全て** を採用

### トレードオフと判断根拠

- **得たもの**: Issue の改善提案3点を完全にカバー。AI-DLC の適用範囲拡大（個人開発・実験リポジトリでも使える）
- **犠牲にしたもの**: ドキュメント記載の追加工数
- **判断根拠**: (b) `--skip-checks` は `checks-status-unknown` 限定でバイパスを許可し、`failed` / `pending` では拒否する設計のため安全性を損なわない。ドキュメント (c) も適用条件を明示するために必要不可欠。3点を分離せずまとめて実装する

## DR-004: #579 Option D（pre-merge コミット同期）の検討と却下

- **ステップ**: Intent明確化（対話）
- **日時**: 2026-04-16

### 背景

DR-001 の議論中にユーザーから「最後のマージ処理の前に残りの差分をコミットするタイミング」という代替案が提示された。pre-merge の 7.13 直前に未コミット差分を自動コミット＋push する案（Option D）。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | Option D: pre-merge コミット同期 | 実装量は最小（ロジック追加のみ）。履歴構造も変更不要 | push により GitHub Actions CI が再実行され、7.12 のマージ前レビューで確認済みのグリーンが無効化される。待ち時間増加・CI が赤になった場合の逆戻り |
| 2 | Option A に戻る | CI 再実行コストなし、構造的保証あり | 実装量は Option D より大きい |

### 決定

**Option D を却下し、Option A に戻る**

### トレードオフと判断根拠

- **得たもの**: 7.12 のマージ前レビューのグリーン確認を維持できる。リリース直前の待ち時間増加を回避
- **犠牲にしたもの**: Option D による「実装量が小さい」利点
- **判断根拠**: ユーザーからの明示的な懸念（CI 再実行）により Option D の難点が顕在化。マージ前完結ルールを維持するには Option A の進捗源移行が最も構造的に強い解決策と判断した

## DR-005: Unit 005 統合AIレビュー時の Codex 使用制限対応

- **ステップ**: Construction Phase 2 ステップ 6（統合とレビュー）
- **日時**: 2026-04-18

### 背景

Unit 005 の統合AIレビュー実行時に Codex が使用制限に達した（回復予定 2026-04-18 03:23 JST）。`review_mode=required` のため代替手段の選択が必要となった。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | セルフレビュー（サブエージェント） | 待ち時間ゼロで継続可能 | 品質が Codex 比で劣る可能性 |
| 2 | Codex 回復まで待機 | 品質を維持できる | 待ち時間が発生（最大〜1時間） |
| 3 | ユーザー承認へスキップ（要スキップ理由） | 進行を止めない | `required` モードの趣旨に反する、履歴に不完全なスキップ理由が残る |

### 決定

**Option 2: Codex 回復まで待機** を採用

### トレードオフと判断根拠

- **得たもの**: 本 Unit の統合レビューを Codex 品質で実施でき、AIレビューサイクル全体を一貫した水準で完走できた
- **犠牲にしたもの**: セッションの一時中断（session-state.md 経由で再開可能）
- **判断根拠**: `review_mode=required` の厳守とレビュー品質の一貫性を優先。セッション中断は session-state.md で機械的に復元可能なため、主要リスクとはならないと判断

## DR-006: Unit 007 のスコープ外化とスキルリポジトリへの移送

- **ステップ**: Construction Phase 2 ステップ 1（実装開始準備）
- **日時**: 2026-04-18

### 背景

Unit 007 は `suggest-permissions` スキル Review モードに acknowledged findings 抑制機構を追加するもの（関連 Issue: #576）。Construction Phase 着手時、対象スキルが本リポジトリ（`ai-dlc-starter-kit`）ではなく別リポジトリ（`ikeisuke/claude-skills`）に存在することが判明した。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | OUT_OF_SCOPE で取り下げ + スキルリポジトリ側 Issue に移送 | スコープ境界が明確、単一リポジトリ完結を維持 | 本サイクルでは実装されない |
| 2 | サイクル継続で別リポジトリを作業対象に追加 | 本サイクル内で対応可能 | Unit 完了定義がリポジトリ跨ぎで複雑化、Construction Phase 整合性が低下 |
| 3 | 本リポジトリ内で guides / template のみ実装 | 部分的に本サイクル内で対応可能 | Unit 定義の主幹（Review ロジックへの機能追加）が実装されず、Unit 完了とは言えない |

### 決定

**Option 1: OUT_OF_SCOPE で取り下げ + スキルリポジトリ側 Issue に移送** を採用

### トレードオフと判断根拠

- **得たもの**: サイクルスコープの一貫性（単一リポジトリ `ai-dlc-starter-kit` 内で完結）、Construction Phase の整合性、管理可能な範囲での Unit 完了定義
- **犠牲にしたもの**: v2.3.5 サイクル内での Issue #576 解決（別サイクル / 別タイミングに繰延）
- **判断根拠**: Unit 定義の責務「suggest-permissions Review モードに機能を追加」は別リポジトリ側でのコード変更が必須。本リポジトリ内では主幹部分が実装できないため、Unit 完了定義を満たせない。スキルリポジトリ側 Issue（`ikeisuke/claude-skills#26`）に移送することで、対応タイミングと実装箇所の一致を確保

### 実施結果

- `ikeisuke/claude-skills#26` を作成（Unit 007 の責務・境界・失敗モード・後方互換・スコープ外項目・技術的考慮事項を完全移送）
- `ikeisuke/ai-dlc-starter-kit#576` にクロスリファレンスコメントを追加してクローズ
- Unit 007 定義ファイルの状態を「取り下げ」に更新
