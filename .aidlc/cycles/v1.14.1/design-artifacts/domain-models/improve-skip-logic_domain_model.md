# ドメインモデル: Operations Phaseスキップ判定改善

## 概要

Operations Phaseのステップ0スキップ判定を改善し、バージョン確認の責務をステップ1からステップ6に移動する。ステップ5（バックログ整理）を常時実行に変更する。

**重要**: このUnitはプロンプトテキストの構造変更のみ。コードの新規作成・変更はない。

## ドメイン概念の変更

### Operations Phaseフロー

現在のフロー構造と変更後のフロー構造を定義する。

#### 変更前のフロー

```text
ステップ0: 変更確認
  → 「はい」: ステップ1-5を順に実行 → ステップ6
  → 「いいえ」: ステップ1-5をスキップ → ステップ6

ステップ1: デプロイ準備
  └── バージョン確認【必須】
      ├── iOS判定
      ├── ビルド番号確認
      └── 通常バージョン確認
  └── デプロイチェックリスト作成

ステップ5: バックログ整理と運用計画（スキップ可能）
ステップ6: リリース準備（6.1 CHANGELOG〜）
```

#### 変更後のフロー

```text
ステップ0: 変更確認
  → 「はい」: ステップ1-4を順に実行 → ステップ5 → ステップ6
  → 「いいえ」: ステップ1-4をスキップ → ステップ5 → ステップ6

ステップ1: デプロイ準備
  └── デプロイチェックリスト作成（バージョン確認を除去）

ステップ5: バックログ整理と運用計画（常時実行）
ステップ6: リリース準備
  └── 6.0 バージョン確認【必須】（新規追加）
      ├── iOS判定
      ├── ビルド番号確認
      └── 通常バージョン確認
  └── 6.1 CHANGELOG〜（既存）
```

## 責務の移動

| 責務 | 変更前 | 変更後 |
|------|--------|--------|
| バージョン確認 | ステップ1（デプロイ準備） | ステップ6.0（リリース準備） |
| スキップ判定対象 | ステップ1-5 | ステップ1-4 |
| バックログ整理 | スキップ可能 | 常時実行 |

## 設計判断

- **バージョン確認の移動理由**: バージョン確認はリリース直前に行う方が適切。デプロイ準備段階で確認しても、その後の作業で変更される可能性がある
- **ステップ5の常時実行理由**: バックログ整理はリリース品質に直結するため、スキップすべきでない
- **6.0内完結修正**: バージョン確認でNG時はステップ1に戻さず、6.0内でその場修正する
