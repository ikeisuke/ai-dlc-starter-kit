# 論理設計: review-flow圧縮

## 概要

review-flow.mdに対し5つの圧縮パターン（RCP-001〜RCP-005）を適用する変更仕様。

## RCP-001: 遷移判定テーブル → if/else形式

**変更前**:
```markdown
### 遷移判定

| mode | CLI利用可 | スキル利用可 | 遷移先 |
|------|----------|------------|--------|
| `disabled` | - | - | パス3 |
| `recommend` | Yes | Yes | パス1（ユーザーが拒否→パス3） |
| `recommend` | No | Yes | パス2推奨（ユーザーが拒否→パス3） |
| `recommend` | - | No | パス2推奨（ユーザーが拒否→パス3） |
| `required` | Yes | Yes | パス1 |
| `required` | No | Yes | パス2 |
| `required` | - | No | パス2（失敗→ユーザー承認へ、スキップ理由必須） |
| `semi_auto` + `recommend` | - | - | 自動的にパス1/2へ（拒否選択をスキップ） |
```

**変更後**:
```markdown
### 遷移判定

1. `disabled` → パス3
2. `required`: CLI利用可→パス1、CLI不可+スキル利用可→パス2、スキル利用不可→パス2（失敗→ユーザー承認へ、スキップ理由必須）
3. `recommend`: CLI利用可→パス1（拒否→パス3）、CLI不可→パス2推奨（拒否→パス3）
4. `semi_auto` + `recommend`: 自動的にパス1/2へ（拒否選択をスキップ）
```

## RCP-002: パス1説明文の圧縮

パス1の手順番号リストを圧縮。エラー時フォールバックテーブルの説明ヘッダを簡略化。

**変更前**: 「`mode=required` でユーザー承認へ進む場合: スキップ理由の入力が必要（空文字・禁止パターン不可）。履歴に記録。」等の冗長テキスト

**変更後**: テーブル直後に1行で条件を記載

## RCP-003: パス2/3説明文の圧縮

**変更前**:
```markdown
### パス2: セルフレビュー

外部CLI利用不可時、またはtools空配列時のフォールバック。

スキル呼び出し: `args="self-review [対象ファイル]"`。スキル側がサブエージェント起動とインラインフォールバックを管理する。

反復・完了処理はパス1と同一（最大3回、指摘ゼロで完了、残指摘→判断フロー）。
```

**変更後**:
```markdown
### パス2: セルフレビュー

外部CLI利用不可時、または`tools=[]`時のフォールバック。呼び出し: `args="self-review [対象ファイル]"`。反復・完了はパス1と同一。
```

パス3も同様に圧縮。

## RCP-004: レビューサマリファイル説明の圧縮

バックログ列の説明テーブルを維持しつつ、前後の説明文を圧縮。

## RCP-005: CallerContextマッピング説明文の圧縮

テーブル後の3行説明を1行に圧縮。

## 変更禁止セクションの検証方法

実装後に以下の手順で禁止セクション本文の全文一致を確認:

1. 変更前のファイルから禁止セクションを抽出: `git show HEAD:skills/aidlc/steps/common/review-flow.md` で変更前を取得
2. 変更後のファイルから同一セクションを抽出
3. 以下の3セクションについて全文一致を確認:
   - 「指摘対応判断フロー」（L92〜L149相当）
   - 「AIレビュー指摘の却下禁止」（L199〜L203相当）
   - 「外部入力検証」（L205〜L236相当）
