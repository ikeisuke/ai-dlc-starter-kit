# 論理設計: フェーズ完了処理の共通化

## 概要

3フェーズのcompletion.mdに対し、ドメインモデルで特定した7つの圧縮パターン（CP-001〜CP-007）を適用する具体的な変更仕様を定義する。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。

## アーキテクチャパターン

インライン圧縮パターン: 新規ファイルを作成せず、既存ファイル内のテキストを最小化する。既存の委譲先（commit-flow.md, context-reset.md）への参照を明確化し、各フェーズcompletion.md内の重複説明文を削減する。

## コンポーネント構成

### 変更対象ファイル一覧

```text
skills/aidlc/steps/
├── inception/
│   └── 05-completion.md  ← CP-001,002,003,004,005,006 適用
├── construction/
│   └── 04-completion.md  ← CP-001,002,004,007 適用
└── operations/
    └── 04-completion.md  ← CP-004,007 適用
```

### 既存委譲先（変更なし）

- `steps/common/commit-flow.md` — Squash統合フロー・コミット手順の正本
- `steps/common/context-reset.md` — コンテキストリセット手順の正本

## 圧縮パターン適用仕様

### CP-001: Squash結果分岐テキスト

**対象**: Inception(ステップ6), Construction(ステップ7), Inception/エクスプレス(ステップ4)

**変更前**（各箇所で重複）:
```markdown
### N. Squash（コミット統合）【オプション】

> **順序制約**: コミットが存在しない状態でPR作成に進んではいけない。

**【次のアクション】** `steps/common/commit-flow.md` の「Squash統合フロー」を読み込んで実行。

- `squash:success` → ステップNスキップ
- `squash:skipped` → ステップNへ
- `squash:error` → エラーリカバリ後ステップNへ
```

**変更後**: `【次のアクション】` 参照は既存例外として維持。結果分岐は3行のまま維持（これ以上の圧縮は機能等価性を損なう）。ただし順序制約の説明文は削除（commit-flow.md側で保証）。

**推定削減**: Inception通常 ~100B + Inception/エクスプレス ~100B + Construction ~100B = ~300B

### CP-002: Gitコミット squash判定テキスト

**対象**: Inception(ステップ7), Construction(ステップ8), Inception/エクスプレス(ステップ5)

**変更前**:
```markdown
### N. Gitコミット

ステップNで squash を実行した場合（`squash:success`）、コミットは既に完了しています。`git status` で確認のみ行ってください。

squash を実行していない場合は、`steps/common/commit-flow.md` の「XXX完了コミット」手順に従ってください。

**コミット失敗時のエラーハンドリング**: ...（エクスプレスモードのみ）
```

**変更後**:
```markdown
### N. Gitコミット

`squash:success` なら `git status` 確認のみ。それ以外は `commit-flow.md` の「XXX完了コミット」に従う。
```

**推定削減**: Inception通常 ~100B + Inception/エクスプレス ~100B + Construction ~100B = ~300B

### CP-003: コンテキストリセットのセミオートゲート説明

**対象**: Inception(ステップ9), Construction(ステップ11)

**変更前**（Inceptionの例）:
```markdown
### 9. コンテキストリセット提示【必須】

**セミオートゲート判定**（`common/rules.md` のセミオートゲート仕様を参照）: `automation_mode=semi_auto` の場合、コンテキストリセット提示をスキップし、Construction Phaseを自動開始する。`automation_mode=manual` の場合は以下の従来フローを実行する。

**重要**: ユーザーから「続けて」「リセットしないで」「このまま次へ」等の明示的な連続実行指示がない限り、以下のメッセージを**必ず提示**してください。デフォルトはリセットです。

**セッションサマリの生成**: メッセージ提示前に、AIが以下の情報を収集してセッションサマリを生成してください:
...
テンプレート（`templates/context_reset_template.md`）に従い、セッションサマリを含むメッセージを出力する。
```

**変更後**:
```markdown
### 9. コンテキストリセット提示【必須】

`semi_auto`: スキップし次フェーズを自動開始。`manual`: 以下を実行。

セッションサマリ（サイクル番号、ブランチ/PR状態、次のアクション）を収集し、テンプレート（`templates/context_reset_template.md`）に従い出力する。
```

**注**: Construction側も同様に圧縮。Constructionは `context_reset_template.md` を使用しないため、対応内容サマリ（CP-006の対象）とセッションサマリの指示を圧縮。`context-reset.md` への新規参照は追加しない（既存completion内でのインライン圧縮のみ）。

**推定削減**: Inception ~500B + Construction ~400B = ~900B

### CP-004: 完了サマリ出力の説明文

**対象**: 全3フェーズ

**変更前**（各フェーズで重複する説明文）:
```markdown
コンテキストリセット提示の前に、以下のフォーマットで完了サマリを出力する。AIがXX Phase作業中のコンテキスト情報から動的に生成する。
...
※ 情報源に存在しない内容を出力しないこと
```

**変更後**:
```markdown
以下の完了サマリを出力する。※ 情報源にない内容は出力しない。
```

テンプレート（```text```ブロック）自体は維持。

**推定削減**: 3フェーズ × ~100B = ~300B

### CP-005: エクスプレスモード完了処理の内部重複

**対象**: Inception内（エクスプレスモード完了処理セクション）

**変更前**: エクスプレスモード完了処理セクション内にsquash/コミットの独自手順（ステップ4,5）と「完了時の必須作業」内のステップ6,7が同一内容で重複。

**変更後**: 重複部分のみ「完了時の必須作業」のステップ番号参照に置換。エクスプレス固有のコミット失敗時エラーハンドリングは保持する。

```markdown
### 4. Squash（コミット統合）

「完了時の必須作業」のステップ6と同じ手順を実行。

### 5. Gitコミット

「完了時の必須作業」のステップ7と同じ手順を実行。

**コミット失敗時のエラーハンドリング**:
（エクスプレス固有: 失敗時メッセージを保持）
```

**推定削減**: ~400B

### CP-006: 対応内容サマリのテーブル定義

**対象**: Construction(ステップ11), Operations(ステップ7)

**変更前**:
```markdown
**対応内容サマリ【必須】**: セッションサマリの後に以下の項目を含める。AIがUnit作業中のコンテキスト情報から生成する。情報源に存在しないIssue番号・ファイル名・作業内容を出力しないこと。

| 項目 | 説明 | 必須 |
|------|------|------|
| 実施内容 | 主要な変更の箇条書き（関連Issue番号付き） | 必須 |
| 変更対象 | 変更したファイル・モジュールの概要 | 必須 |
| 未対応事項 | スコープ外としたレビュー指摘や残課題（なければ「なし」） | 必須 |
| 次回の着手点 | 次セッションで最初に行うべきこと | 必須 |

情報不足時は該当項目に「（コンテキスト情報不足のため省略）」の固定文言を出力する。
```

**変更後**:
```markdown
**対応内容サマリ**: 実施内容（Issue番号付き）・変更対象・未対応事項（なければ「なし」）・次回の着手点を含める。情報不足時は「（コンテキスト情報不足のため省略）」。
```

テーブル形式を廃止し、必須項目をインラインリストに圧縮。テーブル廃止の理由: 4項目のみで構造化の利点が薄く、インラインリストでも十分に明確。テーブル形式のMarkdown記法（ヘッダ行+区切り行+データ行×4）が約300Bのオーバーヘッドを持つ。

**推定削減**: Construction ~400B + Operations ~400B = ~800B

## NFRへの対応

### パフォーマンス

- **要件**: 3ファイル合計 < 29,722B
- **対応策**: 6つの圧縮パターンで合計~3,000B削減。目標値 ~26,700B

## 処理フロー概要

### 実装手順

1. Inception 05-completion.md にCP-001,002,003,004,005,006を適用
2. Construction 04-completion.md にCP-001,002,004,007を適用
3. Operations 04-completion.md にCP-004,007を適用
4. 各ファイルの `wc -c` で合計サイズを検証
5. 機能等価性の検証（全ステップの入出力が変更前と同一であること）

## 実装上の注意事項

- `context-reset.md` への新規参照は追加しない（CP-003は既存completion内でのインライン圧縮のみ）
- `commit-flow.md` の `【次のアクション】` 参照は既存例外として維持
- 完了サマリのテンプレート（```text```ブロック）自体は圧縮対象外（フェーズ固有のため）
- Operationsの完了処理はサイクル完了という特殊な文脈のため、圧縮は説明文のみに限定
- CP-005でエクスプレス固有のコミット失敗時エラーハンドリングは保持する（通常側ステップ7には含まれないため）
