# ユーザーストーリー

## Epic: ブランチ・サイクル管理の堅牢性向上

### ストーリー 1a: main最新化チェックの判定ロジック実装（#307）
**優先順位**: Must-have

As a AI-DLC利用開発者
I want to setup-branch.shがmainブランチの最新状態を判定できるようにしたい
So that ブランチ作成時にmainの乖離状態を把握できる

**受け入れ基準**:
- [ ] setup-branch.shにmainブランチの最新化チェック機能が実装されている（`git fetch origin` 後に `origin/main` との差分を検出）
- [ ] mainが最新でない場合、警告メッセージが出力される（エラーではなく警告、処理は続行）
- [ ] mainが最新の場合、警告は表示されず処理が続行される
- [ ] `git fetch origin` 失敗時（ネットワーク不通等）は警告チェックをスキップし、`main_status:fetch-failed` を出力して処理を続行する
- [ ] 出力に `main_status:up-to-date` / `main_status:behind` / `main_status:fetch-failed` のステータス行が含まれる

### ストーリー 2: write-history.shの名前付きサイクル対応（#301）
**優先順位**: Must-have

As a 名前付きサイクルを利用する開発者
I want to write-history.shが `name/vX.X.X` 形式のサイクルパスを正しく処理してほしい

**受け入れ基準**:

- validate_cycle() の正規表現を拡張: `^([a-z0-9][a-z0-9-]*/)?v[0-9]+\.[0-9]+\.[0-9]+$` 等
- パストラバーサル防止のバリデーションは維持

## Epic: 名前付きサイクル機能の完成度向上

### ストーリー 4: バックログへの名前付きサイクル情報付与（#305）
**優先順位**: Should-have

As a 名前付きサイクルを利用する開発者
I want to バックログ登録時にどの名前付きサイクルで発見されたかを記録してほしい
So that バックログ項目がどのサイクルに属するか一目でわかる

**受け入れ基準**:
- [ ] Issue作成時のボディの「発見サイクル」セクションに `waf/v1.0.0` 形式でサイクル名が記録される
- [ ] ファイルベースのバックログ作成時の「発見サイクル」フィールドに `waf/v1.0.0` 形式でサイクル名が記録される
- [ ] 通常サイクル（`vX.X.X`）の場合は従来通り `vX.X.X` のみが記録される（影響なし）

**技術的考慮事項**:
- サイクル変数 `{{CYCLE}}` が `name/vX.X.X` 形式の場合、そのまま記録に使用する（テンプレートの変更不要）
- 既存のバックログテンプレートの「発見サイクル」フィールドは値が変わるだけで構造変更なし

**ストーリー間依存**: ストーリー2（write-history.sh）が先に完了していること（名前付きサイクルのパス解決が前提）

### ストーリー 5: Inception時の名前付きサイクル候補表示（#302）
**優先順位**: Should-have

As a 名前付きサイクルを利用する開発者
I want to サイクル名入力時に既存の名前付きサイクル名が候補として表示されてほしい
So that 命名の一貫性を保ちつつ入力の手間を減らせる

**受け入れ基準**:
- [ ] `rules.cycle.mode = "named"` でサイクル名入力時に、`docs/cycles/` 配下の既存名前付きサイクル名がAskUserQuestionの選択肢に昇順で表示される（候補抽出ルール: `v[0-9]` で始まらず、`backlog`/`backlog-completed` でもないディレクトリで、配下に `v[0-9]+.[0-9]+.[0-9]+` パターンのディレクトリを持つもの）
- [ ] 「新規作成（自由入力）」オプションが選択肢の最後に含まれる
- [ ] 既存の名前付きサイクルがない場合は候補表示なしで直接入力を求める
- [ ] `mode = "ask"` で「名前付きサイクル」を選択した場合も同様に候補が表示される
- [ ] `docs/cycles/` の読み取り失敗時は候補なしで自由入力にフォールバックする

**技術的考慮事項**:
- inception.md ステップ5.5の名前入力フロー内で候補を表示
- ステップ5.6の既存ロジック（名前付きサイクル継続確認）と統合整理が必要（候補表示をステップ5.5に統一し、5.6は継続確認のみに絞る等）

## Epic: 開発体験の改善

### ストーリー 6: 環境依存ツールチェックの回避設定（#306）
**優先順位**: Should-have

As a codex等の外部ツールを使用しない開発者
I want to 特定ツールのチェックを設定で無効化したい
So that 毎回のツール存在チェックと許可プロンプトを回避できる

**受け入れ基準**:
- [ ] `aidlc.toml.local` で `[rules.reviewing].tools = []` と設定した場合、外部CLIのwhichチェックがスキップされる
- [ ] 空配列設定時、AIレビューはセルフレビューモードで実行される（レビュー自体はスキップされない）
- [ ] `aidlc.toml`（チーム共有設定）の `tools` リストは従来通り動作する
- [ ] `aidlc.toml.local` の `tools` 設定は `aidlc.toml` の同キーを完全上書きする（read-config.shの既存マージルール通り）
- [ ] 設定未指定時はデフォルト `["codex"]` で従来の動作を維持する
- [ ] `tools` が配列以外の型（文字列・数値等）の場合は警告を出しデフォルト `["codex"]` にフォールバックする（review-flow.mdの既存エラー処理と同一）

**技術的考慮事項**:
- review-flow.md ステップ3のツール可用性チェックロジックを修正
- `tools = []` は「外部CLIを使用しない」の明示的な意思表示として扱う
- 設定優先順位はread-config.shの既存マージルール（local優先・配列完全置換）に従う
