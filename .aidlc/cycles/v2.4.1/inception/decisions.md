## DR-001: v2.4.1 として patch リリースにまとめる

- **ステップ**: Inception ステップ1 セットアップ（サイクルバージョン決定）
- **日時**: 2026-04-25

### 背景

v2.4.0 マージ直後のサイクル開始時、対応候補として backlog / feedback の複数 Issue が存在していた。patch（v2.4.1）／minor（v2.5.0）／major（v3.0.0）のどのバージョン単位にまとめるかを決める必要があった。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | v2.4.1 patch として複数 Issue をまとめる | patch 単発でリリース粒度が小さく、レビュー・検証負荷が低い。ユーザーインパクトが最小化される | Issue の規模感が不揃いで、Unit 間の工数バラつきが出やすい |
| 2 | v2.5.0 minor として新機能を伴わせる | より大きな改善を統合可能 | 対象 Issue の多くが bug/docs であり、minor バンプは過剰 |
| 3 | v3.0.0 major として互換性破壊を伴う改修 | 大規模リファクタリングの好機 | 本サイクル対象 Issue には互換性破壊要素がなく、不適切 |

### 決定

**1. v2.4.1 patch として複数 Issue をまとめる** を採用

### トレードオフと判断根拠

- **得たもの**: リリース粒度の最小化、レビュー負荷低減、Operations/Construction 手順書の堅牢性強化という明確な outcome を単発で届けられる
- **犠牲にしたもの**: patch 粒度のため、スコープが拡大しがちな feature 級 Issue（#605 等）は別サイクルに先送り
- **判断根拠**: 対象 Issue（#601 / #598 / #594 / #600 / #602）はいずれも patch 規模の bug/docs 修正であり、minor/major のバンプ条件（新機能追加・破壊的変更）を満たさない。v2.3.x 系でも同じパターン（patch に複数 Issue を束ねる運用）が確立している

## DR-002: #601（merge_method 設定保存）の解決方針

- **ステップ**: Inception ステップ3 Intent 明確化
- **日時**: 2026-04-25

### 背景

Operations 7.13 で `merge_method=ask` 選択後に `write-config.sh` が `.aidlc/config.toml` を書き換えた際、その変更が PR に反映されず未コミットで残る bug（#601）の解決方針を決める必要があった。Issue 本文では 3 つの案が提示されていた。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 案A: Inception Phase で merge_method を事前確定 | PR に確実に追従。根本解決となる | Operations フローの情報構造大規模変更を伴う。patch 外 |
| 2 | 案B: マージ前にコミット+push フロー明示 | 既存フロー拡張で完結。patch スコープで実装可能 | ユーザー操作が追加されるため手順の説明負荷あり |
| 3 | 案C: 警告のみ（follow-up PR を案内） | 最小修正 | 実運用で手動 follow-up PR が残り続ける |

### 決定

**2. 案B（マージ前にコミット+push フロー明示）** を採用

### トレードオフと判断根拠

- **得たもの**: patch スコープで実装可能な最小修正で根本解決の一歩を踏める。案A は v2.5.0 以降で別途検討可能な形で残す
- **犠牲にしたもの**: 「Inception 側で事前確定する根本解決」には今サイクルでは到達しない。ユーザー操作（AskUserQuestion の 3 択）が増える
- **判断根拠**: patch サイクルは情報構造の変更を避けるのが原則。案A は minor 級の改修で、別サイクルでの扱いが適切。案C では実運用の手動作業が残るため不採用

## DR-003: #598（必須 Checks）の解決方針

- **ステップ**: Inception ステップ3 Intent 明確化
- **日時**: 2026-04-25

### 背景

v2.3.6 で導入した paths フィルタ + Draft skip により、required check が "Expected — Waiting for status to be reported" のまま報告されず PR が merge 不可になる事象（#598）の解決方針を決める必要があった。Issue 本文では複数の対応案が提示されていた。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 常に空ジョブで PASS を返す方式 | required check を安定化。既存 check 名を維持できる。Branch protection 設定変更不要 | workflow ファイル改修が必要 |
| 2 | paths フィルタを廃止しジョブ内で判定 | required check 安定化 | runner 課金増大。v2.3.6 改善が無効化される |
| 3 | Branch protection 設定見直し | 最小変更 | 必須 checks を外すと安全性が低下 |
| 4 | merge_group イベント移行 | Merge Queue の枠組みで整合化 | リポジトリ規模に対して過剰 |

### 決定

**1. 常に空ジョブで PASS を返す方式** を採用（方針レベル固定、具体実装は Construction で 2 案から選定）

### トレードオフと判断根拠

- **得たもの**: required check を安定化しつつ v2.3.6 の Draft skip による runner 課金抑制効果を維持。Branch protection 設定変更が不要で安全性を落とさない
- **犠牲にしたもの**: workflow ファイル 3 本の改修が必要。具体実装は Construction で再検討するため Intent 段階では outcome のみ固定
- **判断根拠**: 方針としては Branch protection の required checks 仕様（報告ジョブが必要）に沿った最も自然な解法。案2 は v2.3.6 の改善を逆行させる、案3 は安全性低下、案4 はリポジトリ規模に対して過剰

## DR-004: #602（Milestone step.md 構造改善）のスコープ

- **ステップ**: Inception ステップ3 Intent 明確化
- **日時**: 2026-04-25

### 背景

v2.4.0 で追加した Milestone 関連 step.md 4 ファイルに対する構造審査の指摘が計 5 件（中 1 / 低〜中 1 / 低 2 + 軽微確認 1）あり、v2.4.1 のスコープとしてどこまで含めるかを決める必要があった。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 中優先度を中心に 4 ファイルすべて対応 | 審査指摘を網羅的に解消。整合性確保 | 1 Unit の修正範囲がやや広がる |
| 2 | 中優先度のみ対応 | 修正範囲を最小化 | 他ファイルの指摘が残り、整合性が次サイクルに持ち越し |

### 決定

**1. 中優先度を中心に 4 ファイルすべて対応** を採用

### トレードオフと判断根拠

- **得たもの**: 4 ファイル間の相互参照整合を同時に保てる。patch スコープで完結できる最小修正案を適用できる
- **犠牲にしたもの**: Unit 005 の修正対象ファイル数が 4 になり Unit 範囲がやや広がる
- **判断根拠**: 指摘の修正案がいずれも「1 行追記 / 見出し併記 / 抽出例追加」の軽量改訂で、4 ファイル合算でも 0.5 日規模。分割して次サイクルに回すオーバーヘッドの方が大きい

## DR-005: Milestone early-link の AND ガード実装（SELECTED_ISSUES 空時の扱い）

- **ステップ**: Inception ステップ3 ユーザーストーリー AI レビュー反映
- **日時**: 2026-04-25

### 背景

Unit 005（#602）の修正項目として `02-preparation.md` §16 の `MILESTONE_ENABLED` ガードと `SELECTED_ISSUES` 空時の挙動の結合関係を明示する必要があった。レビュー段階では「`--issues ""` で呼ぶか early-link をスキップするか等」と未確定な表現が残っており、Testable な受け入れ基準に落とし込むため一方に固定する必要が生じた。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | 呼び出し側でスキップ（AND ガード）| 呼び出し側の責務が明確。無駄なスクリプト起動を避ける | 既存スクリプト出力 `early-link:no-issues-provided` が実運用で発生しなくなる |
| 2 | 既存仕様維持（スクリプト内部でハンドリング） | 変更が最小 | 呼び出し側のガード記述が増える。MILESTONE_ENABLED との AND ガードとしての表現が曖昧になる |

### 決定

**1. 呼び出し側でスキップ（AND ガード）** を採用

### トレードオフと判断根拠

- **得たもの**: Testable な受け入れ基準に落とし込める。呼び出し側の責務が明確化
- **犠牲にしたもの**: 既存スクリプトの `early-link:no-issues-provided` 出力は下位互換用として残るが、実運用では発生しなくなる
- **判断根拠**: `MILESTONE_ENABLED` のガードと自然に整合する AND ガード方式が仕様として一貫する。スクリプト側の出力は下位互換維持で残すため後方互換性の懸念もない

## DR-006: patch スコープでは実装本体を変えず、呼び出し側/文書側の明確化に留める（#594 / #600）

- **ステップ**: Inception ステップ4 Unit 定義
- **日時**: 2026-04-25

### 背景

Unit 003（#594 Squash ステップ誤省略）と Unit 004（#600 aidlc-setup 01-detect 独立チェック指針）の境界設定で、いずれも「実装本体（スキル/判定スクリプト）に分岐ロジックを追加する」代替案が存在した。patch サイクルとしてどこまで踏み込むかを決める必要があった。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | patch スコープでは実装本体を変えず、呼び出し側/文書側の明確化に留める | patch 粒度を維持、blast radius を最小化、既存呼び出し側の責任分界が明確 | 実装本体が「呼ばれると無条件実行」のままで、将来誤呼び出しが起きうる |
| 2 | 実装本体にもガードを追加する（#594: squash-unit に squash_enabled チェック追加、#600: 01-detect 判定スクリプトの独立評価化）| 誤呼び出し耐性が本体側に備わる | patch 粒度を超える改修。責任分界が不明瞭になりうる |

### 決定

**1. patch スコープでは実装本体を変えず、呼び出し側/文書側の明確化に留める** を採用

### トレードオフと判断根拠

- **得たもの**: patch 粒度を維持しつつ、AI エージェント誤判定の主因（手順書の曖昧さ）を解消できる。既存設計の「呼び出し側が分岐責任を持つ」原則と整合
- **犠牲にしたもの**: 実装本体は現行仕様のまま。将来の誤呼び出し耐性は呼び出し側の遵守に依存する
- **判断根拠**: #594 / #600 はどちらも「AI エージェントが手順書を誤解釈して呼ばない/誤って呼ぶ」ことが根本原因。手順書を明確化すれば再発抑制効果は十分で、本体にガードを追加するのは minor 級の責任分界見直しが必要になる。minor/major サイクルで改めて議論する方が安全
