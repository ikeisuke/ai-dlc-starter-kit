# 論理設計: レビューフロー判定ルール改善

## 概要
review-flow.md内の3セクションの具体的な変更内容を定義する。成果物はMarkdownプロンプトの変更であり、ソフトウェアコードではない。

**重要**: この論理設計では**コードは書かず**、コンポーネント構成とインターフェース定義のみを行います。

## アーキテクチャパターン
プロンプトドキュメント内のセクション分離パターン。各変更は独立したセクションに影響し、セクション間の参照（SoT原則）で一貫性を維持する。

## コンポーネント構成

### 変更対象ファイル構成

```text
prompts/package/prompts/common/review-flow.md
├── 「千日手検出（フロー実行前チェック）」セクション
│   └── [変更1] 「同種の指摘」判定基準の追加
├── 「指摘対応判断フロー」セクション
│   └── 「禁止パターン」箇条書き（SoT - 変更なし、参照元として利用）
└── 「外部AIレビューで続行不能時」セクション（ステップ6）
    └── [変更2+3] 理由記録必須化 + write-history.sh統一
```

### コンポーネント詳細

#### 変更1: 千日手検出セクション
- **責務**: 「同種の指摘」の判定基準を明文化し、AIエージェントが一意に判定可能にする
- **依存**: なし
- **変更箇所**: 「千日手検出（フロー実行前チェック）」セクション内に判定基準サブセクションを追加

#### 変更2: ステップ6フォールバック承認
- **責務**: mode=required時の選択肢2の承認プロセスを強化する
- **依存**: 「指摘対応判断フロー」セクションの「禁止パターン」箇条書き（セクション名で参照）
- **変更箇所**: ステップ6の `mode = "required"` の選択肢2の処理フロー

#### 変更3: スキップ記録統一
- **責務**: ステップ6のスキップ記録をwrite-history.sh形式に統一する
- **依存**: write-history.shの呼び出しフォーマット（他セクションのパターンに準拠）
- **変更箇所**: ステップ6の選択肢2の履歴記録部分

## 処理フロー概要

### 変更1: 千日手判定フロー

追加するテキスト（「千日手検出（フロー実行前チェック）」セクション内、「同種の指摘が3回とも繰り返し出現している場合」の直前に判定基準サブセクションを挿入）:

```markdown
   #### 「同種の指摘」の判定基準

   以下の3つのキーがすべて実質的に一致する指摘を「同種」とみなす:

   1. **レビュー種別**: 同一のレビュー種別（code / architecture / security / inception）
   2. **対象ファイル**: 同一のファイルパス（パスが指定されていない指摘はワイルドカードとして扱い、常に一致とみなす）
   3. **指摘内容の要約**: 同一の問題を指摘している（表現の差異は無視し、指摘の本質が同じかで判定）

   **判定手順**:
   1. 各ラウンドの指摘リストを取得する
   2. ラウンド間で上記3キーを比較し、同種の指摘をグルーピングする
   3. 3ラウンドすべてに出現する同種グループが1つ以上あれば「千日手」と判定する

   **サンプルケース**:

   | ケース | ラウンド1 | ラウンド2 | ラウンド3 | 判定 |
   |--------|----------|----------|----------|------|
   | 同種 | code, src/a.ts, 「未使用変数」 | code, src/a.ts, 「未使用の変数がある」 | code, src/a.ts, 「不要な変数」 | 千日手（本質が同じ） |
   | 非同種 | code, src/a.ts, 「未使用変数」 | code, src/b.ts, 「未使用変数」 | code, src/a.ts, 「未使用変数」 | 非千日手（ラウンド2のファイルが異なる） |
   | 非同種 | code, src/a.ts, 「型安全性」 | code, src/a.ts, 「エラーハンドリング」 | code, src/a.ts, 「型安全性」 | 非千日手（ラウンド2の内容が異なる） |
```

### 変更2: ステップ6理由記録必須化フロー

ステップ6の `mode = "required"` の選択肢2を以下のように変更:

**現在**:
```text
選択肢2の場合: 以下を履歴に記録してステップ7へ:
  ### AIレビュースキップ
  - **理由**: 外部AIレビュー継続不能（ユーザー承認済み）
  ...
```

**変更後**:
```text
選択肢2の場合:
  1. スキップ理由の入力を要求:
     「AIレビュースキップの理由を入力してください。
     【注意】「指摘対応判断フロー」セクションの禁止パターンに準ずるバリデーションが適用されます。」
  2. バリデーション:
     a. 空チェック: 空文字・空白のみは不可（結果: EMPTY → 再入力要求）
     b. 禁止パターン: 「指摘対応判断フロー」セクションの禁止パターン箇条書きを参照（結果: PROHIBITED → 再入力要求）
     c. 受理: 上記を通過した場合（結果: OK → 次へ）
  3. 履歴に記録してステップ7へ（write-history.sh形式）
```

### 変更3: write-history.sh統一

ステップ6選択肢2の履歴記録を以下のwrite-history.sh形式に変更:

```bash
docs/aidlc/bin/write-history.sh \
    --cycle {{CYCLE}} \
    --phase {{PHASE}} \
    --unit {N} \
    --unit-name "[Unit名]" \
    --unit-slug "[unit-slug]" \
    --step "AIレビュースキップ" \
    --content "$(cat <<'CONTENT_EOF'
【AIレビュースキップ】外部AIレビュー継続不能
【スキップ理由】{ユーザーが入力した理由}
【対象成果物】{成果物名}
CONTENT_EOF
)"
```

- `{{PHASE}}`: 呼び出し元のフェーズ
- `--unit`, `--unit-name`, `--unit-slug`: constructionフェーズの場合のみ指定。inceptionフェーズではこれらの引数を省略する

## 非機能要件（NFR）への対応

### セキュリティ
- **要件**: write-history.sh呼び出し時にUnit 001で確立した安全なパターンを使用
- **対応策**: heredoc形式（`CONTENT_EOF`）を使用し、ユーザー入力値の直接展開を回避

## 実装上の注意事項
- 禁止パターンの定義は「指摘対応判断フロー」セクションの「禁止パターン」箇条書きを唯一の参照元（SoT）とし、ステップ6では「指摘対応判断フローの禁止パターンに準ずる」と記述する（二重定義を避ける）
- write-history.shの呼び出しフォーマットは既存のステップ5（AIレビューフロー）や千日手判断の記録フォーマットと統一する
- 千日手判定基準は条件3つ（Unit定義の「判定条件3つ以下」制約に合致）
- mode=recommend時のステップ6は変更しない（Unit定義の境界に従う）
