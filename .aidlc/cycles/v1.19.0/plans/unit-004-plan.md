# Unit 004 計画: Reverse Engineering強化

## 概要

Inception Phaseのステップ2（既存コード分析）を「Reverse Engineering」ステージとして拡張し、体系的な解析手順を組み込む。

## 変更対象ファイル

1. **`prompts/package/prompts/inception.md`** - ステップ2の拡張（唯一の変更ファイル）

## 設計方針

### 現行ステップ2の構造

現在のステップ2は4行のみの簡素な記述:

```markdown
### ステップ2: 既存コード分析（brownfieldのみ、greenfieldはスキップ）

**タスク管理機能を活用してください。**

- **ステップ開始時**: progress.mdでステップ2を「進行中」に更新
- 既存コードベースを分析
- `docs/cycles/{{CYCLE}}/requirements/existing_analysis.md` を作成
- **ステップ完了時**: progress.mdでステップ2を「完了」に更新、完了日を記録
```

### 拡張方針

- 見出しを「既存コード分析」から「Reverse Engineering（既存コード分析）」に変更
- brownfield/greenfield判定のConditional分岐を維持（冒頭に明記）
- 4つの解析手順をサブステップとして追加
- `existing_analysis.md` への構造化出力フォーマットをプロンプト内指示として定義
- エラー時のフォールバック動作を記載
- 大規模コードベースでのサブエージェント並行解析の推奨を記載（適用目安・分割単位を明記）

## 実装計画

### ステップ2の新構成: 実行フェーズと確定フェーズ

ステップ2を「実行フェーズ（2.1-2.4: 解析実行）」と「確定フェーズ（2.5: 出力・検証・進捗更新）」に分離する。

```
### ステップ2: Reverse Engineering（brownfieldのみ、greenfieldはスキップ）

[タスク管理・progress.md更新指示]
[greenfield判定: greenfieldの場合はステップ全体をスキップ]

--- 実行フェーズ ---

#### 2.1 ディレクトリ構造・ファイル構成の解析
- プロジェクトルートのディレクトリツリーを取得
- 主要ディレクトリの役割を特定（src, lib, test, docs等）
- ファイル命名規則・配置規則を把握

#### 2.2 使用パターン・アーキテクチャの検出
- アーキテクチャパターンの特定（MVC, レイヤードアーキテクチャ, クリーンアーキテクチャ等）
- デザインパターンの検出（Repository, Factory, Observer等）
- コーディング規約・スタイルの推定

#### 2.3 技術スタック推定
- プログラミング言語・バージョン（根拠ファイル: package.json, Gemfile, go.mod等）
- フレームワーク・ランタイム（根拠ファイル: 設定ファイルやインポート文）
- 主要ライブラリ・ツール（パッケージマネージャの設定ファイルから取得）

#### 2.4 依存関係マッピング
- 内部モジュール間の依存関係（境界単位: パッケージ/ディレクトリ）
- 外部ライブラリの依存関係
- エントリポイントとデータフローの概要
- チェック観点: 依存方向の一貫性、循環依存の有無

**大規模コードベースの場合のサブエージェント並行解析**:
適用目安: トップレベルディレクトリが多数あり単一エージェントでの探索に時間がかかる場合（目安: 主要ディレクトリ6個以上またはファイル総数500以上）
分割単位: トップレベルディレクトリ単位で各サブエージェントに割り当て
統合時: 重複記述のマージ・抜け漏れチェックを実施

--- 確定フェーズ ---

#### 2.5 解析結果の確定
- `existing_analysis.md` への構造化出力（下記フォーマット参照）
- 出力内容の検証: 全セクションに記載があるか、未完了箇所が明示されているか
- エラー発生時のフォールバック処理（下記参照）
- progress.md更新

Done条件: existing_analysis.mdが作成され、全4セクション（ディレクトリ構造、アーキテクチャ、技術スタック、依存関係）に記載がある（エラー時は未完了箇所が特記事項に明示されている）
```

### 構造化出力フォーマット

`existing_analysis.md` の出力フォーマットをプロンプト内指示として定義する。テンプレートファイルは作成しない（Unit定義の境界に明記）。

各セクションに必須フィールドを定義し、推定には根拠ファイルを明記させる:

```markdown
# 既存コードベース分析

## ディレクトリ構造
- ディレクトリツリー（深さ2-3程度）
- 主要ディレクトリの役割と責務

## アーキテクチャ・パターン
各項目に「根拠」（検出元のファイルやコードパターン）を付記:
- アーキテクチャパターン（例: レイヤードアーキテクチャ / 根拠: src/domain/, src/infrastructure/ の分離）
- デザインパターン（例: Repository / 根拠: src/repositories/*.ts）
- コーディング規約・スタイル（例: ESLint設定 / 根拠: .eslintrc.json）

## 技術スタック
| 項目 | 値 | 根拠ファイル |
|------|-----|-------------|
| 言語 | 例: TypeScript 5.x | tsconfig.json |
| フレームワーク | 例: Next.js 14 | package.json |
| 主要ライブラリ | 例: Prisma, Zod | package.json |

## 依存関係
- 内部モジュール間（境界単位: パッケージまたはディレクトリ）
- 外部ライブラリ
- エントリポイントとデータフロー概要
- 依存方向の一貫性・循環依存の有無

## 特記事項
- 解析中に発見した注意点・制約
- 未完了の解析項目（エラー発生時）とその理由
```

### エラー時のフォールバック動作

エラーを2分類し、対応を分ける:

**非致命的エラー**（個別ファイルの読み取り不可、特定ライブラリの解決失敗等）:
- 取得済みの解析結果を記録し、未完了箇所を「特記事項」セクションに明示
- 自動的に次の解析手順へ継続（ユーザー確認不要）

**致命的エラー**（プロジェクトルートへのアクセス不可、ファイルシステム全体の問題等）:
- 取得済みの解析結果があれば `existing_analysis.md` に記録
- ユーザーに継続可否を確認

この分類により `automation_mode=semi_auto` での自動進行と整合する。

### 既存要素の維持

- greenfieldスキップのConditional分岐: 冒頭に維持
- progress.md更新指示: ステップ開始時・完了時の記述を維持

- Depth Level分岐: ステップ2にはUnit 003で追加されたDepth Level分岐は存在しないため、影響なし
