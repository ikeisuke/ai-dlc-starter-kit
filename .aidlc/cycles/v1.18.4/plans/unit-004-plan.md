# Unit 004 計画: PRマージ前レビューコメント確認

## 概要
Operations Phaseのステップ6.6.6（リモート同期確認）と6.7（PRマージ）の間にレビューコメント確認ステップを追加する。プロジェクト固有ルールとして `docs/cycles/rules.md` に定義。

## 変更対象ファイル
- `docs/cycles/rules.md` - 「PRマージ前レビューコメント確認」セクションを追加

## 実装計画

### 現状の問題
Codex PRレビューで指摘されたコメントが未対応のままマージされる可能性がある。現状の6.7では `reviewDecision` のみ確認しており、個別のレビューコメントの対応状況は確認していない。

### 修正方針
`docs/cycles/rules.md` に「PRマージ前レビューコメント確認」ルールを追加。Operations Phaseステップ6.6.6完了後・6.7実行前に、AIがこのルールを参照して実行する。

### ステップ設計（ステップ6.6.7相当）

1. `gh:available` でない場合: スキップ（手動確認を案内）
2. PRレビュー状態を取得（`--paginate` で全件取得）:

   ```bash
   gh api --paginate repos/{owner}/{repo}/pulls/{PR番号}/reviews
   ```

3. PRレビューコメントを取得（`--paginate` で全件取得）:

   ```bash
   gh api --paginate repos/{owner}/{repo}/pulls/{PR番号}/comments
   ```

4. 未対応指摘の判定（REST APIベースの簡易判定）:

   **a. CHANGES_REQUESTED判定**:
   レビュアーごとに最新の `submitted_at` のレビューを取得し、その `state` が `CHANGES_REQUESTED` であるレビュアーが存在するか判定する。
   （注: 6.7の `reviewDecision` が主判定。ここでは個別レビュアーの未解決状態を補助的に確認）

   **b. 未返信コメント判定**:
   `in_reply_to_id` が null のトップレベルコメントに対し、同じ `id` を `in_reply_to_id` に持つ返信コメントが存在しないもの。

5. 判定結果に応じた処理:
   - **未対応指摘なし**: 「レビューコメント確認完了。未対応指摘なし。」と表示し、6.7へ進む
   - **未対応指摘あり**: 未対応指摘一覧を表示し、ユーザーに対応方法を確認:
     - 指摘に対応して修正 → pushした後 `@codex review` を再実行 → 再度レビューコメント確認
     - 指摘を確認済みとしてマージに進む（ユーザー判断）

6. API失敗時のエラーハンドリング:

   | エラーパターン | 対応 |
   |-------------|------|
   | 認証切れ（401） | `gh auth login` の実行を案内 |
   | 権限不足（403） | リポジトリへのアクセス権限確認を案内 |
   | PR不存在（404） | PR番号の確認を案内 |
   | レート制限（403 rate limit） | 待機後の再実行を案内 |
   | その他のエラー | エラー内容を表示 |

   すべてのエラーパターンで、以下の手動確認項目を満たした場合のみ6.7へ進行可:
   - 「GitHub上でPRのレビューコメントを直接確認し、未対応指摘がないことを確認しましたか？」→ ユーザーが「はい」の場合のみマージ可

既存の「Codex PRレビューの再実行ルール」セクションの直後に追加。
