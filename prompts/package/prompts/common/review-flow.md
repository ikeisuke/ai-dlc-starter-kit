# AIレビューフロー

ユーザーに承認を求める前に、AIレビュー（Skills）を実行する。

## 設定確認

`docs/aidlc.toml` の `[rules.reviewing]` セクションを読み、`mode` の値を確認:

- `mode = "required"`: AIレビュー必須（外部AIレビューツールまたはセルフレビューによる実施を推奨。ユーザー判断により例外的にユーザーレビューへ移行可能。承認は履歴に記録する）
- `mode = "recommend"`: AIレビュー推奨（スキップ可能、デフォルト）
- `mode = "disabled"`: AIレビューを行わない（AIレビューをスキップし、直接ユーザーレビューへ進む）

## tools設定と優先ツールヒント

`docs/aidlc.toml` の `[rules.reviewing]` セクションから `tools` を読み取る。

**設定形式**:

```toml
[rules.reviewing]
mode = "required"
tools = ["codex", "claude", "gemini"]  # 優先順位順
```

- `tools`: AIレビューに使用するツールの優先順位リスト（配列）
- デフォルト: `["codex"]`（未設定時）
- `tools` リストの先頭を「優先ツール」としてスキル呼び出し時の引数に含める
- **ツール選択の最終決定はスキル内部の責務**。ここで提供するのは最善努力のヒントのみ

**有効なレビュー種別とスキル名**:

| レビュー種別 | Skills呼び出し |
|-------------|----------------|
| code | `skill="reviewing-code"` |
| architecture | `skill="reviewing-architecture"` |
| security | `skill="reviewing-security"` |
| inception | `skill="reviewing-inception"` |

**スキル呼び出し引数フォーマット**:

```text
skill="reviewing-[type]", args="[レビュー対象] 優先ツール: [codex|claude|gemini]"
```

**フォールバック**: 優先ツール指定はヒントであり強制ではない。スキルが解釈できない場合は `allowed-tools` の順序で自動選択。

**エラー処理**:

- `tools` 未設定: デフォルト `["codex"]` を使用
- 空配列: 警告を出し、デフォルトを使用
- 未知のツール名: 警告を出し、そのツールをスキップ
- 配列以外の型: 警告を出し、デフォルトを使用

## レビュー種別の決定

呼び出し元のコンテキストに基づき、実行するレビュー種別を決定する。

**CallerContextマッピングテーブル**:

| 呼び出し元ステップ | デフォルトのレビュー種別 |
|---|---|
| 計画承認前 | architecture |
| Phase 1 ステップ3（設計レビュー） | architecture |
| Phase 2 ステップ4（コード生成後） | code |
| Phase 2 ステップ6（統合とレビュー） | code, security |
| Intent承認前 | inception |
| ユーザーストーリー承認前 | inception |
| Unit定義承認前 | inception |

**決定ルール**:

1. 呼び出し元ステップを確認し、上記テーブルからデフォルト種別を取得
2. 呼び出し元が不明な場合はユーザーに選択を求める
3. ユーザーが追加の種別を指定した場合はマージする

**複数種別実行時のルール**:

- 複数種別は**直列実行**（1つずつ順番に実行）
- 各種別ごとに反復レビュー（最大3回）を独立して実施
- **全種別で指摘0件**になった時点で「AIレビュー完了」とする
- 履歴記録には実行した全種別を記載

## AIレビューツール利用可否の確認

**Skills確認**:

1. レビュー種別リストの**全種別**について、対応するスキルが利用可能か個別に確認
   - 各 `skill="reviewing-[type]"` で呼び出し可能かを確認
2. 全種別が利用可能 → AIレビュー実行へ
3. 一部のみ利用可能:
   - `mode = "required"`: 利用不可の種別ごとにユーザーにスキップ承認を求める（承認は履歴に記録）。全て承認後、利用可能な種別のみでAIレビュー実行。承認拒否 → AIレビュー不可として処理
   - `mode = "recommend"`: 利用不可の種別について警告を表示し、利用可能な種別のみでAIレビュー実行
4. すべて利用不可: AIレビュー不可として処理

**環境別の確認方法**:

- Claude Code: Skillツールが存在し、`skill="reviewing-[type]"` で呼び出し可能かを確認
- KiroCLI等の他環境: Skill一覧取得APIがあればそれを利用

## 処理フロー

1. **mode確認**: `docs/aidlc.toml` を読んでmodeを確認
   - 空または取得失敗時は「recommend」として扱う
   - `disabled` の場合: ステップ7（ユーザーレビューフロー）へ
   - `required` または `recommend` の場合: 次のステップへ

2. **レビュー種別決定**:
   - 「レビュー種別の決定」セクションのルールに従い、実行するレビュー種別を決定
   - 結果: レビュー種別リスト（例: `[code, security]`）

3. **優先ツール決定・利用可否チェック**:
   - `tools` リストの先頭を優先ツールとする（デフォルト: `codex`）
   - レビュー種別リストの**全種別**について `skill="reviewing-[type]"` が利用可能か個別に確認
   - 全種別が利用可能 → ステップ4へ
   - 一部のみ利用可能:
     - `mode = "required"`: 利用不可の種別ごとにユーザーにスキップ承認を求める（承認は履歴に記録）。全て承認後、利用可能な種別のみでステップ4へ。承認拒否 → ステップ6へ
     - `mode = "recommend"`: 利用不可の種別について警告を表示し、利用可能な種別のみでステップ4へ
   - すべて利用不可 → ステップ6（外部AIレビューで続行不能時）へ

4. **AIレビューツール利用可能時の選択**:
   - `mode = "required"` の場合: ステップ5（AIレビューフロー）へ
   - `mode = "recommend"` の場合: 推奨メッセージを表示しユーザーに選択を求める

     ```text
     【レビュー推奨】AIレビューツール（Skills）が利用可能です。
     品質向上のため、この成果物のレビューを実施することを推奨します。
     レビューを実施しますか？
     ```

     - 「はい」の場合: ステップ5（AIレビューフロー）へ
     - 「いいえ」の場合: ステップ7（ユーザーレビューフロー）へ

5. **AIレビューフロー**:
   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う

   - **種別ごとの反復レビュー**（レビュー種別リストを先頭から順に直列実行）:

     各種別で以下を実行:

     1. AIレビューを実行: `skill="reviewing-[type]", args="[レビュー対象] 優先ツール: [tool]"`
     2. レビュー結果を確認
     3. 指摘があれば修正を反映
     4. 指摘がゼロになるまで1-3を繰り返す（最大3回）
     5. **全種別で指摘がゼロになった時点で**: 以下のメッセージを出力し、履歴に記録してからレビュー後コミット→ユーザーへの承認提示ステップへ進む

        ```text
        【AIレビュー完了】指摘0件
        ```

        **履歴への記録**:

        ```bash
        docs/aidlc/bin/write-history.sh \
            --cycle {{CYCLE}} \
            --phase {{PHASE}} \
            --unit {N} \
            --unit-name "[Unit名]" \
            --unit-slug "[unit-slug]" \
            --step "AIレビュー完了" \
            --content "$(cat <<'CONTENT_EOF'
        【AIレビュー完了】指摘0件
        【対象タイミング】{呼び出し元のステップ名（例: 設計レビュー、統合とレビュー）}
        【対象成果物】{成果物名}
        【レビュー種別】{実行した全種別（例: code, security）}
        【レビューツール】{使用したツール名}
        CONTENT_EOF
        )"
        ```

        > **注意**: `CONTENT_EOF` 終端行は、実行時には行頭（インデントなし）に配置すること。上記コード例ではドキュメントの可読性のためインデントしている。

        - `{{PHASE}}`: 呼び出し元のフェーズ（`construction` または `inception`）
        - `--unit`, `--unit-name`, `--unit-slug`: constructionフェーズの場合のみ指定。inceptionフェーズではこれらの引数を省略する
        - `{呼び出し元のステップ名}`: このAIレビューフローを呼び出したステップを記載
          - Phase 1 ステップ3: `設計レビュー`
          - Phase 2 ステップ6: `統合とレビュー`
          - Intent承認前: `Intent承認前`
          - ユーザーストーリー承認前: `ユーザーストーリー承認前`
          - Unit定義承認前: `Unit定義承認前`

        **レビューサマリ更新**: 「## レビューサマリファイル更新手順」に従いサマリファイルを更新する（計画承認前を除く）。

     6. 3回後も指摘が残る場合は、「### 指摘対応判断フロー」を実行
     7. 判断フロー後、「修正する」を選択した指摘がある場合は以下を通知し、反復レビューの1に戻る:

        ```text
        指摘の修正を行いました。さらに最大3回のAIレビューを実施します。
        ```

   ### Codex セッション管理（Codexを選択した場合）

   **Codex Skillを使用したAIレビューでは、反復レビュー時に必ずresume機能を使用する**

   #### セッションID保持

   1. **初回レビュー実行後**: Codex出力末尾の `session id: xxx` を記録
   2. **2回目以降のレビュー**: `codex exec resume <session-id> "<request>"` を使用
      - resumeにより前回の指摘内容・コンテキストが継続される
      - `-C` や `-s` オプションは通常継承されるため省略可能

   #### 使用例

   ```text
   [1回目] codex exec -s read-only -C . "コードをレビューしてください"
           → 指摘あり、session id: abc123 を記録

   [修正を実施]

   [2回目] codex exec resume abc123 "指摘を修正しました。再度レビューしてください"
           → 前回の文脈を踏まえて継続レビュー
   ```

   #### フォールバック処理

   resume失敗時（セッション期限切れ、エラー等）は新規セッションで再開:

   ```text
   【フォールバック】セッション期限切れのため、新規セッションで再開します。
   ```

   新規セッション開始後、session idを更新して記録する。

   #### Unit切り替え時のリセット

   - 別Unitの作業を開始する際は、セッションIDを破棄
   - 新しいUnitでは新規セッションで開始する
   - 理由: 異なるUnitのコンテキストが混在することを防ぐ

   ### 指摘対応判断フロー

   AIレビューで指摘が残っている場合に、各指摘への対応を判断するフロー。

   #### 起動条件

   - AIレビュー反復（最大3回）後に指摘が残っている場合
   - ステップ5（AIレビューフロー）またはステップ5.5（セルフレビューフロー）の反復レビューから呼び出される

   #### 千日手検出（フロー実行前チェック）

   判断フロー実行前に、過去3回の反復レビュー結果を確認する。

   - **同種の指摘が3回とも繰り返し出現している場合**: 千日手と判断し、ユーザーに判断を求める:

     ```text
     【千日手検出】AIレビューが収束していません。
     過去3回の反復レビューで同種の指摘が繰り返されています。
     どのように対応しますか？
     1. さらに継続する
     2. 現状でユーザー承認へ進む
     3. 処理を中断する
     ```

     - 選択肢1: 通常の判断フローへ進む
     - 選択肢2: 以下を履歴に記録し、ユーザー承認ステップへ進む:

       ```bash
       docs/aidlc/bin/write-history.sh \
           --cycle {{CYCLE}} \
           --phase {{PHASE}} \
           --unit {N} \
           --unit-name "[Unit名]" \
           --unit-slug "[unit-slug]" \
           --step "千日手判断" \
           --content "$(cat <<'CONTENT_EOF'
       【千日手検出】AIレビューが収束せず、ユーザー判断によりユーザー承認へ移行
       【対象成果物】{成果物名}
       【反復回数】{過去の反復レビュー回数}
       【繰り返し指摘】{繰り返されていた指摘の要約}
       CONTENT_EOF
       )"
       ```

       - `{{PHASE}}`: 呼び出し元のフェーズ（`construction` または `inception`）
       - `--unit`, `--unit-name`, `--unit-slug`: constructionフェーズの場合のみ指定。inceptionフェーズではこれらの引数を省略する

       **レビューサマリ更新**: 履歴記録後、「## レビューサマリファイル更新手順」に従いサマリファイルを更新する（計画承認前を除く）。結論は「指摘対応判断完了」とし、指摘一覧の対応列に「千日手によりユーザー承認へ移行」と記載する。

     - 選択肢3: 処理を中断

   - **それ以外**: 通常の判断フローへ進む

   #### フロー

   1. **指摘一覧の表示**:

      ```text
      【確認】AIレビューで指摘が残っています。各指摘について対応を選択してください。

      残りの指摘:
      1. [指摘内容1]
      2. [指摘内容2]
      ...
      ```

   2. **各指摘への判断要求**（順番に繰り返す）:

      ```text
      指摘 #[N] について、どのように対応しますか？
      「[指摘内容]」

      1. 指摘を修正する（推奨）
      2. 技術的理由で対応不可（理由を記録）
      3. スコープ外として次サイクルで対応（理由を記録）
      ```

   3. **選択肢2または3が選ばれた場合、理由入力を要求**:

      ```text
      指摘 #[N] の先送り理由を入力してください。
      「[指摘内容]」

      【注意】以下のような理由は単独では受け付けられません:
      - 「パッチだから」「小さい変更だから」
      - 「時間がないから」「急ぎだから」

      具体的な技術的理由や根拠を含めて記載してください。
      ```

   4. **理由のバリデーション**:
      - 空文字・空白のみは不可
      - 禁止パターンのみで構成される理由は拒否し、再入力を要求

      **禁止パターン**（以下のみで構成される理由は拒否）:
      - 「パッチだから」「パッチなので」
      - 「小さい変更だから」「軽微な変更なので」
      - 「時間がないから」「急ぎだから」

      **拒否時のメッセージ**:

      ```text
      【理由が不十分です】
      入力された理由: "[入力内容]"

      この理由では先送りを承認できません。
      具体的な技術的理由や根拠を含めて、再度入力してください。

      例:
      - 「依存ライブラリXのバグにより、現バージョンでは対応不可。Issue #123 で追跡中」
      - 「この変更はUnit 005のスコープに含まれており、そちらで対応予定」
      ```

   5. **先送り判断の履歴記録**（選択肢2または3の場合、各指摘ごと）:

      ```bash
      docs/aidlc/bin/write-history.sh \
          --cycle {{CYCLE}} \
          --phase {{PHASE}} \
          --unit {N} \
          --unit-name "[Unit名]" \
          --unit-slug "[unit-slug]" \
          --step "AIレビュー指摘対応判断" \
          --content "$(cat <<'CONTENT_EOF'
      【指摘 #{index}】{指摘内容の要約}
      【判断種別】{TECHNICAL_BLOCKER|OUT_OF_SCOPE}
      【先送り理由】{ユーザーが入力した理由}
      CONTENT_EOF
      )"
      ```

      - `{{PHASE}}`: 呼び出し元のフェーズ（`construction` または `inception`）
      - `--unit`, `--unit-name`, `--unit-slug`: constructionフェーズの場合のみ指定。inceptionフェーズではこれらの引数を省略する

   6. **全指摘の判断完了後、サマリを履歴に記録**:

      ```bash
      docs/aidlc/bin/write-history.sh \
          --cycle {{CYCLE}} \
          --phase {{PHASE}} \
          --unit {N} \
          --unit-name "[Unit名]" \
          --unit-slug "[unit-slug]" \
          --step "AIレビュー指摘対応判断サマリ" \
          --content "$(cat <<'CONTENT_EOF'
      【AIレビュー指摘対応判断サマリ】
        指摘 #1: RESOLVE（修正予定）
        指摘 #2: TECHNICAL_BLOCKER（理由記録済み）
        指摘 #3: OUT_OF_SCOPE（理由記録済み）
      【次のアクション】{修正実施後に再レビュー|ユーザーレビューへ}
      CONTENT_EOF
      )"
      ```

      - `{{PHASE}}`: 呼び出し元のフェーズ（`construction` または `inception`）
      - `--unit`, `--unit-name`, `--unit-slug`: constructionフェーズの場合のみ指定。inceptionフェーズではこれらの引数を省略する

   7. **次のアクション決定**:
      - 「修正する（RESOLVE）」を選択した指摘がある場合: 修正を実施し、呼び出し元の反復レビューへ戻る（ステップ5 項目7 またはステップ5.5 項目9）
      - 全て先送りの場合:
        1. **レビューサマリ更新**: 「## レビューサマリファイル更新手順」に従いサマリファイルを更新する（計画承認前を除く）
        2. ユーザーレビューフローへ進む

   - **レビュー後コミット**: `common/commit-flow.md` の「レビュー反映コミット」手順に従う

   - 修正後の成果物をユーザーに提示
   - ユーザーの承認を求める

5.5. **セルフレビューフロー**（Skills利用不可時のフォールバック）:

   **注意**: セルフレビューは外部ツールに比べて品質が劣る可能性があります。重要な変更では、可能な限り外部AIレビューツールの利用を推奨します。

   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う

   - **レビュー実行方式の判定**:

     セルフレビューは**サブエージェント方式**（コンテキスト分離）を優先し、利用できない場合は**インライン方式**にフォールバックする。

     - **Claude Code環境**: サブエージェント方式を試行（Task ツールの `subagent_type` を使用。権限制約は指示テンプレート内で指定。技術的な強制はプラットフォーム依存）
     - **その他のAIツール**: サブプロセスやサブセッションでコンテキスト分離を試行（参考情報、実行方式はツール固有のドキュメントを参照。正式サポートはClaude Codeのみ）
     - **サブエージェント起動失敗時**: インライン方式（従来のメインエージェント自身によるレビュー）にフォールバック

     **注意**: サブエージェントに委譲するのは「レビュー実行（読み取り・評価・結果出力）のみ」。フィードバック反映・修正判断・反復制御・履歴記録はメインエージェントの責務。

   - **セルフレビュー実行手順**:

     #### 共通準備（方式共通）

     1. レビュー種別（code / architecture / security / inception）に応じた観点を確認
        - 各種別の観点は対応するレビュースキルのSKILL.mdを読み取る（注: Skillの実行は不可でもSKILL.mdファイルはディスク上に存在するため参照可能）
        - SKILL.mdが見つからない場合: review-flow.md 内の該当種別の説明（ステップ5のレビュー種別一覧等）を観点として使用する
        - 読み取った「レビュー観点」セクションの内容を `review_criteria` として保持する
     2. 対象の成果物ファイルパス一覧を `target_files` として整理する

     #### サブエージェント方式（推奨）

     3. サブエージェントを起動し、以下の指示テンプレートを渡す:

        **Claude Code での起動**:

        ```text
        Task ツール呼び出し:
          subagent_type: "general-purpose"
          description: "セルフレビュー実行"
          prompt: [以下の指示テンプレート]
        ```

        **指示テンプレート**:

        ````text
        以下のファイルをレビューしてください。
        あなたの役割は読み取り専用のレビュアーです。ファイルの読み取りと評価のみを行い、ファイルの編集・コマンド実行・外部通信は行わないでください。

        **レビュー種別**: {review_type}

        **対象ファイル**:
        {target_files を改行区切りで列挙}

        **レビュー観点**:
        {review_criteria（共通準備で取得した観点本文）}

        **出力フォーマット**:
        レビュー結果を以下のフォーマットで出力してください。

        指摘がある場合:

        指摘 #1
        - 重要度: {高 | 中 | 低}
        - 内容: {指摘内容の要約}
        - 推奨修正: {修正方法の提案}

        指摘 #2
        ...

        合計: {N}件（高: {n}件 / 中: {n}件 / 低: {n}件）

        指摘がない場合:
        指摘0件
        ````

     4. サブエージェントの結果を受信し判定する:
        - **指摘なし**: 指摘0件として処理（手順6へ）
        - **指摘あり**: 各指摘から「重要度」「内容」「推奨修正」の3項目を解析。3項目が揃わない指摘は無効として除外する
        - **起動失敗 / 結果解析不能（全指摘が無効）**: インライン方式にフォールバック（手順3'へ）。フォールバック発生時は履歴に理由を記録する（例: 「サブエージェント起動失敗のためインラインフォールバック」）

     #### インライン方式（フォールバック）

     3'. サブエージェント方式が利用できない場合、メインエージェント自身がレビューを実施する:
        - `review_criteria` の観点に基づいて対象ファイルをレビューし、出力フォーマットに従って記録する

     #### 反復・完了処理（方式共通）

     5. 指摘がある場合は修正を反映
        - メインエージェントが各指摘に `id`（指摘番号）と `judgement`（未対応/対応済）を付与・管理する
     6. 反復レビュー（最大3回）: 修正後に再度レビューを実施（サブエージェント方式の場合は新しいサブエージェントを起動。サブエージェントはステートレス）
     7. **全種別で指摘がゼロになった時点で**: 以下のメッセージを出力し、履歴に記録してからレビュー後コミット→ユーザーへの承認提示ステップへ進む

        ```text
        【AIレビュー完了（セルフレビュー）】指摘0件
        ```

        **履歴への記録**:

        ```bash
        docs/aidlc/bin/write-history.sh \
            --cycle {{CYCLE}} \
            --phase {{PHASE}} \
            --unit {N} \
            --unit-name "[Unit名]" \
            --unit-slug "[unit-slug]" \
            --step "AIレビュー完了" \
            --content "$(cat <<'CONTENT_EOF'
        【AIレビュー完了（セルフレビュー）】指摘0件
        【対象タイミング】{呼び出し元のステップ名}
        【対象成果物】{成果物名}
        【レビュー種別】{実行した全種別}
        【レビューツール】{self-review(subagent) | self-review(inline)}
        【注記】セルフレビューによる実施。
        CONTENT_EOF
        )"
        ```

        - `{{PHASE}}`: 呼び出し元のフェーズ（`construction` または `inception`）
        - `--unit`, `--unit-name`, `--unit-slug`: constructionフェーズの場合のみ指定。inceptionフェーズではこれらの引数を省略する

        **レビューサマリ更新**: 「## レビューサマリファイル更新手順」に従いサマリファイルを更新する（計画承認前を除く）。

     8. 3回後も指摘が残る場合は、「### 指摘対応判断フロー」（ステップ5と同じフロー）を実行
     9. 判断フロー後、「修正する」を選択した指摘がある場合は以下を通知し、セルフレビュー実行手順の5に戻る:

        ```text
        指摘の修正を行いました。さらに最大3回のセルフレビューを実施します。
        ```

   - **出力フォーマット**:

     指摘ありの場合:

     ```text
     【セルフレビュー結果】
     レビュー種別: {code | architecture | security | inception}

     指摘 #1
     - 重要度: {高 | 中 | 低}
     - 内容: {指摘内容の要約}
     - 推奨修正: {修正方法の提案}
     - 判定: {未対応 | 対応済}

     指摘 #2
     ...

     合計: {N}件（高: {n}件 / 中: {n}件 / 低: {n}件）
     ```

     指摘なしの場合:

     ```text
     【セルフレビュー結果】指摘0件
     レビュー種別: {種別}
     ```

     **指摘対応判断フローとの接続ルール**:
     - 判断フローへの入力は `判定: 未対応` の指摘のみ（対応済は除外）
     - 反復レビュー時、同一指摘IDの判定を `未対応 → 対応済` に更新する
     - 3回後の収束判定: `判定: 未対応` の指摘が残っている場合に指摘対応判断フローを起動

   - **外部入力検証の非適用**: セルフレビューは同一エージェントシステム内（サブエージェント含む）が実施するため、「外部入力検証ルール」の「AIレビュー応答の検証」は適用しない（自己の判断を自ら検証することは無意味であるため）。ただし以下のチェックは適用する:
     - **構造チェック**: 手順4の「重要度」「内容」「推奨修正」3項目が揃っていること
     - **妥当性チェック**: メインエージェントが各指摘の修正判断時に、指摘内容の妥当性（根拠の有無、禁止操作の提案がないか等）を確認する

   - **レビュー後コミット**: `common/commit-flow.md` の「レビュー反映コミット」手順に従う

   - 修正後の成果物をユーザーに提示
   - ユーザーの承認を求める

6. **外部AIレビューで続行不能時**（全種別利用不可、または一部利用不可で承認拒否の場合）:
   - `mode = "required"` の場合:

     ```text
     【警告】AIレビューが必須設定ですが、外部AIレビューを継続できません。

     どのように対応しますか？
     1. セルフレビューを実施する（推奨） - エージェント自身がレビューを実施
     2. ユーザー承認へ進む - レビュースキップを履歴に記録
     3. 処理を中断
     ```

     - 選択肢1の場合: ステップ5.5（セルフレビューフロー）へ。セルフレビューは「必須AIレビュー充足」として扱う
     - 選択肢2の場合: 以下を履歴に記録してステップ7へ:

       ```markdown
       ### AIレビュースキップ
       - **理由**: 外部AIレビュー継続不能（ユーザー承認済み）
       - **日時**: YYYY-MM-DD HH:MM:SS
       - **対象成果物**: {成果物名}
       ```

     - 選択肢3の場合: 処理を中断し、ユーザー再指示待ち状態へ

   - `mode = "recommend"` の場合:

     ```text
     【レビュー推奨】AIレビューツール（Skills）が利用できませんが、セルフレビューを実施できます。
     品質向上のため、セルフレビューの実施を推奨します。
     セルフレビューを実施しますか？
     1. はい - セルフレビューを実施する（推奨）
     2. いいえ - ユーザーレビューへ進む
     ```

     - 「はい」の場合: ステップ5.5（セルフレビューフロー）へ
     - 「いいえ」の場合: ステップ7（ユーザーレビューフロー）へ

7. **ユーザーレビューフロー**（mode=disabled または 外部AIレビューで続行不能時）:
   - **レビュー前コミット**: `common/commit-flow.md` の「レビュー前コミット」手順に従う

   - 成果物をユーザーに提示
   - ユーザーの承認を求める
   - 修正依頼があれば修正を反映
   - **レビュー後コミット**: `common/commit-flow.md` の「レビュー反映コミット」手順に従う

   - 再度ユーザーに提示・承認を求める

## レビューサマリファイル更新手順

AIレビュー完了時にレビューサマリファイルを生成・追記する共通手順。ステップ5（AIレビューフロー）、ステップ5.5（セルフレビューフロー）、および指摘対応判断フローから参照される。

### 前提条件

- AIレビュー（外部ツールまたはセルフレビュー）が実施された場合のみ実行する
- **計画承認前のレビューでは実行しない**（計画レビューは設計前段階のため対象外）
- レビュー未実施（mode=disabled、スキップ等）の場合はファイルを作成しない

### ファイルパス決定ルール

| フェーズ | 呼び出し元ステップ | ファイルパス |
|---------|------------------|------------|
| Construction | Phase 1 ステップ3（設計レビュー） | `docs/cycles/{{CYCLE}}/construction/units/{NNN}-review-summary.md` |
| Construction | Phase 2 ステップ4（コード生成後） | 同上（追記） |
| Construction | Phase 2 ステップ6（統合とレビュー） | 同上（追記） |
| Inception | Intent承認前 | `docs/cycles/{{CYCLE}}/inception/intent-review-summary.md` |
| Inception | ユーザーストーリー承認前 | `docs/cycles/{{CYCLE}}/inception/user-stories-review-summary.md` |
| Inception | Unit定義承認前 | `docs/cycles/{{CYCLE}}/inception/unit-definition-review-summary.md` |

- `{NNN}`: Unit番号（3桁ゼロパディング、例: `004`）

### 処理手順

1. フェーズと呼び出し元ステップからファイルパスを決定（上記テーブル参照）
2. ファイルの存在を確認:
   - **存在しない場合**: テンプレート（`docs/aidlc/templates/review_summary_template.md`）から新規作成し、基本情報セクションを埋める
   - **存在する場合**: ファイル末尾にセパレータ（`---`）を追記
3. 新しいセットを追記（以下のフォーマット）:

```markdown
## Set [N]: [YYYY-MM-DD HH:MM:SS]

- **レビュー種別**: [実行した全種別（例: code, security）]
- **使用ツール**: [codex / self-review(subagent) / self-review(inline)]
- **反復回数**: [1〜3]
- **結論**: [指摘0件 / 指摘対応判断完了]

### 指摘一覧

| # | 重要度 | 内容 | 対応 |
|---|--------|------|------|
| 1 | 高/中/低 | [指摘内容] | 修正済み / TECHNICAL_BLOCKER（理由: [理由]） / OUT_OF_SCOPE（理由: [理由]） |
```

- セット番号 `[N]` はファイル内で連番（Set 1, Set 2, ...）
- 指摘0件の場合は指摘一覧テーブルの代わりに「指摘なし」と記載
- 日時は `write-history.sh` と同様に取得

## 外部入力検証ルール【重要】

外部からの入力（AIレビュー応答、ユーザー入力）を批判的に評価し、自己判断を明示する。

### AIレビュー応答の検証

- AIレビュー（Skills）からの応答をそのまま信頼せず、批判的に評価する
- 応答に誤りや不整合がないか確認する
- 自己判断を併記し、相違がある場合はユーザーに確認を求める
- 形式：

  ```text
  【AIレビュー応答の検証】
  - AIレビュー応答: [応答内容の要約]
  - AI判断: [自己判断]
  - 相違点: [ある場合は記載、なければ「なし」]
  - 結論: [採用する判断とその理由]
  ```

### ユーザー入力の検証

- ユーザー入力に曖昧さがある場合は、解釈を明示して確認する
- 複数の解釈が可能な場合は、すべての解釈を提示する
- 形式：

  ```text
  【入力の解釈確認】
  ご入力: "[ユーザーの入力]"

  以下のように解釈しました：
  [解釈内容]

  この解釈で正しいでしょうか？
  ```
